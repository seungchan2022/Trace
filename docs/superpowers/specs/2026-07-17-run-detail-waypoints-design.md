# MVP15 — 러닝 경험 보강: 디테일 폴리시 + 포인트 찍기 설계

- 날짜: 2026-07-17 (킥오프 브레인스토밍)
- 상태: 사용자 설계 승인 + 스펙 리뷰(ce-doc-review) 완료 + 발화 문안 표 사용자 확정(2026-07-17) — 플랜 작성 단계로 이행
- 마일스톤: ① `run-detail-polish` (경량 사이클) ② `run-waypoints` (표준 사이클)

## 0. 배경과 방향 결정

MVP14까지로 두 기둥(코스 계획, 러닝 트래킹·경험)이 각각 완성됐다. 다음 후보였던
코스 연동(코스 골라 뛰기)은 **러닝 기둥을 먼저 단단하게 만든 뒤** 착수하기로 결정
(2026-07-17, 사용자) — 두 기둥을 잇기 전에 러닝 기능이 제대로 돌아가야 연동 QA에서
문제 원인을 분리할 수 있다.

이번 MVP의 재료는 전부 실사용 피드백에서 나왔다:

| # | 실사용 불편 | 해결 | 마일스톤 |
|---|---|---|---|
| 1 | 시작 버튼을 누르면 준비 시간 없이 바로 시작됨 | 시작 카운트다운 (§1.1) | ① |
| 2 | 종료 길게 누르기 중 아무 시각 변화가 없음 | 홀드 진행 링 (§1.2) | ① |
| 3 | 발화 멘트 톤이 제각각 + 말 속도가 빠름 | 문안 통일 + rate 조정 (§1.3) | ① |
| 4 | 목표 거리·시간을 피커로만 선택 가능 | 직접 입력 + 단위 상시 표시 (§1.4) | ① |
| 5 | 달리는 중 임의 지점 간 거리를 잴 방법이 없음 | 포인트 찍기 (§2) | ② |

**비범위(명시적으로 이번에 안 하는 것):**

- **러닝 탭 UI 구조 개편** (대기 화면 지도 제거/대체, 탭·시트 형태, NRC식 표현 문법) —
  사용자 구상이 아직 안 익어 MVP16 후보로 백로그에 씨앗만 기록. MVP15 실사용 중
  아이디어를 쌓은 뒤 MVP12 방식(design-direction 문서 사이클)으로 시작한다.
- **포인트 → 코스 변환** — 러닝 GPS 궤적은 라우팅을 거치지 않은 원시 좌표라, 코스로
  만들면 "라우팅 안 된 두 번째 코스 종류"가 생긴다. 플래너의 편집 규칙(attach·스냅·왕복)이
  전부 라우팅된 구간을 전제하므로, GPS 코스와 라우팅 코스의 공존은 코스 연동 MVP에서
  정식 설계한다. 이번엔 원본(포인트 스트림)만 저장해 소급 변환 가능성을 남긴다.
- **자동 일시정지, 잠금화면 일시정지 버튼** — MVP14 이연 결정 유지 (트리거 미충족).

**사이클 정책** (MVP14 선례와 동일):

- 스펙은 이 문서 1개, 문서 리뷰(ce-doc-review) 1회.
- 플랜은 사이클마다 별도 작성, 실행도 사이클 단위 순차 진행.
- ① 경량: 태스크별 리뷰만, 최종 브랜치 리뷰 생략 (열린 설계 결정 없음).
- ② 표준: 태스크별 리뷰 + 최종 브랜치 리뷰(opus) (신규 기능 + 스키마 확장 + Widget Extension 배선).

## 1. 마일스톤 ① `run-detail-polish` — 기존 기능 마감 손질

### 1.1 시작 카운트다운

- 시작 버튼(자유/거리/시간 모드 공통) 탭 → 화면 전체 큰 숫자 **3 → 2 → 1** (1초 간격)
  → 트래킹 시작.
- 각 숫자마다 **발화("삼", "이", "일") + 햅틱** 동반, 카운트다운 종료 시 시작 발화
  (§1.3 문안 표)가 시작 신호 역할 (사용자 요구 2026-07-17: 카운트에도 오디오, 끝나면
  시작 안내).
- **덕킹은 한 번만**: 오디오 세션을 카운트다운 시작 시 잡고 시작 발화가 끝난 뒤 놓는다 —
  3초 사이에 덕킹을 켰다 껐다 반복하지 않는다.
- **취소**: 카운트다운 중 화면 탭 → 취소하고 대기 화면 복귀 (발화 즉시 중단).
- **백그라운드 진입 시에도 계속 진행** (결정 2026-07-17, 스펙 리뷰): 카운트다운 3초 사이
  앱이 백그라운드로 가도(잠금·전화·알림) 카운트다운·발화·세션 시작이 그대로 이어진다 —
  "시작 누르고 바로 폰 잠그고 뛰기"가 표준 흐름이고, 설계 상수(러닝 중 = 화면
  꺼짐/백그라운드)와 일치. 취소는 화면을 다시 켜서 탭하는 것으로만.
- **GPS 예열**: 카운트다운 시작과 동시에 위치 스트림을 켠다. 세션 시작 시각·거리 적산은
  카운트다운 종료 시점부터 (기존 "GPS 신호 대기" 시간이 카운트다운 3초에 흡수되는 효과).
  기존 세션 활동 시간 의미론(일시정지 제외 등)은 변경 없음 — 시작 시각의 기준점만
  카운트다운 뒤로 이동.
- **권한·정확도 확인은 카운트다운 전에 완료**: 위치 권한·임시 전체 정확도 요청(시스템
  프롬프트 포함)은 시작 버튼 탭 직후, 카운트다운 시작 전에 처리한다. 확보 실패 시
  카운트다운을 시작하지 않고 기존 실패 안내를 보여준다. 카운트다운과 동시에 하는 것은
  스트림 켜기(예열)뿐 — "삼, 이, 일" 발화 위로 시스템 다이얼로그가 겹치지 않게 한다.
- **카운트다운 종료 시 GPS 미확보(콜드 스타트·빌딩 밀집지)**: 세션은 그대로 시작하고
  기존 "신호 확보 중" 표시로 전이한다(시작 발화도 예정대로). 이 상태에서 활동 시간은
  거리 0인 채 흐르고, 거리 적산은 첫 유효 샘플부터다. 이 분기는 §3 상태 전이 테스트에
  포함한다.
- 구현 노트: "덕킹 한 번만"과 "발화 즉시 중단"은 현재 announcer(`announce(_:)`만 있고
  큐가 비면 오디오 세션을 해제하는 구조)로는 표현할 수 없다 — 오디오 세션 명시
  hold/release(또는 `postUtteranceDelay` 배치 큐잉)와 stop(취소) API를
  `VoiceAnnouncerProtocol`에 추가하는 확장이 필요하다.

### 1.2 종료 홀드 진행 링

- 현재: `onLongPressGesture(minimumDuration: 1.0)` + 눌림 축소 효과뿐 — 진행 상태가
  안 보인다.
- 변경: 종료 버튼 둘레에 **누르는 동안 1초에 걸쳐 차오르는 원형 링** 표시, 다 차면
  햅틱과 함께 종료. 중간에 떼면 링은 리셋(짧은 되감기 애니메이션).
- 구현 노트: `onLongPressGesture`는 진행률 콜백이 없으므로 커스텀 제스처(press 시작/종료
  추적) + 타이머/애니메이션으로 교체. **동작 시간 1초는 유지** — 길이가 아니라 피드백
  부재가 문제였다.

### 1.3 발화 문안 통일 + 속도 조정

현재 `RunAnnouncementBuilder`의 문안은 톤이 혼재한다(명사형 "일시정지" vs 동사형
"재개합니다" vs 감탄 "목표 달성!"). **정중한 통보체(~합니다)로 통일**하고, 데이터
낭독부(거리·시간·페이스)는 현행 형식을 유지한다.

| 시점 | 현재 | 변경안 |
|---|---|---|
| 카운트다운 | (없음) | "삼" / "이" / "일" (신규) |
| 시작 | "러닝 시작" | "러닝을 시작합니다" |
| 일시정지 | "일시정지" | "일시정지합니다" |
| 재개 | "재개합니다" | (유지) |
| km 경계 | "3킬로미터. 총 시간 18분 30초. 평균 페이스 6분 10초" | (유지 — 데이터 낭독) |
| 목표 절반 | "절반 왔습니다" | (유지) |
| 목표 달성 | "목표 달성! 5킬로미터, 29분 10초" | "목표를 달성했습니다. 5킬로미터, 29분 10초, 평균 페이스 5분 50초" (평균 페이스 절 추가 — 종료 발화와 동일 규칙: 페이스 산출 불가 시 절 생략) |
| 종료 | "러닝 종료. 총 5.2킬로미터, 31분 40초, 평균 페이스 6분 5초" | "러닝을 종료합니다. 총 5.2킬로미터, 31분 40초, 평균 페이스 6분 5초" |
| 포인트 (②에서 추가) | (없음) | "포인트 2, 0.87킬로미터" (신규, §2.2) |

- 문안은 스펙 리뷰에서 사용자가 직접 확정한다 — 위 표가 단일 출처가 되고, 구현은 이 표를
  그대로 옮긴다.
- **속도**: 현재 `utterance.rate` 미설정(시스템 기본 0.5)이 빠르게 들린다는 실사용 피드백 →
  **0.45로 낮춰 시작**하고 실기기 QA에서 미세 조정해 확정한다. 설정 화면은 만들지 않는다
  (YAGNI — 기본값 튜닝으로 충분하다고 판단되면 종료, 불편 피드백이 나오면 그때 설정화).

### 1.4 목표 거리·시간 직접 입력

- 현재: 휠 피커 2종 (`goalDistanceKm`, `goalTimeMinutes`, 시간 최소 5분).
- 변경: **텍스트 입력**으로 교체.
  - 거리: km 단위 소수 1자리 (예: `5.5`), `decimalPad`.
  - 시간: 분 단위 정수 (예: `30`), `numberPad`.
- **단위 상시 표시** (사용자 요구 2026-07-17): 입력 필드 옆에 "km" / "분" 고정 라벨(suffix)
  + 플레이스홀더에 예시값 — 처음 쓰는 사람도 단위를 바로 알 수 있게.
- 검증: 0 이하·파싱 불가·비정상 값이면 시작 버튼 비활성 + 필드 아래 인라인 안내(예:
  "0보다 큰 값을 입력하세요")로 이유를 보여준다. 피커의 "최소 5분" 제약은 직접
  입력에서는 두지 않는다 (1분부터 허용 — 임계값 없음 원칙과 일관).
- 키보드 UX: `decimalPad`/`numberPad` 위에 "완료" 툴바를 두어 명시적으로 내릴 수 있게
  하고, 키보드가 떠 있는 동안 시작 버튼이 가려지지 않도록 레이아웃(스크롤 또는 재배치)을
  보장한다.
- 입력 필드는 직전에 사용한 목표값으로 미리 채운다(모드별 기억) — "지난번과 같은 목표로
  또 뛰기"가 지배적 흐름이므로. 플레이스홀더 예시값은 최초 사용 시에만 보인다.
- 저장 스키마(목표 타입·값 옵셔널 컬럼)는 변경 없음.

## 2. 마일스톤 ② `run-waypoints` — 달리면서 포인트 찍기

### 2.1 개념과 목적

달리는 중 버튼을 누르면 **직전 포인트(첫 포인트면 세션 시작)부터 지금까지의 거리**를
알려주고 기록에 남기는 기능. 목적: 지도에 없는 길(공원 샛길 등)은 플래너로 그릴 수
없으므로(탭·그리기 모두 `MKDirections` 보행 라우팅을 통과 — 보행 네트워크 위 경로만
생성 가능), 실제로 뛰면서 구간 거리를 재는 것이 유일한 수단이다. 플래너의 구멍을 러닝
기둥이 메운다.

원래 워치용으로 구상한 기능이지만 폰 단독으로도 성립하며(잠금화면 버튼 §2.3), 워치
연동 MVP에서 워치 버튼이 같은 세션 연산에 연결될 수 있도록 세션 로직과 입력 채널을
분리해 둔다.

### 2.2 트래킹 중 UX

- 트래킹 화면에 **포인트 버튼** 추가 (정확한 위치·모양은 구현 시 확정 — UI 세부).
- 탭하면:
  - **발화(주 채널)**: "포인트 2, 0.87킬로미터" — 뛰는 중 화면을 안 보는 상황이 기본.
  - **화면 카드(보조)**: "포인트 2 · 0.87 km" 몇 초 표시 후 사라짐.
- 발화 톤은 ①(§1.3)에서 정비한 문안 표를 따른다.
- **구간 거리 정의**: 직전 포인트~현재의 누적 이동 거리. 새 거리 계산을 만들지 않고
  기존 `RunSession` 총거리 적산의 스냅샷 차이로 구한다 — 일시정지 제외·정확도 필터 등
  기존 규칙을 자동 상속.
- **일시정지 중 버튼 비활성** (거리가 안 쌓이는 상태). **첫 유효 GPS 샘플 확보 전에도
  비활성** — 같은 근거(거리가 안 쌓이는 상태)이며, 덕분에 포인트 좌표(§2.4)는 항상
  존재가 보장된다. 비활성 상태는 낮은 불투명도(dimmed)로 표시해 눌러도 안 되는 상태임을
  보이게 한다.
- **GPS 공백 중 탭**: 스냅샷은 마지막 유효 샘플 기준이며, 공백 구간 거리(공백이 끝날 때
  한꺼번에 가산되는 직선 거리)는 다음 구간에 귀속된다 — 수용하는 한계로 문서화하고, §3의
  "GPS 공백 낀 경우" 테스트는 이 문장을 기대값(오라클)으로 삼는다.
- **포인트 발화는 즉시성 우선**: km 경계 등 데이터 낭독이 진행 중이면 중단하고 포인트
  발화를 바로 재생한다(시작/일시정지 등 상태 전환 발화보다는 후순위). 발화가 주(잠금화면
  에선 유일) 확인 채널이라, 침묵을 실패로 오해해 재탭하는 연쇄를 막기 위함.
- 연타 방지 임계값 없음 — 0.00 km 구간도 허용 ("아주 짧은 러닝도 저장"과 같은 원칙).
  오탭 복구는 기록 상세의 포인트 삭제(§2.5)가 담당한다.

### 2.3 잠금화면 포인트 버튼 (Live Activity)

- 기존 Widget Extension(run-live-activity)의 Live Activity에 **App Intent 기반 버튼** 추가 —
  iOS 17+ 인터랙티브 위젯은 **잠금 해제 없이** 탭 가능. "달리다가 폰 들어 바로 찍고 계속
  달리기" 시나리오의 핵심.
- 인텐트(`LiveActivityIntent`)는 앱 프로세스에서 실행 — 백그라운드에서 살아 있는
  `RunSession`에 직접 연결, 별도 IPC 불필요.
- 피드백: 잠금화면에서는 화면 카드가 안 보이므로 **발화가 확인 신호** (기존 백그라운드
  오디오·덕킹 경로 재사용, 신규 인프라 없음). Live Activity 본문에도 **마지막 포인트
  한 줄**(예: "P2 · 0.87 km")을 작게 추가해 발화를 놓쳐도 눈으로 확인 가능하게.
- **일시정지 중엔 잠금화면 버튼도 비활성** — 앱 내 버튼과 동일 규칙(샘플 미확보 시에도
  동일). 비활성 표시는 SwiftUI `.disabled()` 표준 스타일을 따른다.
- **무세션 가드**: 인텐트 실행 시 활성 세션이 없으면(러닝 중 강제종료 후 잠금화면에 남은
  카드 등) no-op으로 처리하고 잔여 Live Activity를 종료한다. "강제종료 → 잠금화면 버튼
  탭" 시나리오를 ② 실기기 QA 체크리스트에 넣는다.
- 포인트 버튼은 기존 지표 행(거리·시간·페이스)과 **별도 행/영역**으로 분리해 최소 탭
  영역을 확보한다 — 달리면서 눌러야 하는 버튼이므로 좁게 욱여넣지 않는다.
- 첫 포인트가 찍히기 전에는 포인트 줄 자체를 표시하지 않는다 (§2.5의 "없으면 숨김" 원칙
  준용).
- 일시정지 버튼은 여전히 잠금화면에 넣지 않는다 (MVP14 결정 유지). 같은 잣대(사용
  빈도·달리는 흐름 유지)로 판단: 일시정지는 저빈도 + 쓰는 상황이면 폰을 꺼내는 동작이고,
  포인트는 고빈도 + 흐름을 끊지 않아야 하는 동작이라 반대 결론이 자연스럽다.

### 2.4 저장 스키마 (additive)

- 기록에 **포인트 스트림** 추가: 포인트 = `타임스탬프 + 좌표 + 그 시점 누적 거리`.
  - 타임스탬프: 원본 사실.
  - 좌표: 지도 마커용 스냅샷 — 탭 시점의 마지막 유효 샘플 좌표 (샘플에서 파생 가능하지만
    스냅샷 저장이 단순하고 견고).
  - 누적 거리: 표시용 캐시 (기존 거리·시간 캐시 컬럼과 같은 성격).
- 기존 "기록 = 타임스탬프 샘플 스트림 묶음" 원칙의 additive 확장 — 스트림 하나를 옆에
  추가. **과거 기록은 포인트가 없을 뿐 그대로 호환** (마이그레이션 없음).
- 러닝 기록 스토어 파일(코스와 분리)은 그대로.

### 2.5 기록 상세 표시

포인트가 있는 기록에만:

- 지도 궤적 위 **번호 마커** (①②③…).
- km 스플릿 표와 별도로 **포인트 구간 표**:
  "시작→① 1.24 km / ①→② 0.87 km / … / ③→종료 0.42 km".
  마지막 포인트~종료 구간도 한 줄로 넣어 **합계가 총거리와 일치**하게 한다.
- 포인트 없는 기록(과거 기록 포함)은 섹션 자체를 숨긴다.
- **포인트 개별 삭제**: 기록 상세에서 포인트 하나를 지울 수 있다(스트림에서 제거 후
  구간 표·마커 재계산). 오탭 포인트가 기록을 영구 오염시키지 않게 하는 복구 경로 —
  "연타 임계값 없음" 결정의 안전판.
- **종료 요약 화면은 변경 없음** — 포인트는 기록 상세 전담 (요약 화면 확장은 MVP16 UI
  개편에서 검토).
