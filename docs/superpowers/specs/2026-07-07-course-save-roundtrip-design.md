# MVP11 — course-save-roundtrip: 코스 저장 + 구간 왕복 설계

2026-07-07. 다음 MVP 브레인스토밍(원점 방향 탐색)에서 확정된 두 실사용 pain을 해결한다:
① 앱을 껐다 켜면 작업 중 코스가 사라져 매번 처음부터 다시 그려야 함, ② 이미 만든 코스의
특정 구간을 다시 갔다오는(왕복) 경로를 라우팅 오차 없이 만들 수 없음.

- 단위: MVP11 = 마일스톤 2개(`course-save`, `roundtrip-insert`) = **사이클 1개**
  (스펙 1·플랜 1·브랜치 `feature/course-save-roundtrip`)
- 무게: **경량** + persistence 설계 섹션(§2)만 문서 리뷰 1회
  (사이클/무게 규칙은 `docs/agent-rules/workflow.md` 참고 — 이번 사이클에서 명문화)

## 0. 배경 — 왜 이 둘인가

- 코스 저장은 2026-07-04 백로그에서 제거됐었다(commit `8015231`) — 기능이 불필요해서가
  아니라, "자동으로 다음 1순위"로 굳는 것을 막고 방향성부터 다시 세우기 위해서였다.
  이번 브레인스토밍에서 Trace의 큰 그림을 "러닝 전(계획) → 러닝 중(계획·실제 경로 동시
  표시) → 러닝 후(비교·아카이빙)"로 정리했고, 그 그림의 모든 미래 단계(코스 골라서 달리기
  시작, 기록 아카이빙, 코스 비교)가 "코스가 여러 개 존재하고 그중 하나를 고른다"는 저장
  기반을 요구함을 확인했다. 저장은 그 기반의 첫 조각으로 **방향이 서고 나서** 착수한다.
- 구간 왕복의 본질 pain: 같은 곳을 탭해도 라우팅이 다른 도로를 고를 수 있고, 그리기도
  정확히 겹치지 않으면 다른 경로가 나온다 → "갔던 길 그대로 돌아오기"를 수동으로는 만들 수
  없다. 해법은 라우팅을 다시 부르지 않고 **확정된 좌표를 복제**하는 것 — 100% 동일 경로가
  구조적으로 보장된다.

## 1. 범위

### 포함

- **course-save**: 작업 중 코스 자동 저장·복원(초안) + 이름 붙여 저장/목록/불러오기/삭제.
- **roundtrip-insert**: 구간 패널에서 임의 구간 뒤에 "왕복"(역방향+정방향) 삽입.
- 규칙·문서 갱신: `workflow.md` 사이클/무게 규칙, `project-decisions.md` Persistence 결정,
  `backlog.md` 정정(stale 2건) 및 "되짚어 오기" 후보 기록.

### 제외 (이번 사이클에서 하지 않음)

- "되짚어 오기"(마지막 구간 역방향만 붙여 코스 끝을 되돌리는 조작) — 왕복 추가로 대부분
  커버되므로 보류, 백로그 후보로 기록. 실사용에서 아쉬움이 확인되면 소규모 후속.
- redo(앞으로가기) 스택의 재시작 복원 — redo는 원래 수명이 짧은 상태(새 편집 시 소멸)라
  복원 가치가 낮음. 재시작 시 비활성으로 시작(사용자 확정, 2026-07-07).
- 코스 이름 변경/정렬/검색, iCloud 동기화, 코스 썸네일 — 목록은 꾸미지 않은 기본 List.

## 2. Persistence 설계 (문서 리뷰 대상)

### 결정: SwiftData + 프로토콜 격리

- 미결이던 Persistence를 **SwiftData**로 확정한다(2026-07-07). 근거: iOS 17+ ·
  `@Observable`과 같은 세대의 네이티브 스택, "네이티브 API 우선" 원칙, 미래 데이터(달리기
  기록 아카이빙)의 조회 확장성. 로컬 전용(동기화 없음 — 기존 기본값 유지).
- 지도 SDK 격리와 동일한 포트-어댑터 패턴: Domain에 `CourseRepositoryProtocol`, 구현은
  `Trace/Infrastructure/Persistence/SwiftData/`의 어댑터. ViewModel/Domain은 SwiftData를
  import하지 않는다. 저장 방식 교체는 어댑터 교체로 끝난다.

### 저장 대상 두 종류

| 종류 | 개수 | 내용 | 쓰기 시점 |
|---|---|---|---|
| 초안(작업 중 코스) | 최대 1 | 세션 상태 스냅샷: entries(**엔트리 `id`** + 세그먼트 + 시간순 `order` + `placedAtFront` + 왕복 anchor) + `nextOrder` | 편집 연산 확정 시마다 + 앱 백그라운드 진입 시 |
| 저장 코스 | 여러 개 | `id` + 이름 + 저장일 + 세그먼트 배열(공간순) + 총 거리 | 사용자가 저장 버튼을 누를 때 |

- 초안은 undo가 재시작 후에도 동작하도록 **세션 상태 그대로**(시간순 정보 포함) 저장한다.
  redo 스택은 저장하지 않는다.
- 저장 코스는 **스냅샷 의미론**: 저장 후 지도에서 계속 편집해도 저장본은 불변. 갱신하려면
  다시 저장(새 항목). 같은 이름 중복 허용(저장일로 구분).
- 데이터 규모: 코스 하나 = 좌표 수백~수천 = 직렬화 수 KB~수십 KB. 편집 연산은 사람 손
  속도(분당 수 회)이므로 연산마다 저장해도 비용은 무시 가능 — MVP3의 카메라 백그라운드
  저장과 같은 패턴의 확장.

### 도메인 모델과 SwiftData 모델 분리

- Domain(`Trace/Domain/CoursePlanning/Entity`): `CourseDraft`(엔트리 배열 스냅샷),
  `SavedCourse`(id·이름·저장일·세그먼트·총거리). 기존 `Course`/`CourseSegment` 재사용.
- Protocol(`Trace/Domain/CoursePlanning/Protocol`): `CourseRepositoryProtocol` —
  `saveDraft`/`loadDraft`/`clearDraft` + `saveCourse`/`fetchCourses`/`deleteCourse`.
- Infrastructure: SwiftData `@Model` 클래스는 어댑터 내부 전용. 좌표 배열은 행으로 펼치지
  않고 인코딩된 `Data` 속성으로 저장(수천 좌표를 레코드로 만들 이유가 없고, 좌표
  단위 쿼리 요구가 없다). 도메인 ↔ SwiftData 변환은 어댑터가 전담.
- **직렬화 포맷은 어댑터 내부 DTO**로 정의하고 blob에 **포맷 버전 필드**를 둔다 — 도메인
  타입에 Codable을 직접 붙이면 도메인 리팩터링(케이스명 변경 등)이 기존 blob을 해독 불가로
  만든다. 초안은 버려도 되지만 저장 코스는 사용자 자산이다.
- `ModelContainer`는 App 계층이 아니라 **어댑터가 내부에서 생성·소유**한다
  (`.modelContainer(for:)` SwiftUI 관용구를 쓰면 `@Model` 타입이 App으로 새어 격리가 깨짐).
  생성은 throws이므로 실패 정책을 명시한다: 실패 시 스토어 파일을 백업 이름으로 옮기고 새로
  생성(자산은 즉시 삭제하지 않고 보존), 그래도 실패하면 in-memory 폴백 — **앱은 어떤 경우에도
  뜬다**(런치 크래시 금지). autosave는 끄고 명시적 `save()` + 오류 포착으로 운용한다.
- `DependencyContainer`에 프로토콜 타입으로 등록. 테스트는 in-memory `ModelContainer`.

### 실패 처리와 순서 보장

- **순서 불변식(스펙 요구)**: 초안 저장은 편집 연산 순서대로 직렬 실행되어야 한다 — 연산
  N+1의 스냅샷이 N보다 먼저 디스크에 닿으면 "나중 저장이 이긴다" 재시도 구조가 깨진다.
  실행 방식(어댑터 actor, mainContext `@MainActor` 활용 등)은 플랜에서 정하되 이 불변식은
  구현이 지켜야 할 요구사항이다. (MVP3 카메라 저장은 메인에서 동기 쓰기라 이 문제가 없었음 —
  같은 패턴이 아니라 확장임을 유의.)
- 저장 실패(디스크 등): 편집 흐름을 막지 않는다 — 초안 자동 저장 실패는 로그를 남기고
  넘어가되(전체 스냅샷 덮어쓰기라 다음 성공이 복구), **연속 실패가 임계(3회)에 달하면
  세션당 1회 사용자에게 알린다** — 지속성 오류(디스크 풀 등)에서 "껐다 켜면 그대로" 약속이
  무음으로 깨지는 것을 막는다. 명시적 "이름 저장" 실패는 즉시 알림.
- 복원 실패(손상된 초안): 빈 상태로 시작하고 손상 초안은 버린다(크래시 금지). 저장 코스
  목록에서 특정 행의 blob만 손상된 경우 그 행만 건너뛰고(로그) 나머지를 반환한다.

## 3. course-save — 동작 설계

### 자동 저장·복원 (초안)

- 트리거: attach 완료(탭 확정·그리기), undo, redo, clear, 왕복 삽입, 코스 불러오기 —
  즉 `CourseEditSession`의 상태가 바뀌는 연산 확정 시점. 추가 안전망: scenePhase
  background 진입 시.
- 앱 시작: 초안이 있으면 세션을 복원해 지도에 그대로 표시. 초기화하고 껐으면 빈 상태.
  사용자 개입 UI 없음 — "껐다 켜면 마지막 모습 그대로."
- `CourseEditSession`에 스냅샷 내보내기/복원 API를 추가한다(`snapshot()` /
  `restore(from:)` 계열). 직렬화 형식은 Domain의 `CourseDraft`. **복원은 엔트리 `id`를
  보존해야 한다** — 기존 `append`/`prepend` 경로를 재사용하면 id가 재발급되어 왕복
  anchor 참조가 끊긴다(§4). 복원 전용 경로로 entries를 그대로 재구성한다.

### 이름 붙여 저장 / 목록 / 불러오기 / 삭제

- 지도 화면 컨트롤 영역에 **저장 버튼**: 이름 입력 알럿 → `SavedCourse` 생성. 코스가
  비어 있으면 비활성.
- **목록 버튼**: sheet로 저장 코스 List 표시(이름·총 거리·저장일, 최신순).
  - 행 탭 = 불러오기. 작업 중 코스가 비어 있지 않으면 "지금 만들던 코스를 대체할까요?"
    확인 후 교체. 불러온 코스는 새 세션의 초기 상태가 된다(엔트리 시간순은 공간순으로
    재부여 — undo는 마지막 구간부터 제거).
  - 스와이프 삭제 + 확인 알럿.
- 저장/목록은 새 페이지가 아닌 sheet — 페이지 추가 없이 `CoursePlannerPage`에 부속.

## 4. roundtrip-insert — 동작 설계 (2026-07-08 정정)

### 연산 정의: "왕복 추가"

> 최초 설계(역+정 병합, 3× 거리, 모든 위치 허용)는 2026-07-08 실기기 QA에서 거리·연결
> 문제가 발견돼 아래 내용으로 교체됐다. 교체 배경은 어드바이저 검토로 확정: 코스 중간
> 구간은 반대쪽이 다른 구간과 이어져 있어 그 구간만 되짚으면 코스가 끊긴다 — 안전하게
> 되짚을 수 있는 쪽은 아무것도 이어지지 않은 코스의 자유 끝(맨 앞/맨 뒤)뿐이다.

구간 패널의 각 구간 줄에 왕복 버튼을 둔다. **코스의 자유 끝 구간(맨 앞 또는 맨 뒤)에서만**
활성화되며, 대상 구간(A→B)의 **역방향(B→A)만** 만들어 그 자유 끝에 붙인다(거리는 대상과
동일한 1×):

```
뒤쪽 끝 (append):   … → 이전구간(…→A) → s(A→B) → 왕복(B→A)          [코스가 A에서 끝남]
앞쪽 끝 (prepend):   왕복(B→A) → s(A→B) → 다음구간(B→…) → …          [코스가 B에서 시작]
```

- 중간 구간(양옆 모두 다른 구간과 연결된 구간)은 왕복 버튼이 비활성이다 — 반대쪽 연결이
  끊기기 때문에 구조적으로 불가능하다.
- 구간이 하나뿐인 코스(양쪽 끝이 같은 구간)는 뒤쪽 끝으로 취급해 append한다.
- 삽입 결과 코스의 시작 또는 끝 지점이 대상 구간의 반대쪽 점으로 이동한다(뒤쪽 끝이면 코스
  끝이 대상 구간 시작점으로, 앞쪽 끝이면 코스 시작이 대상 구간 끝점으로). 총 거리는 대상
  구간의 거리만큼 늘어 결과적으로 그 구간이 2×가 된다.
- 라우팅 호출 없음: `CourseSegment.reversed()`로 대상 구간의 확정 좌표를 뒤집은 순수 데이터
  연산.
- **"왕복임"의 표현**: `CourseSegment.roundTrip` 케이스(변경 없음, MVP11 최초 설계 유지).
- 버튼 비활성 가드: 중간 구간, 좌표가 2개 미만인 구간(방어적 경계), 삽입 후 코스 총 좌표
  수가 상한(20,000)을 넘는 경우.

### undo/redo와의 상호작용

- 삽입은 기존 편집 연산과 동일하게 redo 스택을 비운다.
- undo로 왕복을 제거하면 redo 스택에 들어간다. redo 시 재삽입 위치는 anchor(대상 구간
  엔트리 id) 기준이되, **anchor 앞/뒤 중 어느 쪽이었는지**(`anchorInsertsBefore`)도 함께
  저장해 뒤쪽 끝(anchor 뒤)·앞쪽 끝(anchor 앞) 모두 정확히 복원한다.
- 초안 직렬화(`CourseDraft`)에 `anchorInsertsBefore`를 포함해 재시작 후에도 일관되게
  동작한다. 이 필드가 없는 구버전 blob(2026-07-07 최초 설계 시점)은 손상이 아니라 `false`
  (= anchor 뒤, 옛 동작과 동일)로 기본 복원된다 — 옛 왕복 엔트리는 항상 append였으므로
  정확한 값이다.

### 패널 표시

- 왕복 세그먼트는 일반 구간과 같은 줄 형식에 왕복 표식(아이콘)과 거리를 표시. 색상은 기존
  색상 키 규칙(attach 생성 순서)을 그대로 따른다.
- 지도 렌더링은 기존 겹침 오프셋(MVP8)이 왕복 구간의 겹침을 그대로 처리 — 신규 렌더링
  코드 없음.

## 4.5. 전체 왕복 (2026-07-08 추가)

지금까지 그린 코스 **전체**를 뒤집어 맨 뒤에 단일 왕복 세그먼트로 붙이는 별개의 액션.
구간 단위 왕복(위 §4)과 달리 특정 구간을 고르지 않고, 컨트롤 바의 별도 버튼으로 코스
전체에 적용한다.

- 항상 코스의 열린 끝(현재 마지막 좌표)에서 시작하는 연산이라 anchor 추적 없이 구조적으로
  항상 연결이 유지된다 — 중간 구간 제약이 없다(적용 대상이 애초에 "코스 전체"이므로).
- 좌표: `PlannedCourse.coordinates`(세그먼트 경계 중복 제거된 전체 좌표)를 뒤집은 것.
  거리: 뒤집기 전 코스 총 거리와 동일(결과적으로 코스 총 거리 2×).
- undo/redo는 일반 append와 동일하게 동작한다(anchor 없음).
- 용도: 막다른 길·산책로처럼 왔던 길 그대로 돌아 나오는 코스를 손으로 다시 그릴 필요 없이
  버튼 한 번으로 완성. 구간 단위 왕복(특정 골목만 왕복)과는 별개 용도로 공존한다.

## 6. 테스트·검증 전략

- TDD (시뮬레이터 검증 가능 범위가 넓다):
  - `CourseEditSession`: 스냅샷 내보내기→복원 후 코스·undo 동작 동일성(라운드트립),
    **복원 시 엔트리 id 보존**(복원 후 왕복 undo→redo가 올바른 위치로), 왕복 삽입
    (중간/마지막/유일 구간/닫힌 코스/왕복 위에 왕복), 좌표 상한 가드.
  - 저장소 어댑터: in-memory `ModelContainer`로 초안 저장/로드/삭제, 코스
    저장/목록(최신순)/삭제, 손상 blob 복원 실패 시 nil 반환, 목록 중 한 행만 손상 시
    나머지 반환, 포맷 버전 필드 존재.
  - ViewModel: 편집 연산 후 초안 저장 호출 여부(모킹), 불러오기 시 세션 교체.
- 실기기 QA (시나리오 카드, 쉬운 말 규칙):
  - 코스를 그리다 앱을 완전히 종료 → 다시 열면 그대로인지 / 뒤로가기가 되는지.
  - 초기화 후 종료 → 빈 상태로 시작하는지.
  - 이름 저장 → 목록 확인 → 다른 코스 작업 중 불러오기(확인창) → 삭제.
  - 중간 구간 왕복 추가 → 지도 경로·총 거리(2배 증가) 확인 → undo 한 번에 원상복구.
- 커밋 전: 빌드·테스트·린트 3종 통과(기존 verification stamp 규칙).
