# run-record-save 설계 — 러닝 기록 저장 + 목록/상세 (MVP13 사이클 2)

2026-07-14. MVP13 "자유 러닝 트래킹 + 기록"의 마지막 마일스톤 ③.
사이클 1 스펙(`2026-07-13-run-tracking-design.md`)의 §2 저장 원칙과 §8 "③ 저장 스키마
확정"을 이어받아, 본 러닝 QA에서 회수한 실측 덤프(약 1km에 샘플 172~173개)를 근거로
스키마를 확정한다.

## 1. 범위

### 포함

- **자동 저장**: 러닝 종료(요약 화면 진입) 시 확인 없이 SwiftData에 저장
  (결정 2026-07-14, `project-decisions.md` — 러닝은 소장품이 아니라 활동 이력).
- **기록 목록**: 러닝 탭 대기 화면의 "기록" 버튼으로 진입. 행 = 날짜 + 거리 + 시간 +
  평균 페이스, 최신순. 밀어서 삭제(확인 알럿).
- **기록 상세**: 목록에서 탭 → 뛴 경로 지도 + 거리·시간·평균 페이스·고도 상승.

### 제외 (이후 결정)

- 기록 이름 붙이기/변경, 메모 — 자동 저장이라 이름 없음. 목록 제목은 시작 시각 기반.
- 통계(주간/누적 거리), "기록" 탭 승격 — 목록 입구는 되돌리기 쉬운 결정이라 지금은
  러닝 탭 안에 두고, 통계 기능이 커지면 그때 탭 분리(사용자 합의 2026-07-14).
- 진행 중 세션 복구(강제 종료 후) — 저장 기반이 생겼으므로 다음 사이클 후보(§8 참고).
- 코스 연동(계획 vs 실제 비교) — 다음 MVP. 단, 이번에 저장하는 원시 샘플이 그 원료다.

## 2. 저장 데이터 모델

### 필드 범위 — 원시 5필드 전부, 다운샘플링 없음 (결정 2026-07-14)

샘플마다 **시각·위도·경도·고도·속도** 5필드를 전부 저장한다. 정확도 2필드(hAcc/vAcc)는
필터 판정 전용이라 저장하지 않는다(사이클 1 §2에서 확정).

- 근거: QA 실측으로 저장량이 문제 아님을 확인 — 1km에 ~172샘플이면 10km 러닝도
  JSON 수백 KB 수준. 고도를 버리면 고도 프로필이 영영 불가, 속도를 버리면 구간 페이스
  차트를 좌표 차분 노이즈로 근사해야 한다. 용량이 나중에 문제가 되면 원본이 있는 한
  압축·솎아내기 등 최적화가 언제든 가능하지만 역방향은 불가.
- 다운샘플링(예: 5초당 1개)은 조기 최적화 + §2 원칙(재계산 가능성 보존) 충돌로 기각.
- **GPS 공백 구간 표시는 별도 저장 안 함**(사이클 1에서 이연된 결정의 종결) — 샘플마다
  시각이 있으므로 공백은 시각 간격에서 언제든 파생 가능. 저장할 것이 없다.

### 요약 컬럼 — 파생값 캐시 (결정 2026-07-14)

총 거리·시간·고도 상승을 blob 밖 **컬럼으로도** 저장한다. §2 "파생값은 저장하지 않고
계산한다" 원칙의 예외가 아니라 **목록 성능용 캐시** — 기록 수백 개의 목록을 열 때마다
전체 blob 디코드+재계산을 하지 않기 위한 것. 원시 샘플이 함께 있으므로 진실의 원천은
여전히 원시 트랙이고, 캐시는 재계산으로 검증 가능(왕복 QA에서 이미 실측 검증한 경로).
평균 페이스는 거리·시간에서 나오므로 컬럼 불필요.

### 스키마

코스 저장 패턴(`CourseRecord` + DTO 버전 blob, MVP11)을 그대로 따른다:

- **`RunRecordModel`**(@Model, 어댑터 내부 전용): `id`(unique) · `startedAt` ·
  `distanceMeters` · `durationSeconds` · `elevationGainMeters` · `payload: Data`.
- **`RunPersistenceDTO`**(어댑터 내부 Codable, `version = 1`): 샘플 배열
  (시각·위도·경도·고도·속도). 도메인 타입에 Codable을 직접 붙이지 않는 이유는 코스와
  동일 — 도메인 리팩터링이 기존 blob을 해독 불가로 만들지 않게.
- **미래 확장(additive)**: 심박·케이던스는 DTO에 스트림 배열 하나를 옆에 추가 +
  version 증가로 끝난다. 기존 blob은 그대로 읽힌다(§2 원칙의 실체).
- **도메인 타입 2개** — 요약 컬럼 캐시가 실제로 효과를 내려면 목록 경로가 blob을
  건드리지 않아야 하므로, 목록용과 상세용을 분리한다:
  - `SavedRunSummary`: `id` · `startedAt` · `distanceMeters` · `duration` ·
    `elevationGainMeters`. 전부 컬럼에서 나옴 — blob 미디코드. 평균 페이스는 계산
    프로퍼티.
  - `SavedRun`: 요약 + 샘플 배열(시각·좌표·고도·속도). 상세 화면 단건 조회 전용.
    `RunSample`을 재사용하지 않는 이유: `RunSample`의 정확도 2필드는 전송용이라
    저장에서 제외되는데, 로드 시 가짜 정확도 값을 채워 넣는 왜곡을 피한다.

### 저장소 분리

**러닝 기록은 코스와 별도 스토어 파일**(`TraceRunStore.store`)에 저장한다.

- 근거: 기존 코스 스토어(`TraceCourseStore.store`)에 모델을 추가하면 실사용자 데이터가
  있는 스토어에 스키마 변경이 가해진다 — 새 파일이면 마이그레이션 리스크가 0.
  두 자산은 수명·용도도 독립(코스=계획 자산, 기록=활동 이력).
- 대안 기각: 같은 컨테이너 합류(위 리스크), 기록별 개별 JSON 파일(SwiftData 패턴이
  이미 자리 잡음 — 손상 폴백·테스트 관례 재사용이 이득).

## 3. 자동 저장 흐름

- **트리거**: `RunSession`이 summary 상태로 전이할 때(사용자 길게 눌러 종료 + 권한
  회수로 인한 자동 종료 모두). 유효 샘플이 0개면 요약 자체가 없으므로(기존 동작)
  빈 기록은 생기지 않는다.
- **아주 짧은 러닝도 일단 저장** — 자동 저장이라 "버릴까요?"를 물을 타이밍이 없고,
  잘못 저장된 것은 목록에서 삭제하면 된다. 임계값(최소 거리/시간) 도입은 "저장됐다고
  생각했는데 없어진" 놀람을 만들므로 기각.
- **저장 주체**: `RunSession`에 `RunRecordRepositoryProtocol`을 주입해 세션이 종료
  시점에 직접 저장한다. 사이클 1 §4는 저장을 "세션의 소비자"로 분류했지만
  (`RunActivityController`식 관찰자), 저장은 잠금화면과 달리 ① 세션 수명 이벤트(종료)와
  원자적으로 묶이고 ② 실패·재시도가 요약 화면 상태의 일부라서, 관찰자 분리보다
  오케스트레이터의 책임이 맞다. Application이 Domain 포트에 의존하는 것은 기존 레이어
  규칙 안이다.
- **저장 상태**: `RunSession`에 저장 상태(진행/성공/실패)를 노출, 요약 화면에 표시.
  실패 시(디스크 오류 등) 알림 + **재시도 버튼**, 세션 데이터는 요약을 닫기 전까지
  유지된다. 실패 상태에서 사용자가 요약을 닫으면 데이터 소멸을 수용(안내 후).

## 4. 화면 — 목록/상세

- **진입**: 러닝 탭 대기(idle) 화면에 "기록" 버튼 → 시트로 목록 표시(코스 탭의 저장
  코스 목록 시트 선례 재사용). 시트 안 NavigationStack으로 목록 → 상세 push.
- **목록**: 최신순. 행 = 시작 시각(날짜·시간) + 거리 + 시간 + 평균 페이스 — 전부
  요약 컬럼에서 나오므로 blob 디코드 없음. 밀어서 삭제 + 확인 알럿. 빈 목록이면
  안내 문구("아직 기록이 없어요" 류).
- **상세**: 경로 전체 카메라 핏 지도(표시 전용 SwiftUI `Map` — 요약 화면과 동일 접근,
  폴리라인은 blob의 샘플 좌표) + 거리·시간·평균 페이스·고도 상승. 요약 화면과 같은
  구성이므로 스탯 패널 컴포넌트 재사용.
- 러닝 중(acquiring/tracking)에는 기록 버튼이 없는 화면이므로 동시 진입 문제 없음.
- 디자인은 기존 디자인 시스템 토큰·컴포넌트로 조립, UI 세부 배치는 진행 중 확정
  (기존 합의: UI 세부는 확인 없이 진행).

## 5. 아키텍처 — 기존 레이어 패턴 확장

- **Domain — `Trace/Domain/RunTracking/`**: `SavedRunSummary`/`SavedRun`(엔티티, §2),
  `Protocol/RunRecordRepositoryProtocol`(포트) — `save(_ run:)` /
  `fetchSummaries()`(최신순, 컬럼만 — blob 미디코드) / `fetchRun(id:)`(단건, blob
  디코드 — 해독 실패 시 nil) / `delete(id:)`. `CourseRepositoryProtocol`의 미러이되
  목록/상세 조회가 분리된 점만 다르다(§2 캐시가 무의미해지지 않게).
- **Infrastructure — `Trace/Infrastructure/Persistence/SwiftData/`**:
  `SwiftDataRunRecordRepository`(actor, 별도 스토어 파일). 컨테이너 생성 실패 정책은
  코스 어댑터와 동일 — ① 정상 생성 → ② 손상 스토어 백업 후 재생성(자산 즉시 삭제
  금지) → ③ in-memory 폴백 → ④ nil(no-op/throw). 런치 크래시 금지. 컨텍스트는 actor
  실행기 위에서 lazy 생성(affinity 규칙, 2026-07-08 QA 교훈).
- **Application**: `RunSession`에 리포지토리 주입 + 저장 상태 노출(§3).
  `DependencyContainer.live()/uiTesting()`에 등록(uiTesting은 in-memory).
- **Pages — `Trace/Pages/RunPage/`**: 목록/상세 화면 + ViewModel.

## 6. 에러 처리

- **저장 실패**: 요약 화면에 실패 표시 + 재시도, 데이터는 요약 닫기 전까지 유지(§3).
- **스토어 손상**: 백업 후 재생성 폴백 체인(§5) — 코스 정책 재사용.
- **blob 해독 실패/미래 버전**: 목록은 컬럼 기반이라 그대로 표시된다(거리·시간은
  유효). 상세 진입 시 `fetchRun`이 nil이면 **우아한 강등** — 요약 숫자는 컬럼 값으로
  표시하고 지도 영역에만 "경로를 불러올 수 없음" 안내. 행 삭제 금지(사용자 자산 —
  코스와 동일 정책, 사용자가 직접 삭제는 가능).
- **삭제 실패**: 알럿 표시, 목록 상태는 실제 스토어와 재동기화(fetch 재실행).

## 7. 테스트·QA

- **단위**(기존 XCTest 패턴): 리포지토리 저장→불러오기 왕복(in-memory)·최신순 정렬·
  삭제, DTO 인코드/디코드 + 손상 blob·미래 버전에서 `fetchRun` nil 반환(목록 요약은
  유효 유지), `RunSession` 자동 저장 트리거(목 리포지토리로 성공/실패/재시도),
  `SavedRunSummary` 평균 페이스 계산.
- **통합(시뮬레이터)**: 러닝 시작→종료→요약(저장 표시)→목록에 나타남→상세 진입→삭제
  흐름. UI 테스트 컨테이너는 in-memory 리포지토리.
- **실기기 QA**: 사이클 1 스펙 §7 계획대로 **가벼운 확인 — 저장·목록 조작**(뛸 필요
  없음). 짧은 산책 1회로 저장→재실행 후 목록 유지(영속성)까지 확인하면 충분.
  시나리오 카드 체크리스트(`docs/qa/`, `testing.md` 템플릿).

- **기록 이름/메모, 통계·"기록" 탭 승격** — 사용량이 말해줄 때.
