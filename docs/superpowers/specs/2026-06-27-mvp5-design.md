# MVP5 설계 — 경로 연결 + 조작 개선

날짜: 2026-06-27

## 범위

백로그 open 항목 3개를 MVP5로 묶는다.

| 마일스톤 슬러그 | 내용 |
|---|---|
| `path-stitching` | 탭↔그리기 양방향 경로 이어붙이기 (세그먼트 배열 모델) |
| `two-finger-pan-ux` | 그리기 모드 두 손가락 이동 UX 보완 |

- 세그먼트 개별 삭제/편집
- 코스 저장

## 마일스톤 1 — path-stitching

### 문제

ViewModel이 탭 상태(`startCoordinate`, `destinationCoordinate`)와 그리기 상태(`drawnStrokes`, `accumulatedCoordinates`)를 완전히 별개 변수로 관리하고, `toggleDrawingMode()`에서 모두 초기화한다. 탭으로 만든 경로와 그리기로 만든 경로를 이어붙이는 것이 구조적으로 불가능하다.

### 설계 방향: 세그먼트 배열 모델

경로를 "조각(segment)의 배열"로 표현한다. 각 조각은 탭 leg이거나 그리기 leg이다.

**신규 파일: `Domain/CoursePlanning/Entity/CourseSegment.swift`**

```swift
// 각 세그먼트는 leaf 데이터만 들고 있다. PlannedCourse를 참조하지 않는다.
enum CourseSegment {
    case tapped(coordinates: [CourseCoordinate], distanceMeters: Double)
    case drawn(coordinates: [CourseCoordinate], distanceMeters: Double)
}

extension CourseSegment {
    var coordinates: [CourseCoordinate] {
        switch self {
        case .tapped(let coords, _), .drawn(let coords, _): return coords
        }
    }

    var distanceMeters: Double {
        switch self {
        case .tapped(_, let d), .drawn(_, let d): return d
        }
    }
}
```

**`PlannedCourse` 변경:**

```swift
struct PlannedCourse: Equatable, Sendable {
    var segments: [CourseSegment]

    // 세그먼트 사이 connector(연결 구간)도 별도 세그먼트로 포함되므로 단순 flatten
    var coordinates: [CourseCoordinate] {
        segments.enumerated().flatMap { i, seg in
            i == 0 ? seg.coordinates : Array(seg.coordinates.dropFirst())
        }
    }

    var distanceMeters: Double {
        segments.reduce(0) { $0 + $1.distanceMeters }
    }
}
```

### 세그먼트 사이 connector 처리

탭→그리기, 그리기→탭 전환 시 기존 끝점과 새 시작점 사이에 연결 구간(connector)이 필요하다.  
connector는 별도 세그먼트로 segments 배열에 포함시킨다. 이렇게 하면 `coordinates` computed property가 단순 flatten만으로 구멍 없는 전체 경로를 반환한다.

### 탭→그리기 전환 시 StrokeDirectionResolver seed

`StrokeDirectionResolver`는 `accumulatedCoordinates.first/last`를 기준점으로 사용한다.  
탭→그리기 전환 시 `accumulatedCoordinates`가 비어 있으면 resolver가 `.initial`을 반환해 이어 그리기가 아닌 새 경로로 시작된다.

**해결:** 탭→그리기 전환 시 기존 tapped 세그먼트의 좌표를 `accumulatedCoordinates`의 초기값으로 seed한다.

### 데이터 흐름 (양방향 예시)

```
탭으로 A→B 생성
  → segments = [.tapped(AB)]

그리기 모드 전환
  → accumulatedCoordinates = AB 좌표로 seed
  → B 좌표를 그리기 시작점으로 자동 연결

그리기로 B→C (connector B'→B 포함)
  → segments = [.tapped(AB), .drawn(connector), .drawn(BC)]

탭 모드 전환
  → C 좌표가 startCoordinate로 자동 설정

탭으로 C→D
  → segments = [..., .tapped(CD)]
```

### undo 동작

세그먼트 배열의 **마지막 항목을 제거**한다. 탭 세그먼트든 그리기 세그먼트든 동일하게 동작한다.  
기존 `undoLastStroke()`는 세그먼트 모델에 맞게 재구현한다.

### ViewModel 변경 핵심

- `segments: [CourseSegment]`를 단일 진실 출처로 추가
- `toggleDrawingMode()` — 초기화 대신 현재 상태를 segment로 보존
  - 탭→그리기: tapped segment 마지막 좌표를 `accumulatedCoordinates`에 seed
  - 그리기→탭: drawn segments 보존, 마지막 좌표를 `startCoordinate`로 설정
- `undoLastStroke()` — `segments.removeLast()` 후 `accumulatedCoordinates` / `course` 재동기화
- `clear()` — 전체 초기화 (기존과 동일)

### 에러 처리

- 세그먼트 이어붙이기 중 스로틀 에러 → 기존 segments 보존, 마지막 stroke만 롤백
- 모드 전환 시 이전 세그먼트는 에러와 무관하게 항상 보존

### 테스트

- `CourseSegment` 단위 테스트 — `coordinates` / `distanceMeters` computed property
- `PlannedCourse` 단위 테스트 — segments flatten, distanceMeters 합산, connector 포함 시 구멍 없음
- ViewModel 모드 전환 테스트 — 탭→그리기 전환 시 seed 여부, 기존 segments 보존 여부
- undo 테스트 — 탭/그리기 혼합 상태에서 마지막 세그먼트 제거 후 상태 정합성
- 양방향 이어붙이기 통합 시나리오 테스트

## 마일스톤 2 — two-finger-pan-ux

### 문제

그리기 모드에서 두 손가락 지도 이동 시 세 가지 버그가 있다:

1. **방향 반전** — `handleTwoFingerPan`에서 위도(위아래)와 경도(좌우) 모두 부호가 반대
2. **3D 틸트 진입** — `updateUIView`에서 그리기 모드 진입 시 `isPitchEnabled = false` 누락
3. **그리기 오염** — `drawGR.maximumNumberOfTouches = 1`이지만 두 번째 손가락 도착 전에 그리기가 시작됨

### 수정 방향

1. `handleTwoFingerPan`: 위도/경도 **양축 모두 부호 반전** (실기기에서 상하·좌우 양방향 검증 필요)
2. `updateUIView`: 그리기 모드 진입 시 `mapView.isPitchEnabled = false`, `mapView.isRotateEnabled = false` 동시 적용, 복귀 시 `true`
3. `handleDraw`: touch count 확인 후 2개 이상이면 stroke 취소 처리

### 평가 방침

세 가지 수정 후 실기기에서 두 손가락 이동이 자연스러운지 평가한다. 수정 후에도 UX가 근본적으로 어색하면 "그리기 일시 중단 버튼" 방식으로 전환을 검토한다.

## 마일스톤 3 — test-ios-versions

### 문제

`@Observable` ViewModel을 XCTest로 테스트할 때 iOS 18.x 시뮬레이터에서 malloc 크래시 발생. 현재는 iOS 26.5 시뮬레이터로 우회 중. 앱 최소 지원 버전(iOS 17)과 테스트 런타임이 불일치하는 상태.

현재 기록(`docs/solutions/workflow-issues/ios18-observable-malloc-crash.md`)의 원인 분석과 해결책은 경험적 관찰에 기반하며 공식 확인되지 않았다.

### 수행 흐름

1. **리서치** — Swift 이슈 트래커(swiftlang/swift), Swift Forums, Apple 릴리즈 노트, 커뮤니티 보고서를 통해 다음을 확인:
   - 이 버그의 정확한 원인 (Swift 런타임인지, `@Observable`+`@MainActor` 조합인지, Xcode 버전 의존인지)
   - Apple이 특정 iOS/Swift 버전에서 공식 수정했는지 여부
   - 커뮤니티에서 검증된 근본 수정 방법
2. **원인 확정** — 리서치 결과를 바탕으로 현재 기록을 업데이트하거나 수정
3. **수정 방법 결정** — 리서치 결과에 따라 근본적인 수정 방법 선택 (`nonisolated deinit`은 후보 중 하나일 뿐)
4. **적용 및 검증** — iOS 18 시뮬레이터에서 실제 테스트 통과 확인
5. **문서 업데이트** — `docs/solutions/` 기록을 팩트 기반으로 갱신

### 완료 조건

- 리서치 결과 **우리 코드로 근본 수정 가능**: 수정 적용 + iOS 18 시뮬레이터 테스트 통과로 완료
- 리서치 결과 **Apple 런타임 버그**: 원인 규명 + 출처 명시 문서화로 완료 (수정 불가 사실 자체가 산출물)
