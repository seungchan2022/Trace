# 겹치는 경로 좌표 오프셋 렌더링 — 겹침 감지 + 표시 전용 평행 이동

날짜: 2026-07-02
상태: 설계 확정 (MVP8 마일스톤 2/2)

## 배경

코스가 자기 자신 위를 지나가면(전형적으로 왕복 구간) 나중에 추가된 폴리라인이 이전 구간을
완전히 덮는다. 이때 이전 구간의 거리 라벨은 원래 자리에 그대로 떠 있으므로, **라벨의 색과
눈에 보이는 선의 색이 어긋나** 라벨이 어느 구간 것인지 혼동된다 (MVP7 실기기 피드백).

MVP8 킥오프(2026-07-02)에서 검토한 대안과 결정:

- 라벨에 구간 번호 표기 → **기각**: 구간 번호는 출발 기준 공간 순서라 prepend 시 전부 밀린다.
  MVP7의 색상 문제(공간 순서 ≠ 생성 순서)와 같은 뿌리이며, 번호는 colorKey처럼 분리로
  안정화할 수 없다.
- 점선/반투명 등 스타일 트릭 → **기각**: 혼동의 뿌리(두 구간이 동시에 안 보임)를 못 건드린다.
- 단일 코스 색 + 방향 화살표(β) → **백로그 보관**: α로도 혼동이 남으면 검토.
- 화면 포인트 기준 오프셋(커스텀 렌더러) → **백로그 보관**: α의 줌아웃 병합이 실사용에서
  문제되면 업그레이드.
- **좌표 오프셋(α) → 채택**: 겹치는 부분의 표시 좌표를 몇 미터 옆으로 밀어 두 구간이 나란히
  보이게 한다. 렌더러를 건드리지 않는 순수 기하 계산이라 마일스톤 1개 규모로 들어온다.

## 접근

렌더링 직전에 **표시 전용 좌표 변환 단계**를 둔다. 도메인 데이터(`CourseSegment.coordinates`)는
불변이고, 거리 계산·라우팅·저장(미래)은 전부 원본 좌표를 쓴다. 오프셋은 지도에 그릴 때만 적용된다.

```
CourseEditSession.segments (원본, 불변)
        │
        ▼
OverlapOffsetResolver.displayCoordinates(for:)   ← 순수 함수, 이번에 신설
        │  겹침 감지 + 수직 오프셋 적용
        ▼
MapViewRepresentable  ← 폴리라인·거리 라벨을 표시 좌표로 그림
```

## OverlapOffsetResolver (신설)

- 위치: `Trace/Pages/CoursePlannerPage/` (표시 전용 프레젠테이션 로직 — `SegmentPalette`와 같은 층).
  MapKit을 import하지 않고 도메인 타입 `CourseCoordinate`로만 계산해 유닛 테스트를 쉽게 유지한다.
- 입력: `[CourseSegment]` (공간 순서), 출력: 세그먼트별 표시 좌표 배열 `[[CourseCoordinate]]`.
- 파라미터(초기값, 실기기 QA로 조정): 겹침 감지 임계 `10m`, 오프셋 간격 `4m`.

### 겹침 감지

- 세그먼트를 앞에서부터 순회하며, 각 세그먼트의 각 점에 대해 **앞서 처리한 세그먼트들의
  폴리라인(가장 가까운 선분)까지의 수직 거리**를 구한다. 임계(10m) 이내면 그 점은 "겹침"으로 표시한다.
- 점-대-점이 아니라 **점-대-선분**인 이유: 라우팅 폴리라인은 정점이 굴곡·교차점에만 있어
  직선 도로에서는 희소하다. 조밀한 그리기 스트로크(.drawn)가 라우팅 구간(.tapped) 위를 정확히
  되짚어도 점끼리는 10m 밖일 수 있으므로, 점-대-점 비교는 소스가 섞이면 겹침을 놓친다.
  선분 기준이면 세그먼트 소스와 정점 밀도에 무관하게 동작한다. 평행한 옆 도로(통상 20m 이상
  이격)는 여전히 임계에 걸리지 않는다.
- 연속된 겹침 점들을 하나의 "겹침 구간(run)"으로 묶는다. 잡음성 단발 겹침(1~2점)은 무시한다.

### 오프셋 적용

- 오프셋은 **뒤에 오는(겹침으로 판정된) 구간에만** 적용하고, 먼저 지나간 구간은 원본 위치(도로 위)에
  남는다. 겹침 점을 **그 점의 로컬 진행 방향(이웃 점으로 계산)에 수직, 진행 기준 오른쪽**으로 민다 —
  왕복이면 돌아오는 선이 자기 진행 방향의 오른쪽(= 가는 선 기준 반대 차선 쪽)으로 밀려,
  "돌아올 땐 길 반대편"이라는 물리적 직관과 일치한다.
- 같은 지점을 세 번 이상 지나는 경우: 그 점에서 겹치는 앞선 구간 수 `n`에 비례해 `4m × n`으로 민다.
- **테이퍼(보간)**: run 경계(0 → 4m)뿐 아니라 **run 내부에서 오프셋 크기가 변하는 모든 지점**
  (n 전환에 의한 4m → 8m 포함)에서 몇 점에 걸쳐 선형 보간해, 직각 꺾임과 단차를 막는다.
  (8자형·부분 재주행 코스에서는 세 번째 통과가 run 중간에서 시작될 수 있다.)
- **핀 지점 예외**: 코스의 첫 좌표와 마지막 좌표(출발/도착 핀이 찍히는 지점)는 오프셋 대상에서
  제외하고, 그 경계에서도 테이퍼가 0으로 수렴하게 한다. 핀은 원본 좌표에 남으므로(핀 로직 변경
  없음), 이 예외가 없으면 왕복 코스에서 선 끝과 도착 핀이 눈에 띄게 어긋난다.

### 거리 라벨

- 거리 텍스트는 원본 좌표 기반 거리(`segment.distanceMeters`) 그대로 사용한다(변경 없음).
- 라벨 annotation의 위치는 **표시 좌표**의 중간점으로 잡아 라벨이 밀린 선 위에 얹히게 한다.

## MapViewRepresentable 연동

- `updateUIView`의 기존 세그먼트 스냅샷 비교(`lastSegmentSnapshots`)를 그대로 활용해,
  **세그먼트가 실제로 바뀐 경우에만** resolver를 호출하고 오버레이를 다시 그린다.
  카메라 이동마다 재계산하지 않는다.
- 폴리라인 생성·거리 라벨 위치만 resolver의 표시 좌표를 쓰도록 바꾼다. 색상(colorKey)·선택
  하이라이트·핀 로직은 변경 없음.

## 성능

- 점별 비교는 O(전체 점 수 × 앞선 점 수)이지만, 코스 좌표는 수백~수천 점 규모라 문제되지 않는다.
  실측에서 느리면 그때 격자 해싱 등으로 최적화한다(추측 선반영 금지).

## 알려진 한계 (의도된 트레이드오프)

- **줌아웃 병합**: 오프셋이 실거리 기준(기본 4m, 다중 겹침 시 4m × n)으로 고정이라 지도를 축소하면 두 선이 다시 한 줄로 보인다.
  단, 거리 라벨은 화면 고정 크기 annotation이라 병합 배율에서도 그대로 읽히므로, 코스 전체를
  조망하는 배율에서는 원래 문제(라벨-선 색 어긋남)가 재현될 수 있다. 이 재현 여부를 실기기 QA에서
  명시적으로 확인하고, 백로그의 "화면 포인트 기준 오프셋" 업그레이드 트리거 판정 기준으로 삼는다.
- **선이 도로에서 살짝 벗어남**: 표시 좌표가 실제 경로에서 최대 4m 옆이다. 지하철 노선도식
  도식화로 의도된 것.

## 테스트

- 유닛 (resolver):
  - 왕복(같은 좌표 역순) 세그먼트 → 겹침 구간 전체에 오프셋 적용, 첫 세그먼트는 원본 유지
  - 겹치지 않는 세그먼트 → 표시 좌표 = 원본 좌표
  - 부분 겹침 → 겹친 run에만 오프셋, 경계 테이퍼 적용
  - 3중 겹침 → 배수 오프셋(4m × n) + n 전환 지점의 연속성(단차 없음) 검증
  - 희소 정점 직선 도로 왕복(정점 간격 큼) → 점-대-선분 감지로 겹침 검출
  - 그리기 스트로크가 라우팅된 구간 위를 지나는 경우(소스 혼합) → 겹침 검출
  - 코스 첫/끝 좌표(핀 지점)는 오프셋 제외
  - 원본 `CourseSegment`가 변형되지 않음(불변성)
- 실기기 QA:
  - 왕복 코스에서 두 선이 나란히 보이고 각 라벨이 자기 선 위에 있는지
  - 왕복 코스에서 도착 핀과 선 끝이 일치하는지
  - 코스 전체 조망 배율에서 라벨-선 색 어긋남이 재현되는지 (화면 포인트 오프셋 백로그 트리거 판정)
  - 줌 배율별 체감(어느 배율부터 병합돼 보이는지 기록 — 백로그 트리거 판단 자료)
  - 감지 임계·오프셋 간격 초기값(10m/4m)의 체감 조정

## 범위 밖

- 단일 코스 색 + 방향 화살표(β) — 백로그 (트리거: α 후에도 혼동 잔존)
- 화면 포인트 기준 오프셋 — 백로그 (트리거: 줌아웃 병합이 실사용 문제로 확인)
- 구간 패널·핀치 줌 — MVP8 마일스톤 1 (`2026-07-02-course-ux-polish-design.md`)

## Deferred / Open Questions

### 2026-07-02 ce-doc-review에서 이월 (플랜 작성 시 결정)

- **스트로크 확정 시 스냅 전환** — 겹치는 스트로크를 그리다 손을 떼면 확정된 선이 최대 4m 옆으로
  즉시 이동한다. 즉시 스냅으로 둘지, 짧은 애니메이션(페이드/슬라이드)을 넣을지. (design-lens)
- **"누가 도로 위에 남는가"의 판정 순서** — 현재 설계는 공간 순서 기준이라, 겹치는 구간이
  prepend되면 기존에 도로 위에 있던 선·라벨이 4m 옆으로 점프한다. colorKey처럼 생성(attach)
  순서 기준으로 바꿔 prepend 안정성을 확보할지. (feasibility, adversarial)
- **테이퍼 보간 단위** — "몇 점에 걸쳐"를 점 개수 기준으로 하면 그리기(조밀)/라우팅(희소)
  정점 밀도 차이로 테이퍼 실거리가 들쭉날쭉해진다. 점 개수 vs 실거리(m) 기준 결정. (feasibility, adversarial)
