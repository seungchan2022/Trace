# MVP10 — 제스처 정합성 설계

2026-07-04. 백로그 open 4건(더블탭 줌 시 포인트 찍힘, 탭 직후 빠른 드래그 원핑거 줌 오발동,
그리기 시작점 실거리 판정, 두손가락 제스처 경쟁)을 하나의 제스처 모델로 묶어 해결한다.

## 배경 — 왜 우리 앱만 섞이는가

iOS 지도에서 "탭"으로 시작하는 제스처는 세 가지다: 싱글탭, 더블탭(줌인), 탭+두 번째 터치
유지 드래그(원핑거 줌). 셋 다 첫 탭까지는 동일한 입력이라, 첫 탭이 끝난 시점에는 정체를 알 수
없고 ~0.3초 안에 두 번째 입력이 오는지로만 판별된다.

현재 Trace는 탭 종료 즉시(0초) 코스 포인트를 확정한다. 그래서 그 탭이 줌 제스처의 첫 탭이었어도
포인트가 이미 찍혀 "줌도 되고 포인트도 찍히는" 섞임이 생긴다. 리서치 결과(2026-07-04):

- 애플·구글 지도는 임의 좌표에 핀을 만드는 무거운 액션을 싱글탭에 걸지 않는다(롱프레스).
  POI 탭 선택 같은 가벼운 액션조차 내부적으로 더블탭 판별을 기다린 뒤 확정한다.
- 루트 플래너(Komoot)는 탭 → 확인 단계의 2단계 커밋을 쓴다.
- UIKit 표준 해법은 `싱글탭GR.require(toFail: 더블탭GR)` — 판별 지연(~0.3초)을 업계가 수용한다.
- 즉 "임의 좌표 + 싱글탭 + 즉시 확정" 조합을 쓰는 메이저 앱은 없다. 탭 자체는 러닝 코스
  스케치 특성상 유지하고, **즉시 확정을 보류 확정으로** 바꾸는 것이 이번 설계의 핵심이다.

기존 시도의 교훈(재발 방지):

- 기본 `UITapGestureRecognizer(2탭)` 기반 `require(toFail:)`는 탭 간 위치를 검사하지 않아
  서로 다른 위치의 정상 연속 탭까지 더블탭으로 오인했다
  (`docs/solutions/design-patterns/uikit-double-tap-require-to-fail-location-agnostic.md`).
  → 판별기는 반드시 **위치 검사**(멀면 즉시 실패 전이)를 포함한다.
- MapKit 내장 인식기를 `makeUIView` 시점에 찾는 방식은 등록 전이라 실패했다.
  내장 인식기 공존은 `shouldRecognizeSimultaneouslyWith` = true 방식을 쓴다.
- `isZoomEnabled = false`는 사실상 핀치만 막고 더블탭 줌은 남는다는 보고가 있어,
  "특정 줌 제스처만 선택적으로 끄기"는 채택하지 않는다.

## 범위

- 변경: 터치를 코스 조작으로 확정하는 시점/기준 (View 레이어 + ViewModel 신호).
- 불변: 코스 데이터 모델, 라우팅, `CourseEditSession` attach 규칙(MVP9), 구간 패널·렌더링.

## 마일스톤 1 — tap-pending-commit: 탭 보류 확정

### 동작 사양 (탭 모드)

| 입력 | 결과 |
|---|---|
| 싱글탭 | 임시 마커 즉시 표시 → ~0.3초 뒤 확정, 라우팅 시작 |
| 더블탭 (같은 자리 빠른 두 탭) | 줌인만. 임시 마커 제거, 코스 변화 없음 |
| 탭 + 두 번째 터치 유지 드래그 | 원핑거 줌만. 임시 마커 제거, 코스 변화 없음 |
| 서로 다른 위치 빠른 연속 탭 | 둘 다 정상 포인트 (위치 검사로 더블탭 오인 방지) |
| 출발/도착 핀 탭 (왕복 등 `pinHit`) | 동일 — 확정 시점에 판정 실행 |
| 탭 직후 텀을 둔 드래그 | 팬 (현행 유지) |

### 구조

- `MapViewRepresentable.Coordinator`에 **탭 판별기**를 둔다:
  - 첫 탭 종료 즉시 "보류 탭(좌표)" 신호 발행 → 임시 마커 표시용.
  - ~0.3초 내 같은 자리 두 번째 탭 또는 두 번째 터치 유지 드래그 감지 → "취소" 신호.
  - 아무것도 이어지지 않으면 → "확정" 신호.
  - UIKit `require(toFail:)` + 위치 검사(멀면 즉시 `.failed` 전이) 조합.
    내장 인식기와는 동시 인식 허용으로 공존. 세부 구현(커스텀 GR 서브클래스 vs 핸들러 레벨
    타이머)은 플랜 단계에서 확정하되, 위 신호 3개(보류/확정/취소) 인터페이스는 불변.
- ViewModel은 보류/확정/취소 3신호만 받는다 (MapKit 비의존 유지).
  확정 신호가 기존 `handleMapTap` 흐름으로 들어간다.
- 기존 임시방편인 시간(0.35s)+거리(40pt) 자체 디바운스는 판별기로 대체·삭제.

### 임시 마커

- 확정 마커와 같은 모양으로 즉시 표시 — 확정 순간 겉모습 변화 없음.
- 줌 제스처 시 잠깐 보였다 사라지는 깜빡임이 실기기 QA에서 거슬리면 **표시만 끈다**
  (내부 보류→확정/취소 구조는 동일, 애플 지도처럼 0.3초 뒤 등장). 기본값: 표시.

## 마일스톤 2 — draw-start-pixel-snap: 그리기 시작점 화면 기준 판정

- 현재: `snappedStrokeStart`가 실거리 20m(`connectionThresholdMeters`)로 코스 끝점 근접을
  판정 → 줌아웃 시 화면상 바로 옆인데 51m라 안 붙는 문제 (2026-07-04 실기기 확인).
- 변경: 그리기 시작 순간 시작점↔출발/도착 핀의 **화면 거리 24pt**(탭 모드 `pinHit`와 동일
  기준)로 판정하고, 해당되면 "이 핀에서 시작" 힌트를 스트로크와 함께 ViewModel로 전달.
- ViewModel은 힌트가 있으면 시작 좌표를 그 핀의 정확한 좌표로 치환. 기존 실거리 20m 판정
  로직(`snappedStrokeStart`)은 삭제.
- `CourseEditSession` attach 규칙(20m)은 불변 — 치환된 좌표가 정확히 일치하므로 자동 통과.
- 효과: 줌 레벨과 무관하게 "눈에 가까우면 이어진다".

## 마일스톤 3 — two-finger-gesture-tuning: 두손가락 제스처 경쟁 (그리기 모드)

- 현재: 커스텀 두손가락 팬 GR과 네이티브 두손가락 탭 줌아웃이 delegate 조정 없이 공존,
  빠른 연속 조작 시 서로 방해 (2026-07-03 실기기 확인).
- 변경: `UIGestureRecognizerDelegate` 동시 인식 허용 / 실패 요구 관계를 조정하고 실기기 튜닝.
- 성격상 실험적 마일스톤 — 수용 기준: "두손가락 탭줌아웃과 두손가락 팬을 빠르게 번갈아 해도
  각각 정상 동작". 시뮬레이터로는 판정 감각 확인이 어려워 실기기 QA 필수.

## 검증

- 유닛 테스트: 보류→확정/취소 상태 전이(시간 주입 가능하게), 그리기 시작 힌트 치환.
  기존 "서로 다른 두 좌표 연속 탭" 테스트는 회귀 방지용으로 유지.
- 실기기 QA: 동작 사양 표의 각 행 + 백로그 4건 재현 시도. 더블탭·원핑거 줌은 시뮬레이터
  재현이 어려워 실기기 확인 필수.
- 커밋 전 빌드/테스트/린트 통상 절차 (`docs/agent-rules/testing.md`).

## 하지 않는 것 (YAGNI)

- 롱프레스 포인트 추가, Komoot식 확인 시트 — 탭 스케치 UX를 유지하기로 결정.
- 특정 네이티브 줌 제스처의 선택적 비활성화 — 신뢰 불가로 기각.
- attach 규칙·세그먼트 모델 변경 — MVP9 결과 유지.
- 구간 패널 prepend 스크롤 보정 — 백로그 보류 유지.
