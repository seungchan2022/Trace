## 항목 1: 탭/시트/페이지 구조 개편

### 동기

지금 앱은 "러닝앱 + 지도앱"이 한 화면 문법에 섞여 있는 느낌이라 방향이 흐릿하다. Trace는 이미 **두 독립 기둥**(코스 탐색 / 러닝)으로 결정돼 있으므로(`project-decisions.md`), UI도 기둥별로 문법을 나눠 그 구조를 그대로 드러낸다.

## 핵심 원칙

> **"지도 + 시트 + 탭" 삼단 배치는 코스 탐색 탭 전용. 나머지는 전체화면 페이지. 글래스 모피즘은 전면 배제(불투명 유지).**

## 1. 앱 루트

- **커스텀 탭바**를 하단 고정. 시스템 탭바 안 씀 — iOS 26에서 시스템 탭바가 글래스로 렌더되어 원하는 모양이 안 나오기 때문.
- 탭을 누르면 **화면 전체가 전환**된다(참고한 "나의찾기"처럼 시트 콘텐츠만 바뀌는 게 아님).
- **지도는 앱 루트에 두지 않는다.** 지도는 코스 탭 *내부에만* 존재.

### 참고 리소스와의 차이(중요)

참고한 "나의찾기" 앱은 **탭바가 시트 *안에* 붙어 있어서**(탭바 = 시트의 일부) 탭을 눌러도 지도는 그대로고 시트 내용만 바뀐다. 그래서 그 영상은 "TabView를 시트 안에 넣기"의 곡예(Option 1/2/3)를 다룬다. **우리 구조는 다르다:** 탭바는 *루트*에 있고 시트는 *코스 탭 안에만* 있다 — 두 요소가 다른 층에 있으므로 그 곡예는 우리에게 해당하지 않는다. 따라서 참고 프로젝트의 Option 3(UIKit 글래스 복원)는 폐기.

## 2. 코스(탐색) 탭 — "나의찾기" 룩

- 지도가 화면을 꽉 채우고, 그 위에 **커스텀 시트**(불투명 다크)가 뜬다. 시트는 하단 커스텀 탭바 위에 얹힌다.
- **시스템 `.sheet`는 쓰지 않는다.** 이유:
  - iOS 26 시스템 시트는 작을 때 화면 폭을 꽉 안 채우고 **둥둥 뜬 카드(floating)** 구조로 그려진다. 배경색은 `.presentationBackground`로 불투명화가 가능하지만, 이 floating 구조 자체는 못 바꿔서 "가장자리까지 꽉 찬 클래식 바텀시트" 모양을 못 맞춘다.
  - 지도에 직접 그리는(MKMapView) 무거운 제스처가 있는 탭이라, 시스템 시트와의 궁합을 실기기로 다시 검증해야 하는 리스크.
  - 이미 커스텀 시트(`SheetDetent` collapsed/medium/full·드래그·불투명)가 절반 이상 구현돼 있음.
  - *메모: `presentationBackgroundInteraction(.enabled)`를 켜면 시스템 시트 뒤 지도가 정상 터치되고 뒤 터치로 시트가 안 닫힌다 — 즉 "그리기가 아예 불가"는 아니다. 결정의 실제 무게는 그리기 충돌이 아니라 '보이는 모양(글래스/floating vs 커스텀 제어)'에 있다.*
- **detent를 비율(fraction) 기반으로 개선.** 현재 하드코딩(80/350pt) → 화면 높이 대비 비율로 바꿔 기기별 동일 비율을 얻는다(시스템 시트 없이도 커스텀에서 구현 가능). 이것이 "시스템 시트로 얻고 싶었던 유일한 이점"의 대체.

### 플로팅 버튼 (되돌리기 / 앞으로 / 초기화 / 현위치) — 우상단

- **노출 규칙:**
  - **경로 없음** → 현위치 버튼만 보임(지도 이동용, 그리기 전에도 쓸모 있음). 되돌리기/앞으로/초기화는 숨김.
  - **경로 있음** → 되돌리기/앞으로/초기화가 등장.
- **시트 연동:**
  - 시트가 올라오면 플로팅 버튼도 **같이 위로 이동**.
  - 시트가 커질수록 **점점 페이드아웃**, **풀 시트에서 완전히 사라짐**(시트 콘텐츠에 자리를 내줌 — 다 떠 있으면 오히려 방해).
  - 구현 메모: 커스텀 시트라 시트 높이가 이미 상태값이므로, 참고 영상(TempView)의 `onGeometryChange`/GeometryReader 없이 그 값으로 바로 `offset`·`opacity`를 준다.

## 3. 러닝 탭 — 나이키식 전체화면

- 지도·시트 삼단배치 **없음**.
- **목표 설정(시간/거리) → 시작 → 뛰는 중**: 숫자(시간·거리·페이스) 중심 **전체화면**, 뛰는 중 **지도 없음**.
- **경로 지도는 "끝난 뒤 기록" 화면에만** 나온다.
- 근거: 뛰는 중엔 화면을 잘 안 본다(잠금/백그라운드가 표준). 지도가 사는 곳은 ① 코스 탐색 탭 ② 러닝 기록(끝난 뒤) 두 군데로 정리됨. 차별점(지도)은 코스 탭에서 가장 강하게 살고 기록에서 한 번 더 보인다.

---

## 4. 기록 탭 (나중에 추가)

- 러닝 탭과 마찬가지로 **전체화면 페이지**로 표현(지도+시트 배치 아님). 경로 지도는 상세 안 콘텐츠로.

## 항목 2: 코스 탐색 — 그리기 제스처 개편

### 문제 (실사용에서 확인)

그리기 모드에선 **한 손가락 = 그리기**라, 다음 그릴 곳을 찾으려 지도를 옮기려 할 때 본능적으로 한 손가락 드래그 → 이동이 아니라 **원치 않는 선이 그려짐.** 두 손가락 팬이 "정답"이지만 그리기에 몰입하면 계속 까먹어 실수가 반복됨. 게다가 **탭 모드에선 한 손가락 = 이동**이라 모드에 따라 한 손가락 의미가 뒤집혀 더 헷갈림.

실제 시나리오: 그리기 모드 진입 → 두 손가락으로 줌 인(잘 보려고) → 한 손가락으로 선 그림 → 줌 상태라 다음 그릴 곳 찾으러 지도 이동 필요 → 순간 본능적으로 한 손가락 드래그 → 이동 대신 선이 그려짐.

### 채택 방향 (A): 한 손가락 = 항상 지도 이동, 그리기는 롱프레스-드래그

- 그리기 모드에서도 **한 손가락 드래그 = 지도 이동**(탭 모드와 동일 → 모드별 뒤집힘 제거).
- 선을 그리려면 **꾹 눌렀다가 드래그**해서 원하는 경로를 그림, 손 떼면 스트로크 종료.
- 근거: "한 손가락 = 이동"이라는 손의 본능을 정답으로 만들어 **의도치 않은 선의 뿌리를 제거** + 탭/그리기 일관.
- 트레이드오프: 매 스트로크 "꾹" 시작 마찰(작음, iOS 롱프레스-드래그 관습과 일치). **롱프레스 임계값(얼마나 눌러야 그리기로 볼지)이 관건 — 실기기 튜닝 필요.** 이 방법도 써보고 불편하면 재검토(현재 방법이 이미 불편으로 확인됐으니 대안 시도가 맞다는 판단).

### 미채택

- **B. 이동/그리기 서브 토글** — 옆을 볼 때마다 토글 껐다 켰다, 탭 늘어 흐름 끊김.
- **C. 두 손가락 팬 유지 + 가장자리 자동 스크롤** — 수동 이동 필요는 줄지만, 스트로크 후 "둘러보려" 옮길 때의 본능 실수는 잔존(증상 완화).

### 선택 옵션 (미결, 나중)

- **모드 통합** — A를 적용하면 탭/그리기 둘 다 "한 손가락 = 이동"이 공통이 되므로, 모드 토글을 없애고 "한 손가락=이동 / 탭=점 추가 / 꾹-드래그=선"을 동시 제공할 수도 있음. 단 탭 판정(더블탭 줌·탭 보류 확정 등 기존 이력, MVP10)과 충돌 위험이 있는 큰 변경 → **A가 실기기에서 깔끔히 동작함을 확인한 뒤에만 검토. 지금은 모드 토글 유지.**

### 참고 (기존 제스처 이력)

- 이 앱은 이미 제스처 설계를 여러 번 거쳤다(MVP4 MKMapView 마이그레이션, MVP8 핀치 네이티브 복원, MVP10 두손가락 팬 튜닝·탭 보류 확정). 지금의 "두 손가락 팬" 모델은 그 결과물인데, 실사용에서 안 먹혀 다시 보는 것. 재설계 시 그 이력(`history/mvp10/`, `docs/solutions/`)과의 충돌을 반드시 점검.
