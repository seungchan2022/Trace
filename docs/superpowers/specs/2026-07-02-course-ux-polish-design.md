# 코스 편집 UX 다듬기 — 구간 패널 스크롤 제한 + 핀치 줌 네이티브 복원

날짜: 2026-07-02
상태: 설계 확정 (MVP8 마일스톤 1/2)

## 배경

MVP7 실기기 피드백(구간 패널 무한 확장)과 MVP4 백로그(핀치 줌 UX)를 하나의 마일스톤으로 묶는다.
두 건 모두 `CoursePlannerPage` 안의 소형 UX 개선이고 서로 독립적이라 브랜치 하나로 처리한다.

- **1. 구간 패널 최대 높이 + 스크롤** — 구간이 늘어나도 패널이 화면을 덮지 않게
- **2. 핀치 줌 네이티브 복원** — 그리기 모드에서 수제 핀치 대신 MKMapView 내장 줌 사용

## 1. 구간 패널 최대 높이 + 스크롤

### 현재 동작

`CoursePlannerPage+SegmentPanelComponent.swift`의 `expandedSegmentList`는 `ForEach`로 구간 행을
그대로 쌓기만 한다. 높이 제한이 없어 구간이 늘어날수록 패널이 지도를 계속 덮는다.

### 변경

- 구간 목록(`ForEach` 부분)을 `ScrollView`로 감싸고, 패널 전체 높이에 **화면 높이의 약 40%** 상한을 둔다.
  - 기준을 화면 비율로 두는 이유: 패널의 본질이 "지도를 가리지 않으면서 구간을 보는 것"이므로,
    제한 기준도 지도 노출 비율로 두면 기기 크기(SE~Pro Max)와 무관하게 일관된다.
  - 구간이 적을 때는 지금처럼 내용 높이만큼만 차지한다(상한은 최대치일 뿐, 항상 40%를 차지하지 않는다).
    단, `ScrollView`는 제안받은 높이를 그대로 채우는(greedy) 컨테이너라 상한만 걸면 구간 1개여도
    항상 40%를 차지한다 — 내부 콘텐츠 높이를 `onGeometryChange`로 측정해
    `min(콘텐츠 높이, 화면 높이 × 0.4)`를 ScrollView 높이로 지정하는 방식으로 구현한다.
  - 헤더("구간" 타이틀 + 접기 버튼)는 스크롤 영역 밖에 고정한다.
- **최신 구간 자동 스크롤 (채팅 앱 방식)**: 새 구간이 추가되면 `ScrollViewReader`로
  **가장 최근에 추가된 구간의 행**으로 스크롤한다 — 단, **사용자가 최신 구간 근처를 보고 있을
  때만**. 위로 스크롤해 옛 구간을 보는 중이면 목록을 움직이지 않아 보던 위치를 지킨다
  (2026-07-02 리뷰 이월 질문에서 결정). 판정은 순수 함수로 분리해 유닛 테스트한다.
  패널을 접었다 펼치면 이전에 보던 위치를 복원한다(최신 근처였다면 최신으로).
  - "마지막 행"이 아닌 이유: `CourseEditSession.attach`는 코스 시작점에 붙는 구간을 prepend로
    배열 맨 앞에 넣으므로, 새 구간이 첫 행일 수 있다. 최신 구간은 배열 위치와 무관하게
    attach 생성 순번이 가장 큰 항목이므로 `viewModel.segmentColorKeys`의 최대값 인덱스 행으로
    `scrollTo`한다 — append면 마지막 행, prepend면 첫 행으로 자연히 수렴한다
    (MVP7 colorKey와 같은 생성 순서 기준).
  - undo/clear로 개수가 줄어들 때는 자동 스크롤하지 않는다(사용자가 보던 위치 유지,
    단 스크롤 위치가 범위를 벗어나면 시스템이 자연스럽게 보정).

### 에러 처리

상태 변경이 없는 순수 레이아웃 변경이라 새로운 에러 경로가 없다.

## 2. 핀치 줌 네이티브 복원

### 현재 동작

`MapViewRepresentable.updateUIView`는 그리기 모드 진입 시 `isZoomEnabled = false`로 내장 줌을 끄고,
커스텀 `UIPinchGestureRecognizer`(`handlePinch`)가 손가락 벌림 비율로 span을 계산해 `setRegion`을
호출한다. 이 수제 줌은 **지도 중심 기준으로만** 확대되어, 두 손가락 사이 지점을 앵커로 잡는
내장 줌보다 체감이 나쁘다.

### 변경

- 그리기 모드에서도 `isZoomEnabled = true`를 유지한다 (`updateUIView`의 모드 전환 블록에서
  `isZoomEnabled` 토글 제거).
- 커스텀 핀치 관련 코드를 전부 삭제한다: `pinchGR` 생성/등록, `pinchGestureRecognizer` 프로퍼티,
  `handlePinch`, `pinchStartSpan`/`pinchStartScale` 상태.
- 유지하는 것: 그리기 중 `isScrollEnabled = false` + 커스텀 2손가락 pan(내장 이동 제스처는
  1손가락에도 반응해 그리기와 충돌하고, 내부 제스처 인식기에 손가락 수를 제한할 공개 API가 없음),
  `isPitchEnabled`/`isRotateEnabled` 비활성(그리기 오염 방지).

### 근거와 리스크

- 그리기 제스처는 `drawGR.maximumNumberOfTouches = 1`이라 두 손가락 핀치와 터치 경쟁이 없다
  (MVP4 백로그에서 분석 완료).
- `isZoomEnabled = true`는 더블탭 줌·더블탭-드래그 줌(1손가락)도 함께 살린다. 그리기 스트로크는
  이동을 동반하고 더블탭은 이동이 없어 원리상 충돌하지 않지만, **실기기 QA에서 더블탭 줌과
  그리기 시작이 간섭하는지 확인**한다. 간섭이 확인되면 그 시점에 대응을 설계한다(추측 선반영 금지).

## 데이터/상태 흐름

- 도메인/Application/ViewModel 변경 없음. View 레이어(`CoursePlannerPage+SegmentPanelComponent`,
  `MapViewRepresentable`)만 수정한다.

## 테스트

- 패널: 뷰 레이아웃 변경이라 유닛 테스트 대상이 아님. 기존 ViewModel 테스트가 회귀 가드.
- 핀치: 제스처 삭제라 유닛 테스트 불가. 빌드 + 기존 테스트 통과로 회귀 확인.
- 실기기 QA 항목:
  - 구간 10개 이상에서 패널이 화면 ~40%에서 멈추고 내부 스크롤되는지
  - 새 구간 추가 시 목록이 최신 구간으로 따라가는지
  - 그리기 모드에서 두 손가락 핀치가 손가락 사이 지점 기준으로 자연스럽게 줌되는지
  - 더블탭 줌이 그리기와 간섭하지 않는지
