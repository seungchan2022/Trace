# MVP10 — attach-nearest-fallback: 원거리 스트로크 최근접 끝점 비교 설계

2026-07-05. MVP10 실기기 QA에서 발견된 그리기 모드 방향 판정 UX 갭
(`docs/backlog.md` MVP10 QA 섹션)을 해결한다: `CourseEditSession.attach` 규칙 4가
상대적 근접성을 무시하고 무조건 도착점 기준으로 붙이는 문제.

## 배경 — 왜 지금 규칙이 이렇게 생겼고, 왜 바꾸는가

- **QA에서 확인된 갭**: 그리기 시작점이 핀 히트(화면 24pt)도 실거리 임계값(20m)도 안 걸리는
  "중간지대"에서는, 출발점에 명백히 더 가까워도(실기기 로그: distToStart=324m,
  distToEnd=1241m) 규칙 4가 무조건 도착점에서 gap 라우팅 append한다. 사용자는 출발점 쪽을
  그었는데 도착점이 1.2km 늘어나는 것처럼 보인다.
- **MVP9가 거리 비교를 버린 진짜 이유**: 구 `resolveOrientation`은 **스트로크 양끝 4쌍
  비교**(스트로크 양끝 × 코스 양끝)로 붙일 위치와 반전 여부를 동시에 추측했다. 왕복
  스트로크는 양끝이 코스 양끝 모두에 가까워 미세한 좌표 차이로 판정이 뒤집혔다
  (`history/mvp9/2026-07-03-edit-consistency-design.md`). 흔들림의 진원지는 "양끝 비교 +
  반전 추측"이지, 거리 비교 그 자체가 아니다.
- **탭 모드는 이미 안전한 비교를 하고 있다**: `CoursePlannerPageViewModel.nearestEndpoint`는
  탭 지점(단일 점)을 두 끝점과 마진 없이 `<=` 비교해 가까운 쪽에서 라우팅하고, 출발점 쪽이면
  attach 규칙 3을 타서 반전 prepend까지 거리 무관하게 동작한다. MVP9의 왕복 흔들림(스트로크
  **양끝**이 코스 양끝에 동시 근접)은 단일 점 비교에서 구조적으로 재발하지 않고, 왕복
  스트로크는 규칙 1·2가 선점한다.
- **수용하는 경계 불연속**: 열린 코스의 두 끝점에서 거의 등거리인 "이등분선 근처"에서 시작한
  스트로크는 몇 m의 터치 차이로 판정(출발점 쪽 ↔ 도착점 쪽)이 갈릴 수 있다. 이것은 규칙 1이
  잡아주는 케이스가 **아니며**(닫힌 코스 판정은 코스 양끝 간 20m 기준일 뿐), 탭 모드
  `nearestEndpoint`의 마진 없는 `<=`가 이미 갖고 있는 동일한 트레이드오프다. 모드 간 판정
  통일을 위해 마진 없이 수용하기로 결정(2026-07-05, Q2=A). 오판은 undo 한 번으로 복구된다.
- **결론**: MVP9는 흔들림의 진원지를 제거하면서 안전한 부분(시작점 단일 비교)까지 과잉
  제거했고, 그 과보정이 탭↔그리기 비대칭으로 드러났다. 규칙 4를 탭과 동일한 최근접 비교로
  교체해 두 모드의 판정 원리를 통일한다.

## 범위

- 변경: `CourseEditSession.attach` 규칙 4 분기 하나.
- 불변: 규칙 1(닫힌 코스)·2(도착점 근접)·3(출발점 근접 반전 prepend) — 왕복 스트로크는
  지금처럼 규칙 1·2가 먼저 잡으므로 MVP9의 안정성 보장이 문자 그대로 보존된다.
  ViewModel(`snappedStrokeStart`, `handleMapTap`, `nearestEndpoint`)과 View도 변경 없음.

## 설계

### 새 규칙 4 — 최근접 끝점 비교

**진입 조건**: `!isClosedCourse && !startsNearEnd && !startsNearStart` — 규칙 1~3이 모두
안 걸린 경우에만 비교가 실행된다. 현재 구현은 규칙 1·2·4가 하나의 append 블록에 융합되어
있으므로(`CourseEditSession.swift`의 "규칙 1·2·4" 주석 블록), 이 블록을 분리해 비교를
게이트해야 한다 — 게이트 없이 비교를 넣으면 닫힌 코스(양 끝점이 근접해 distToStart ≈
distToEnd)가 출발점 쪽으로 새어 반전 prepend되어 규칙 1의 "닫힌 코스는 무조건 append"
보장이 깨진다.

진입 조건을 통과한 원거리 스트로크는, 탭의 `nearestEndpoint`와 동일한 `<=` 비교로
붙일 쪽을 정한다:

| 판정 | 동작 |
|---|---|
| `distToEnd < distToStart` (도착점 쪽) | 현행 그대로 — 도착점→시작점 gap 라우팅 후 append |
| `distToStart <= distToEnd` (출발점 쪽, **등거리 포함** — 탭과 동일한 tie 규칙) | **반전 prepend + gap 라우팅** — 스트로크를 반전하고(달리는 순서: 스트로크 끝 → 스트로크 시작), 스트로크 시작점 → 기존 출발점 gap 라우팅을 이어붙여 하나의 세그먼트로 prepend |

- 규칙 3은 이 케이스의 "gap이 필요 없는 근거리 특수형"이 된다 (반전 후 끝 좌표 ≈ 기존
  출발점이라 gap 생략).
- QA 케이스(324m vs 1241m)는 출발점 쪽 반전 prepend로 붙는다.
- 반전 예외는 여전히 "출발점 쪽 시작 = 출발 방향 연장" 하나다. MVP9 스펙의 "원거리에서
  출발점을 **향해** 긋는 스트로크는 그린 그대로 append" 의도는 이번 변경으로 유일하게
  동작이 바뀌는 지점 — 규칙 3의 의미를 원거리로 일관 확장한 수용된 트레이드오프이며,
  실기기 QA로 체감을 확인한다.
- **기각한 대안 — 스트로크 끝점 기준 비반전 prepend**: "끝점이 출발점 근처면 그린 그대로
  앞에 붙이기"는 "그린 방향 = 달리는 방향"을 지키지만, 판정 입력에 스트로크 **끝점**을
  쓰는 순간 왕복 스트로크(시작점≈코스 도착점 **그리고** 끝점≈코스 출발점이 동시 성립)의
  모호성이 재발한다 — MVP9가 제거한 바로 그 구조. 판정 입력은 스트로크 시작점 하나로
  제한한다.

### 병합 세그먼트 구성 (출발점 쪽)

- 좌표: `reversed(stroke)` + `gapRoute(strokeStart → existingStart).dropFirst()`
- 거리: 스트로크 거리 + gap 거리
- `Entry.placedAtFront = true`로 prepend — undo/redo/색상 키(`order`)는 기존 메커니즘
  그대로 동작한다.

### 에러 처리

출발점 쪽 gap 라우팅 실패(`routeNotFound`/스로틀)는 기존 도착점 쪽과 동일한 경로를 탄다 —
entry 추가 전에 throw되므로 세션 상태 불변, redo 스택 보존(MVP9 규칙), ViewModel의 기존
에러 메시지 그대로.

## 테스트

단위 테스트 (`CourseEditSessionTests`, 기존 규칙 1~4 테스트 패턴 유지):

- 원거리 시작점이 출발점에 더 가까움 → 반전 prepend + gap 병합 (좌표 순서·`placedAtFront`·
  거리 합산 검증)
- near-tie 경계 검증: `distToStart`가 `distToEnd`보다 근소하게 작음(예: −1m) → 출발점 쪽,
  근소하게 큼(예: +1m) → 도착점 쪽 — 경계 양쪽이 각자 문서화된 결과를 결정론적으로 내는지
  확인 (Haversine 실거리에서 부동소수점 완전 등거리는 실사용 도달 불가라 완전 일치 케이스는
  검사하지 않음; tie 시 출발점 쪽이라는 `<=` 의미는 규칙 표에 명문화로 충분)
- 원거리 시작점이 도착점에 더 가까움 → 현행 gap append 유지 (회귀)
- 닫힌 코스 + 원거리 스트로크 → 여전히 gap append (규칙 1 선점 = 진입 조건 게이트 검증)
- 출발점 쪽 gap 라우팅 실패 → 세션 상태 불변 + redo 스택 보존
- 기존 규칙 1~3 테스트 전부 그린 유지

실기기 QA 시나리오 (체크리스트에 추가):

- 324m 케이스 재현 — 출발점 쪽에 붙는지
- 원거리에서 출발점을 **향해** 긋기 — 반전이 체감상 자연스러운지 (유일한 동작 변경점)
- 왕복 그리기 회귀 — 규칙 1·2 선점이 유지되는지

QA 실패 기준(사전 결정): "출발점을 향해 긋기" 시나리오에서 반전된 달리는 순서가 의도와
다르게 느껴지는 사례가 재현되면, 부분 마진 같은 절충을 새로 설계하지 않고(Q2=A 결정 유지)
규칙 4를 무조건 append로 되돌린 뒤 backlog 항목을 reopen한다 — QA가 확인 절차가 아니라
실제 판정이 되도록.
