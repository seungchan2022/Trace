# 러닝 코스 계획 MVP 설계

## 목표

Trace의 첫 화면은 달리기 전에 러닝 코스를 계획하는 화면이다.

이번 MVP는 "러닝을 기록하는 것"이 아니라 "오늘 어디를 뛸지 정하는 것"에 집중한다.
첫 사용 흐름은 다음과 같다.

1. 지도를 연다.
2. 출발 지점을 탭한다.
3. 도착 지점을 탭한다.
4. 두 지점 사이의 실제 도보 경로를 표시한다.
5. 전체 경로 거리를 표시한다.

검색, 코스 저장, 러닝 기록, 계정, 동기화, 커뮤니티 기능은 이번 MVP 범위에서 제외한다.

## 제품 방향

Trace는 개인용 러닝 코스 계획 앱이다. 사용자가 달리기 전에 지도에서 코스를 만들고, 실제 경로와 거리를 확인한 뒤, 나중에 마음에 드는 코스를 저장할 수 있게 한다.

첫 버전은 핵심 가치를 검증해야 한다. 사용자가 두 지점을 선택했을 때 앱은 그 사이를 단순 직선으로 잇지 않고, 실제 이동 가능한 길을 따라가는 경로를 보여줘야 한다.

## 핵심 요구사항

경로선은 A와 B 사이를 단순히 직선으로 연결하면 안 된다.

앱은 실제 도보 경로를 요청하고, 그 결과로 받은 경로 형상을 지도에 그려야 한다. 사용자가 바다, 건물, 막힌 도로, 강 같은 장애물이 있는 두 지점을 선택했을 때, 경로선이 그 사이를 뚫고 지나가면 안 된다. 지도/라우팅 제공자가 알고 있는 실제 이동 가능한 길을 따라가야 한다.

초기 구현은 Apple `MapKit` 라우팅을 사용한다.

- `MKDirections`로 경로를 요청한다.
- 이동 방식은 `.walking`을 사용한다.
- 지도에는 `MKRoute.polyline`을 그린다.
- 총 거리는 `MKRoute.distance`를 사용한다.
- 도보 경로를 찾지 못하면 직선으로 대체하지 않고 에러 상태를 보여준다.

`MapKit` 경로 품질은 Apple 지도 데이터에 의존한다. MVP에서는 서버나 API 키 없이 빠르게 만들 수 있다는 장점이 있으므로 이 선택을 한다. 대신 나중에 한국에 더 적합한 지도/라우팅 제공자로 교체할 수 있도록 구조를 분리한다.

## 사용자 흐름

첫 화면은 코스 계획 화면이다.

- 초기 상태: 지도만 보이고 선택된 경로는 없다.
- 첫 번째 지도 탭: 출발지를 설정한다.
- 두 번째 지도 탭: 도착지를 설정하고 도보 경로 계산을 요청한다.
- 경로 계산 중: 출발지와 도착지 핀은 유지하고, 경로 계산 중임을 표시한다.
- 경로 계산 성공: 출발지 핀, 도착지 핀, 실제 경로선, 총 거리를 보여준다. 경로선이 표시된 뒤에도 두 지점의 마커는 경로선 위 레이어에 보이도록 유지한다.
- 경로 계산 실패: 핀은 유지하고, 도보 경로를 찾을 수 없다는 상태를 보여준다.
- 세 번째 지도 탭: 기존 경로를 지우고, 탭한 지점을 새 출발지로 삼아 다시 선택을 시작한다.

## 구조

MVVM 구조를 사용하고, 경로 계산은 포트/어댑터 경계로 분리한다.

상태 관리는 iOS 17+ 기준의 Observation API를 사용한다. ViewModel은 `ObservableObject`/`@Published`가 아니라 `@Observable` 매크로 기반으로 작성하고, UI에서 소유하는 ViewModel은 `@State`로 보관한다.

Swift 동시성은 Swift 6 이후의 기준에 맞춘다. 비동기 경로 계산은 `async`/`await`로 표현하고, UI 상태를 바꾸는 ViewModel은 `@MainActor`로 격리한다. 경계를 넘는 서비스 프로토콜과 값 타입은 가능한 한 `Sendable`을 만족하도록 설계한다.

### View

`CoursePlannerPage`

- SwiftUI 지도 화면을 담당한다.
- 지도 탭 좌표를 ViewModel에 전달한다.
- ViewModel 상태를 바탕으로 핀, 경로선, 거리, 로딩, 에러 상태를 표시한다.
- `MapKit` 경로 계산 API를 직접 호출하지 않는다.

### ViewModel

`CoursePlannerPageViewModel`

- 선택된 출발지/도착지 좌표를 관리한다.
- 두 지점이 모두 선택되면 경로 계산 서비스를 호출한다.
- 경로 형상, 총 거리, 로딩 상태, 에러 상태를 View에 제공한다.
- 사용자가 두 번째 지점 이후 새 경로를 시작하면 기존 상태를 초기화한다.

### UseCase

첫 MVP에서는 별도 UseCase 타입을 만들지 않는다. 경로 생성 규칙이 아직 "두 좌표로 도보 경로를 요청한다" 수준이라 ViewModel이 `CoursePlanningServiceProtocol` 포트를 직접 호출해도 충분하다.

나중에 코스 저장, 선호 거리, 회피 구간, 순환 코스 추천, 한국 지도 제공자 선택 같은 앱 정책이 생기면 `PlanCourseUseCase`를 추가한다. UseCase는 앱의 의사결정 규칙을 담고, 구체적인 지도 SDK나 네트워크 구현에는 의존하지 않는다.

### 도메인 모델

Routing 도메인 타입은 특정 화면에 묶지 않고 `Trace/Domain/CoursePlanning/Entity`에 둔다. 저장 코스, 코스 상세, 검색, 경로 재계산에서도 같은 좌표/경로 타입을 재사용할 가능성이 높기 때문이다.

`CourseCoordinate`

- 앱 내부에서 사용하는 경로 좌표 타입이다.
- `latitude`, `longitude`만 가지며 `MapKit` 타입에 직접 의존하지 않는다.
- `MapKit`, 한국 지도 SDK, 서버 라우팅 API 같은 외부 구현체는 경계에서 이 타입으로 변환한다.

`PlannedCourse`

- View가 그릴 수 있는 경로선 또는 좌표 형상을 포함한다.
- 총 거리를 미터 단위로 포함한다.
- 나중에 예상 시간, 경로 이름, 저장 코스 메타데이터를 추가할 수 있다.

### 포트

`CoursePlanningServiceProtocol`

- 앱이 사용하는 경로 계산 인터페이스를 정의한다.
- 출발지/도착지 좌표를 받아 `PlannedCourse`를 반환한다.
- 경로를 찾지 못하면 도메인 수준 에러를 던진다.
- `Trace/Domain/CoursePlanning/Protocol`에 둔다. 특정 화면이 아니라 앱 전체의 routing 포트이기 때문이다.

ViewModel은 `MKDirections`가 아니라 이 프로토콜에 의존한다.

### 어댑터

`MapKitCoursePlanningService`

- `CoursePlanningServiceProtocol`의 첫 구현체다.
- 좌표를 `MKMapItem`으로 변환한다.
- `.walking` 설정의 `MKDirections.Request`를 사용한다.
- 선택된 `MKRoute`를 앱의 `PlannedCourse`로 변환한다.
- 실패 시 직선 대체 경로를 반환하지 않는다.

`Infrastructure/CoursePlanning/MapKit`에는 이런 어댑터 구현체를 둔다. 즉 이 영역은 유스케이스가 아니라 `MapKit`, 네트워크, 저장소, 위치 권한, 외부 SDK처럼 앱 바깥 세계와 맞닿는 구현 세부사항을 담는다.

나중에 한국에 더 적합한 제공자가 필요해지면 `KoreaCoursePlanningService` 같은 새 어댑터를 추가한다. ViewModel과 View는 같은 `CoursePlanningServiceProtocol` 인터페이스를 계속 사용하므로 변경 범위를 줄일 수 있다.

## 에러 처리

다음 상태를 명시적으로 처리한다.

- 경로 없음: 도보 경로를 찾을 수 없다는 사용자 메시지를 보여준다.
- 경로 요청 실패: 재시도 가능한 실패 상태를 보여준다.
- 경로 계산 중 사용자가 선택을 바꿈: 이전 요청을 취소하거나 오래된 결과를 무시해서 최신 선택만 반영한다.

실패 시 직선을 그리지 않는다. 직선 대체는 Trace의 핵심 약속을 깨뜨린다.

## 제외 범위

- 장소 검색
- 현재 위치 권한
- 코스 저장
- 코스 목록
- 러닝 활동 기록
- GPX 가져오기/내보내기
- 계정, 동기화, 백엔드, 분석, 수익화
- 커뮤니티 또는 공유 코스
- 한국 전용 라우팅 제공자 연동

## 성공 기준

- 첫 화면에 지도가 보인다.
- 한 번 탭하면 출발지 마커가 생긴다.
- 두 번 탭하면 도착지 마커가 생긴다.
- 두 지점 사이의 도보 경로를 요청한다.
- 표시되는 선은 직선이 아니라 반환된 실제 경로 형상을 따른다.
- 경로선이 표시된 상태에서도 출발지/도착지 마커가 경로선 위에 명확히 보인다.
- 전체 거리가 표시된다.
- 경로를 찾지 못하면 에러를 보여주고 직선을 그리지 않는다.
- 세 번째 탭은 새 경로 선택을 시작한다.
- 경로 계산은 `CoursePlanningServiceProtocol`를 통해 접근해서 `MapKit` 구현체를 나중에 교체할 수 있다.

## 검증 계획

자동화는 두 층으로 나눈다. 핵심 화면 동작은 안정적인 가짜 경로 서비스로 UI 테스트를 자동화하고, 실제 `MapKit` 라우팅은 외부 지도 데이터와 네트워크 상태에 영향을 받으므로 MVP에서는 수동 스모크 검증으로 둔다.

- `CoursePlannerPageViewModel`의 선택 상태 전이를 단위 테스트한다.
- 경로 계산 성공 시 거리와 경로 형상이 노출되는지 단위 테스트한다.
- 경로 계산 실패 시 에러가 노출되고 경로 형상이 없는지 단위 테스트한다.
- ViewModel 테스트에서는 가짜 `CoursePlanningServiceProtocol`를 사용한다.
- UI 테스트에서는 앱을 테스트 모드로 실행하고, 가짜 `CoursePlanningServiceProtocol`가 정해진 경로와 거리를 반환하게 한다.
- UI 테스트는 실제 사용 흐름처럼 지도 영역을 두 번 탭하고, 출발지/도착지 마커, 경로 표시 상태, 거리 표시가 나타나는지 확인한다.
- UI 테스트는 실패 모드도 실행해서 경로를 찾지 못했을 때 에러가 보이고 직선 대체 경로가 표시되지 않는지 확인한다.
- 실제 `MapKit`을 사용하는 E2E 테스트는 자동화 대상에서 바로 제외하지는 않되, MVP 첫 구현에서는 안정적인 필수 게이트로 두지 않는다. 이후 지도 제공자와 테스트 좌표가 안정되면 별도 스모크/E2E 시나리오로 추가한다.
- Xcode로 앱을 빌드한다.
- 테스트 타깃을 실행한다.
- Simulator에서 실제 `MapKit` 모드로 두 지점을 탭했을 때 직선이 아닌 실제 경로선과 거리가 표시되는지 수동 확인한다.
