## Keep (잘 된 것)

**subagent-driven-development 7태스크 + 전체 브랜치 리뷰(opus)가 전부 clean 승인**
attach 규칙 교체, redo 스택, 핀 히트 분기, 화면 24pt 히트 판정, 병합 핀, 경유점 마커까지 태스크별로
implementer→reviewer 루프를 거쳤고, 마지막 전체 브랜치 리뷰에서도 Critical/Important 코드 결함이
0건이었다. attach 규칙 교체(Task 1)의 반전 우선순위처럼 교차 태스크 통합 지점도 opus가 직접
`CourseEditSession.attach`/`handleMapTap` 코드를 손으로 추적해 검증했다.

**임시 print 디버그 로그로 "버그처럼 보이지만 실은 정상 동작"을 확정**
그리기 모드 반전이 실기기에서 안 된다는 보고를 받고, MVP8에서 확립한 패턴(임시 print + 사용자 재현 +
콘솔 로그)을 그대로 다시 썼다. 로그 한 번으로 `distToStart=51m > threshold=20m`을 확인해, 코드는
설계대로 정확히 동작 중이고 문제는 "육안 근접"과 "실거리 임계값"의 정밀도 불일치였다는 걸 대화 여러
턴 없이 바로 확정했다.

**사용자가 빠른 패치를 거부하고 근본 재설계를 요구한 게 결과적으로 더 나은 결정이었다**
더블탭 문제를 "모든 탭에 0.3초 지연"으로 완전히 없앨 수 있다고 제안했을 때, 사용자는 다른 지도 앱이
지연 없이 이걸 해내는 걸 직접 확인하고 트레이드오프를 거부했다. 그리기 모드 임계값도 "단순히 숫자를
올리면 경계 문제가 다른 곳으로 옮겨갈 뿐"이라는 정확한 지적으로 기각됐다. 둘 다 다음 MVP로 미뤘지만,
근본 원인은 이번 세션에서 이미 다 밝혀놔서 재조사 없이 바로 설계에 들어갈 수 있다.

## Problem (아쉬운 것)

**검증 스탬프 신선도 문제가 같은 세션 안에서 두 번 재발**
Task 1에서 오래된 스탬프로 커밋이 통과된 걸 발견해 문서화(`docs/solutions/workflow-issues/stale-verify-stamp-false-pass.md`)까지
했는데, Task 6에서 똑같은 패턴이 다시 나왔다. 서브에이전트에게 "확인하라"는 지시만으로는 재발을
막지 못했고, "`rm -f`로 먼저 지우고 시작하라"는 더 구체적인 지시로 바꾼 뒤에야(Task 7부터) 안정화됐다.
지시의 구체성이 재현성보다 훨씬 중요하다는 걸 두 번째 재발 이후에야 체감했다.

**서브에이전트의 "더 견고한" 수정이 실제로는 회귀를 냈다**
더블탭 충돌을 고치려고 서브에이전트가 만든 자체 센티널 `UITapGestureRecognizer` + `require(toFail:)`
방식이, 로컬 전체 테스트 실행에서 서로 다른 두 지점을 탭하는 기존 UI 테스트 2건을 깨뜨렸다. 커밋
전에 로컬 검증으로 잡았기 때문에 실사용에는 영향 없었지만, "이론적으로 더 견고해 보이는" UIKit
표준 기법이 실제로는 위치를 구분 못 해 서로 무관한 정상 플로우까지 삼켜버린 사례였다.

**경유점 z-order를 세 번 시도해서야 진짜로 고쳤다**
zPosition → 추가 순서 재정렬 → 오버레이 전환, 이 순서로 세 번 손댔다. 앞의 두 시도는 "애노테이션끼리의
z-order를 조정하는" 접근이었는데 둘 다 실기기에서 재발했다. 세 번째(오버레이로 완전히 옮기기)만
MapKit의 구조적 보장(오버레이는 항상 애노테이션 아래)에 기대서 확실했다. 처음부터 "애노테이션끼리는
z-order가 근본적으로 불안정할 수 있다"를 의심했으면 두 번 덜 시도했을 것이다.

## Surprise (예상 밖)

**유닛 테스트로 완벽했던 로직이 실기기 네트워크 서비스 앞에서 무너졌다**
`CourseEditSession.attach`의 20m 임계값 규칙은 스텁 서비스로는 완벽했지만, 실제 `MKDirections` 라우팅이
좌표를 도로망에 스냅하면서 생기는 드리프트 때문에 그리기 모드에서 거의 항상 실패했다. "정확한 스텁으로
통과하는 테스트"와 "실제 네트워크 서비스의 암묵적 오차"가 다른 종류의 신뢰라는 걸 다시 확인했다.

**경유점을 재확인하다가 완전히 다른 버그(거리 라벨 위치)를 우연히 발견**
경유점-라벨 겹침을 재테스트하던 사용자가 "겹치는 부분은 위로 오는데, 라벨 자체가 찍히는 위치가
이상하다"고 지적했다. 원인은 `coords[coords.count/2]`(배열 인덱스 중간)이 라우팅 결과의 불균일한 좌표
밀도 때문에 실제 경로 거리 중간과 무관했던 것 — 계획했던 QA 항목이 아니라 재확인 과정에서 우연히
튀어나온 버그였다.

**"버그처럼 보이는 것"과 "설계 정밀도 문제"를 구분하는 데 로그가 결정적이었다**
그리기 모드 반전 실패는 코드 리뷰만으로는 "버그다/아니다"를 확정할 수 없었다. 디버그 로그 한 번으로
"코드는 맞고, 사용자의 육안 판단과 20m 임계값이 안 맞았다"는 걸 명확히 갈랐다 — 이게 명확해지자
"임계값을 조정할까 재설계할까"라는 훨씬 생산적인 논의로 바로 넘어갈 수 있었다.

## 마일스톤별 핵심 의사결정

| 항목 | 결정 | 이유 |
|---|---|---|
| attach 규칙 | 4쌍 거리 비교(자동 방향 감지) 폐기, 순서 규칙(반전은 출발점 연장 단일 예외)으로 교체 | 왕복 스트로크가 반전되어 핀 라벨이 뒤집히던 버그의 근본 원인 |
| 그리기 모드 라우팅 | 라우팅 전에 원본 좌표를 기존 끝점으로 스냅(탭 모드와 동일 원칙) | 실기기에서 도로망 스냅 오차가 20m 임계값을 넘어 반전 판정이 거의 항상 실패하던 문제 수정 |
| 핀 히트 판정 | 지도거리(줌 종속)가 아니라 화면 24pt 반경 | 줌 레벨과 무관하게 일관된 히트 판정 |
| 더블탭 vs 싱글탭 | `require(toFail:)`(위치 무관) 대신 자체 시간(0.35s)+거리(40pt) 디바운스 | require(toFail:)가 서로 다른 두 지점 탭까지 삼켜버리는 회귀를 실제로 확인 |
| 경유점 렌더링 | 애노테이션이 아니라 오버레이로 그림 | zPosition·추가 순서 둘 다 애노테이션 간 z-order로는 실기기에서 불안정, 오버레이는 항상 애노테이션 아래라는 구조적 보장 이용 |
| 거리 라벨 위치 | 배열 인덱스 절반 대신 누적 실거리 절반 지점(보간) | 라우팅 좌표 밀도가 불균일해 인덱스 기준으로는 라벨이 랜덤한 위치에 찍힘 |
| 그리기 모드 근접 임계값(20m) | 지금은 그대로 두고 다음 MVP로 이연 | 단순 상향은 경계 문제를 옮길 뿐 — 탭 모드처럼 화면 픽셀 기준 재설계가 근본 해결 |
| 더블탭 완전 차단 | 지금은 "1개까지만 찍힘"으로 개선, 완전 차단은 다음 MVP로 이연 | 지연 없이 해결 가능해 보이나(타 지도 앱 확인) 방법 미확정, 트레이드오프 없이 제대로 조사 필요 |

## 남은 기술부채 (`docs/backlog.md` 연결)

- **그리기 모드 "출발점 근처" 판정을 화면 픽셀 기준으로 재설계** — 탭 모드의 24pt 히트 판정과 동등한
  개념을 그리기 시작점에도 적용. 그리기 제스처 시작점을 화면 좌표로 View→ViewModel까지 전달하는 배관
  작업 필요, 규모 있음.
- **탭모드 더블탭 완전 차단(지연 없이)** — 네이티브 더블탭 인식기의 실제 인식 시점을 gesture delegate로
  관찰하는 방법 등 조사 필요.
- **라우팅 스냅 오차로 병합 핀이 안 뜰 수 있는 케이스** — 코드리뷰 Important 발견, 아직 실기기 재현 안
  됨. 지속 관찰만 하고 급하지 않음.
