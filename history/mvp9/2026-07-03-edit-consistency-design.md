# MVP9 · edit-consistency — 편집 정합성 (왕복·핀·redo) 설계

- 날짜: 2026-07-03
- MVP/마일스톤: MVP9 `edit-consistency` (`docs/roadmap.md`)
- 출처 백로그: 탭 모드 왕복 불가, 왕복 시 출발/도착 핀 라벨 혼동 (`docs/backlog.md` MVP8 QA 피드백)
- 상태: 사용자와 브레인스토밍으로 확정 후 `ce-doc-review`(coherence·feasibility·design-lens·adversarial) 1라운드 반영

## 배경과 문제

MVP8 실기기 QA에서 두 가지 편집 정합성 문제가 확인됐다.

1. **탭 모드로 왕복 코스를 만들 수 없다.** 탭 한 번의 의미가 "가장 가까운 끝점 → 탭한 지점"이라,
   원래 출발점을 다시 탭하면 가장 가까운 끝점이 출발점 자신이 되어 0m 경로로 귀결된다.
2. **왕복을 그리면 출발/도착 핀이 뒤집힐 수 있다.** 현재 `CourseEditSession.resolveOrientation`은
   새 구간의 시작점·끝점과 기존 코스의 출발점·도착점 사이 **4쌍 거리**를 비교해 가장 가까운 쌍으로
   붙일 위치(앞/뒤)와 방향(뒤집기 여부)을 정한다. 왕복 스트로크는 양 끝이 기존 코스의 양 끝과
   모두 가까워서 몇 미터 차이로 prepend가 선택될 수 있고, 그러면 배열 맨 앞이 물리적 도착 지점이
   되어 "출발" 핀이 거기 붙는다.

추가로, 되돌리기(undo)를 실수로 눌렀을 때 복구 수단(redo)이 없어 구간을 처음부터 다시 만들어야 한다.

## 목표

- 이어붙이기 동작을 사용자 멘탈 모델("그린 방향 = 달리는 방향")과 일치시켜 왕복 모호성을 원천 제거한다.
- 탭 모드에서도 왕복 코스를 만들 수 있게 한다.
- 왕복/구간 구조가 지도에서 읽히도록 핀 표현을 보강한다.
- undo 실수를 redo로 복구할 수 있게 한다.

## 범위 밖 (명시적 제외)

- **그리기 모드 제스처 경쟁** (두 손가락 탭 줌아웃 ↔ 커스텀 pan) — 실기기 튜닝 성격이라 별도 기회에. 백로그 유지.
- **구간 패널 prepend 스크롤 밀림** — 위쪽 추가 시 밀림은 리스트의 자연스러운 동작으로 판단, 보류.
- **코스 저장** — 다음 MVP 후보로 유지. 이번 MVP가 "출발점" 의미를 안정시켜 저장 설계의 선행 리스크를 줄인다.
- **전용 "왕복" 버튼** — 출발핀 탭(아래)이 같은 동작을 커버. 실사용에서 불편하면 그때 버튼만 추가(내부 로직 재사용).
- **경유점 번호 표시** — 무번호 점으로 시작, 순서 니즈가 확인되면 확장.
- **clear의 redo 복구** — clear는 redo 대상이 아니다(의도된 비대칭). 실수 clear 복구 니즈가 확인되면 백로그로.

## 설계

### 1. 이어붙이기 규칙 교체 — `CourseEditSession`

`resolveOrientation`의 4쌍 거리 비교(자동 방향 감지)를 버리고, **임계값 기반 순서 규칙**으로
교체한다. 아래 if-else를 위에서부터 평가한다 (연결 임계값 = 기존 gap 판정과 동일한 20m):

1. **닫힌 코스면 무조건 append** — 기존 코스의 첫·끝 좌표가 20m 이내(닫힌 코스 판정과 동일
   기준)면 비교를 생략하고 그린 그대로 뒤에 append한다. 닫힌 코스에서는 새 구간 시작점이
   양 끝점과 거의 등거리가 되어 어떤 거리 비교도 ε 차이 동전던지기가 되기 때문이다(원버그의
   재발 경로를 차단하는 퇴화 규칙).
2. **새 구간 시작점이 기존 도착점 반경 20m 이내면 append** — 그린 그대로 뒤에 붙인다(뒤집기 없음).
3. **새 구간 시작점이 기존 출발점 반경 20m 이내면 반전 prepend** — "출발 방향 연장"으로 해석해
   구간을 반전해 앞에 붙인다. **이것이 유일한 반전이며, "그린 방향 = 달리는 방향" 원칙의
   명시적 예외다** (출발점에서 바깥으로 그린 스트로크를 달리는 순서에 맞추기 위한 반전).
4. **그 외(양 끝점 모두에서 먼 스트로크)는 항상 append** — 도착점에서 새 구간 시작점까지
   gap 라우팅으로 이어붙이고, 그린 그대로 뒤에 붙인다. 거리 비교로 앞/뒤를 추측하지 않는다 —
   "새 경로는 항상 도착점에서 이어진다"가 기본값이고, 출발 방향 연장은 3번(출발점에서 그리기
   시작)으로만 트리거된다.

효과:

- **자동 방향 감지(거리 비교 기반 스트로크 뒤집기) 제거** — 반전은 3번 규칙 하나로 좁혀지고,
  그 외 모든 스트로크는 그린 그대로 붙는다. 중간까지 갔다 돌아와 다른 길로 새는 등 어떤 모양도 유지된다.
- 왕복 스트로크(도착점 근처에서 시작해 출발점까지 되짚기)는 2번 규칙으로 항상 append —
  배열 순서 = 달리는 순서가 유지되고, 핀 라벨 뒤집힘이 사라진다.
- 코스 밖 원거리에서 출발점을 **향해** 그리는 스트로크도 4번 규칙으로 그린 그대로 append —
  구규칙처럼 반전되어 역주행 루프가 생기지 않는다.
- 탭 구간도 같은 `attach`를 타므로 규칙이 자동으로 통일된다. (탭으로 출발점 방향 연장:
  `handleMapTap`이 출발점에서 새 지점으로 라우팅 → 시작점이 출발점 → 3번 규칙 → 반전 prepend.)
- undo/색상의 시간순(`order`) 체계는 그대로 유지한다.

동작 변경 유의: 기존의 "어느 방향으로 그리든 뒤집어 이어주는" 자동 감지가 사라지고, 방향은
그린 대로(예외: 출발점 연장) 존중된다. 예측 가능성을 위한 의도된 트레이드오프이며
`project-decisions.md`에 결정으로 기록한다.

엣지 노트: 전체 길이가 20m 미만인 초단거리 코스는 닫힌 코스로 오판정될 수 있으나(1번 규칙 +
병합 핀), 그 규모의 코스는 실사용 의미가 없어 수용한다.

### 2. 탭 모드 왕복 — 화면 기반 핀 히트 + `CoursePlannerPageViewModel`

끝점 근접 판정은 지도거리 상수가 아니라 **화면 포인트 기반 핀 히트**로 정의한다. 지도거리
20m는 줌 배율에 따라 화면에서 수 px~수백 px로 변해, 고배율에서 미세 연장 탭을 삼키고
저배율에서 핀 탭을 놓치기 때문이다. "핀을 탭했다"(화면상 제스처)와 "핀 근처 좌표를 탭했다"
(지리 판정)를 분리한다.

- **판정 위치**: `MapViewRepresentable`의 탭 핸들러가 탭 화면 좌표와 출발/도착(또는 병합) 핀의
  화면 좌표를 비교해 **반경 약 24pt 이내면 핀 히트**로 판정하고, 그 결과를
  `handleMapTap(at:hitPin:)` 형태로 ViewModel에 전달한다(ViewModel은 화면 계산 없이 판정
  결과만 받아 테스트 가능성 유지). 24pt는 시작값이며 실기기 QA에서 조정한다.
- **분기 (상호배제, 위에서부터)**:
  1. 닫힌 코스의 병합 핀 히트 → **무시** + 짧은 안내("이미 닫힌 코스입니다").
  2. 출발핀 히트 → **왕복**: 도착점에서 출발점 좌표까지(좌표 스냅) 라우팅해 attach. 새 구간
     시작점이 도착점이므로 규칙 2로 항상 append — 달리는 순서 유지. 탭 왕복의 대표 케이스.
  3. 도착핀 히트 → **무시** + 짧은 안내("이미 도착점입니다"). 기존 동작은 "도착→도착" 0m 라우팅
     시도로 에러가 떴을 상황이라 가드 겸용. ("도착핀 탭"에 별도 의미를 부여할지는 니즈 확인 후 백로그로.)
     - 설계 노트: 도착핀 탭을 "출발→도착 두 번째 경로"로 대칭 처리하는 안은 기각 — 그렇게 만든
       구간은 시작점이 출발점이라 반전 prepend되어, 출발핀이 물리적 도착 지점으로 이동하는
       라벨 혼동이 재발한다.
  4. 핀 히트 아님 → 기존 동작(가까운 끝점에서 탭 지점으로 연장). 핀에 지리적으로 가까운 좌표라도
     핀 히트가 아니면 미세 연장으로 처리된다.
- **발견성**: 탭 모드에서 코스가 있고 아직 닫히지 않았으면 statusPanel 프롬프트에 힌트를 표시한다
  (예: "출발핀을 탭하면 왕복이 완성됩니다"). 앱에 온보딩/툴팁 패턴이 없어 statusPanel 문구가
  유일한 안내 수단이므로, 숨은 제스처가 시행착오로만 발견되는 것을 막는다.
- 무시(no-op) 케이스의 안내 문구는 기존 statusPanel 표시 체계를 재사용하는 일시 표시로 하고,
  에러 스타일이 아닌 안내 스타일로 구분한다.

### 3. 핀 표시 — `CoursePlannerPage` + `MapViewRepresentable`

- **닫힌 코스 병합 핀**: 코스 첫·끝 좌표가 20m 이내면 닫힌 코스로 보고 "출발/도착" 핀 하나만
  표시한다. 시각: 출발 핀(초록 `figure.run`)을 기본으로 하고 우측 하단에 작은 체크무늬(도착)
  배지를 오버레이. 그 외에는 지금처럼 출발·도착 각각 표시.
- **경유점 마커**: 인접 구간이 만나는 경계 좌표(각 구간의 마지막 좌표, 최종 구간 제외)마다
  작은 무번호 점(흰 원 + 회색 테두리)을 표시한다. 출발/도착 핀보다 확실히 작고 조용한 시각 위계.
- **충돌 우선순위(명시 값)**: 출발/도착/병합 핀은 기존대로 `displayPriority = .required`,
  `collisionMode = .none`(항상 표시). 경유점 마커는 `displayPriority = .defaultLow`,
  `collisionMode = .circle` — 겹칠 때 항상 출발/도착 핀이 이기고, 경유점끼리·거리 라벨과
  겹치면 경유점이 양보한다(조용한 위계의 의도된 동작).
- **구현 노트 — annotation diff**: 현재 `MapViewRepresentable`의 핀 갱신 diff는 좌표·개수
  비교라 스타일만 바뀌는 전환(출발+도착 → 병합 핀)을 놓칠 수 있다. 경유점 마커는 **별도
  annotation 타입으로 분리**해 diff 충돌을 피하고, 핀 diff는 title/스타일 변화까지 비교하도록
  확장한다.

### 4. redo(앞으로 돌리기) — `CourseEditSession` + 컨트롤 UI

- `Entry`에 "앞/뒤 어디에 붙었는지"(prepend 여부)를 기록한다.
- `undo()`가 제거한 entry를 **복원 스택(LIFO)**에 보관한다. `order` 값은 보존한다
  (`nextOrder`는 undo로 감소하지 않으므로 복원 시 충돌 없음).
- `redo()`는 스택에서 꺼내 기록된 자리(맨 앞 또는 맨 뒤)에 되살린다. 여러 번 누르면 차례로 복원.
- **복원 스택은 "entry를 실제로 추가한(성공한) attach" 또는 clear 시에만 비운다.**
  gap 라우팅 실패(`routeNotFound`/스로틀 등)로 중단된 attach는 스택을 보존한다 — attach 진입
  시점에 비우면 라우팅 실패만으로 되돌린 구간들이 영구 소실된다. clear 자체는 redo 대상이 아니다.
- 라우팅 재요청 없음: 저장된 좌표를 그대로 되살리므로 새로운 실패 모드가 없다.
- ViewModel에 `canRedo`/`redo()` 노출. `redo()`도 undo와 동일하게 `selectedSegmentIndex`를
  초기화한다(prepend 자리 복원 시 배열 인덱스가 밀리므로). UI는 되돌리기 버튼 옆 같은 스타일의
  앞으로 돌리기 버튼, 스택이 비면 비활성화.

## 에러 처리

- 탭 왕복 라우팅 실패는 기존 에러 메시지 경로(`routeNotFound`/스로틀/일반)를 그대로 탄다.
- redo는 네트워크/라우팅을 타지 않으므로 추가 에러 처리가 없다.
- 핀 히트 무시 케이스는 에러가 아니라 안내(위 2절)로 처리한다.

## 테스트

단위 테스트 (`TraceTests`):

- attach 새 규칙(규칙 1~4 각각): 닫힌 코스면 append 고정, 도착점 근접 시작=append(뒤집기 없음),
  출발점 근접 시작=반전 prepend, 원거리 스트로크=도착점 gap append(뒤집기 없음), gap 라우팅 경계(20m)
- undo → redo 왕복 복원(자리·order·색상 키 보존), 다중 undo/redo, **성공한** attach 후 redo 스택
  초기화, **실패한 attach 후 redo 스택 보존**, clear 후 redo 불가
- 핀 히트 분기(ViewModel에 판정 결과 주입): 병합 핀 히트=무시, 출발핀 히트=왕복 append,
  도착핀 히트=무시, 히트 없음=기존 동작, 좌표 스냅
- 닫힌 코스 판정(첫·끝 좌표 20m)
- **기존 테스트 주의**: 4쌍 비교를 전제한 테스트는 기대값 수정만으로는 부족하다 — 예컨대
  `testAttach_prependNoGap`류의 prepend fixture는 새 규칙에서 append로 판정되므로, prepend를
  유발하려면 fixture 자체를 "시작점이 기존 출발점 근처"로 재구성해야 한다.

실기기 QA (MVP 완료 시 체크리스트 작성):

- 왕복 그리기/탭 왕복에서 핀·경유점·겹침 오프셋 확인
- 핀 히트 반경(24pt) 체감, 닫힌 코스 판정(20m) 확인 — 라우팅 반환 좌표가 도로망에 재스냅되어
