# MVP12 완료 회고 — Swift 6 + 디자인 (2026-07-10 ~ 07-13)

## 요약

마일스톤 3개: **design-direction**(기존 목업 검토 → 디자인 시스템 추출, 경량 문서 전용 사이클),
**swift6-migration**(암묵/명시 isolation 정리, Sendable 정합, `SWIFT_VERSION = 6` 전환 + 경고 0,
실기기 크래시 발견 후 격리 기본값 전략을 반대 방향으로 재전환), **design-apply**(디자인 시스템을
플래너 화면 전체에 적용 — 토큰·탑바·FAB·시트·구간리스트·핀/폴리라인·저장/목록/왕복/redo).

design-apply는 P1 적용 자체는 빠르게 끝났지만, 이후 실기기·시뮬레이터 실사용 중 바텀시트 관련
버그가 연속으로 발견되어(드래그-리사이즈 3단계 확장, 안전영역 피드백 루프, 헤더 베이스라인 정렬,
FAB 스택 가림) 같은 브랜치에서 여러 차례 추가 수정 사이클을 거쳤다. 전체 테스트 178개 그린,
실기기 QA 체크리스트 통과.

## Keep

- **실측 우선 디버깅이 결국 통했다.** 접근성 트리 정적 덤프(ZStack maxHeight 버그), 개별 하위 뷰
  `onGeometryChange` 실측(안전영역 피드백 루프, 헤더 베이스라인 정렬), XCUITest 프레임 비교(FAB
  스택 가림) — 매번 "코드를 읽고 추론"이 아니라 "실제 값을 찍어서 좁히는" 방식으로만 근본 원인에
  도달했다. 코드 추론만으로 몇 시간을 헤맨 뒤에야 이 전환이 일어난 경우가 반복됐다 — 다음엔 더
  일찍 전환할 것.
- **격리 기본값 전략을 실기기 크래시 후 뒤집은 결정이 옳았다.** MainActor 기본 → 클래식(기본
  nonisolated) 전환은 계획에 없던 재작업이었지만, 프레임워크 연동 타입(MapKit NSObject 서브클래스)
  실제 크래시라는 구체적 증거에 기반한 결정이라 흔들리지 않았다.
- **사용자가 "이전 분석이 잘못됐을 수도 있다"고 직접 의심하고 확인을 요청했다.** 그 결과
  `self-measuring-frame-causes-layout-jiggle.md`가 실제로는 다른 증상만 고치고 진짜 원인(베이스라인
  정렬)을 놓쳤던 게 드러났다. 사용자의 의심이 없었다면 이 정정은 없었을 것 — 완료 선언을 그대로
  믿지 않는 태도가 유효했다.

## Problem

- **"완료"로 표시된 항목을 재검증 없이 신뢰했다.** 이전 세션에서 "움찔거림 원인 찾음"으로
  문서화된 자기측정 순환 버그가, 실제로는 이름이 비슷한 별개 증상을 고쳤을 뿐이었다. 증상
  이름의 유사성만으로 "같은 원인"이라 단정한 게 근본 원인 — 재현 확인 없이 문서를 신뢰한 대가로
  하루 가까이 같은 버그를 다시 쫓았다.
- **디버깅 세션 초반에 여러 차례 "고쳤다"고 성급히 보고했다가 정정했다.** 실제로 시뮬레이터에서
  보지 않고 코드 추론만으로 "이렇게 하면 될 것"이라고 결론 내린 순간마다 나중에 뒤집혔다.
- **XCUITest accessibility identifier가 bottomSheet 서브트리에서 부모로 뭉개지는 문제를 세션
  중반까지 몰랐다.** 이 때문에 초반 진단 시도들이 엉뚱한 이유로 실패했다 —
  `child-accessibility-identifiers-collapse-to-parent-in-bottomsheet.md`로 기록, 다음엔 좌표
  기반 우회를 더 일찍 시도할 것.

## Surprise

- 개별 하위 뷰(칩, 서브타이틀, 좌/우 컬럼)의 높이가 전부 동일한데도 부모 `HStack`의 측정 높이만
  흔들리는 현상 — SwiftUI의 `.firstTextBaseline` 정렬이 "첫 자식의 타입"(Text vs 비-Text)에 따라
  암묵적 베이스라인 기준점 자체를 다르게 잡는다는 사실은 사전에 예상 못 한 SwiftUI 내부 동작이었다.
- `GeometryProxy.safeAreaInsets.top`이 형제 뷰가 화면을 거의 다 채울수록 더 작은 값을 보고하는
  피드백 루프 — "시스템이 보고하는 고정값"이라 믿었던 값이 실제로는 레이아웃 상태에 따라 흔들릴
  수 있다는 것도 예상 밖이었다.

## 남은 기술 부채

- P2 6건(시트 드래그 리사이즈·지도 halo는 이미 당겨서 완료, 남은 건 km 마커·점선 애니메이션·
  다크 글로우·커스텀 저장 다이얼로그) — `docs/backlog.md` "MVP12 design-apply P2" 섹션.
- FAB 아이콘 버튼의 시각적 눈에 띄는 정도(대비) 개선 — 사용자가 "실기기 써보고 불편하면 재조정"
  으로 보류. 지금은 백로그에 명시적으로 올리지 않았으니, 실기기 사용 중 다시 언급되면 그때 등록.
- bottomSheet 서브트리의 accessibility identifier 뭉개짐 원인은 특정하지 못했다(우회만 문서화) —
  재현 조건을 더 좁히면 근본 수정 가능할 수 있음.
