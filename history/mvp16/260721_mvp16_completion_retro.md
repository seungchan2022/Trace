## Keep (잘 된 것)

**"스모크 통과"로 뭉뚱그리지 않고, 검증 불가능한 영역을 정직하게 INCONCLUSIVE로 보고했다**
draw-gesture Task 3에서 이 마일스톤의 핵심 동작(롱프레스-드래그로 실제 그려지는지)을 XcodeBuildMCP로 검증하려 했으나 도구가 "누른 채 유지 후 이동"을 합성하지 못한다는 게 drag/swipe 양쪽 다 시도해봐야 확실해졌다. 여기서 억지로 통과 처리하지 않고 "시뮬레이터 합성 불가"로 명시한 뒤 실기기 QA로 이관했고, 실기기에서 실제로 전체 통과를 확인했다. 최종 브랜치 리뷰(opus)도 이 정직함을 별도로 짚어 강점으로 평가했다.

**구현 서브에이전트가 세션 API 한도로 중단되는 사고에서도 커밋 기반 복구가 작동했다**
draw-gesture Task 1 구현자가 코드 변경·커밋까지 마친 직후 세션이 끊겼다. 커밋 자체는 살아있었으므로 컨트롤러가 diff를 plan 텍스트와 줄 단위로 대조하고, 빌드·테스트·린트를 독립적으로 재실행해 검증을 완료했다 — 서브에이전트의 자가 보고 없이도 커밋과 실제 코드 상태만으로 신뢰할 수 있는 검증이 가능함을 확인했다.

**이전 MVP에서 확립된 QA 인프라·컨벤션이 그대로 재사용됐다**
GPX 시뮬레이션(run-fullscreen 요약 화면 QA), 시나리오 카드+세션 단위 묶기 체크리스트 형식, 실기기 검증 필수 규칙(`testing.md`) 모두 추가 설계 없이 바로 적용됐다. draw-gesture QA 체크리스트도 최종 브랜치 리뷰의 Minor 권고(정지 홀드-릴리즈 엣지케이스)를 받아 코드 수정 대신 체크리스트 한 줄 추가로 정확히 대응했다 — 결정 게이트가 금지하는 인식기 규칙 추가를 피하면서도 갭을 메웠다.

## Problem (아쉬운 것)

**landscape-sheet-overflow는 파생 회귀 2건을 미해결 상태로 남긴 채 병합됐다**
원버그(가로 시트 붕괴)는 3중 방어로 고쳤지만, 그걸 고치는 과정에서 플랜이 이미 범위 밖으로 못박아둔 "가로 전용 레이아웃 최적화(화면 끝까지 늘리기)" 쪽으로 실제로 들어가 버렸다가, 다이내믹 아일랜드·FAB 배치 문제까지 얽힌다는 게 드러나서야 되돌아왔다. YAGNI 경계를 플랜 텍스트로 미리 박아뒀다는 것 자체는 잘한 일이지만, 그 경계를 실제로 지키는 건 별개 문제였다.

**플랜 체크박스 실시간 갱신이 마일스톤마다 들쭉날쭉했다**
`run-fullscreen`·`landscape-sheet-overflow`는 실행 중 체크박스가 한 번도 갱신되지 않아 아카이빙 시점에 소급 노트로 대신해야 했다(MVP15 회고에서 이미 "습관화하자"고 다짐했던 항목). 반면 `draw-gesture`는 이번엔 태스크 완료마다 실시간으로 체크박스를 갱신했다 — 부분적으로 개선됐지만 마일스톤 전체에 걸쳐 일관되진 않았다.

**세션 마무리 메모의 브랜치 상태가 실제와 어긋났다가 나중에야 정정됐다**
landscape-sheet-overflow 세션을 마무리할 때 "미병합 유지"로 적었지만, 실제로는 그 이후 커밋들이 `main`에 병합되고 브랜치가 삭제됐다. 작성 시점의 *의도*를 그대로 기록하고 실제 git 상태를 재확인하지 않아서 생긴 오기록으로, 다음 세션이 "미병합 브랜치가 어딘가 있다"고 오인할 뻔했다(다음 세션에서 발견해 정정).

## Surprise (예상 밖)

**XcodeBuildMCP의 터치 합성 한계가 네 프리미티브를 다 시도해봐야 명확해졌다**
`drag`는 도구 자체 버그(`FBSimulatorHIDEvent does not support touch move events`)로 즉시 실패했고, 대안으로 쓴 `swipe`는 오류 없이 실행되지만 `preDelay`가 "터치다운 후 대기"가 아니라 "제스처 시작 전 대기"로 동작하는 것으로 보여 홀드 임계값이 인식될 기회 자체를 못 얻었다. 하나의 도구 버그처럼 보였던 문제가 실은 두 경로 모두 이 제스처 클래스를 지원 못한다는 더 근본적인 한계였다(`docs/solutions/workflow-issues/xcodebuildmcp-cannot-synthesize-long-press-drag.md`로 기록).

**가로모드 시트 붕괴라는 국소적 버그 수정이 더 큰 설계 질문을 열어버렸다**
파생 회귀 2건을 쫓다 보니 "시트/탭바를 가로에서 화면 끝까지 늘린다"는 접근 자체가 옳은지 재검토가 필요하다는 게 드러났다 — 버그 수정으로 시작한 작업이 화면 문법 자체에 대한 설계 결정으로 번진 경우.

## 다음 MVP 방향

- roadmap에 이미 명시된 **코스 연동(코스 골라 뛰기 + 계획 vs 실제 비교)**이 유력한 다음 MVP 후보다.
- **가로모드 시트 재설계**는 독립적으로 먼저 정하지 않기로 했다 — 다음 MVP가 이 화면을 다시 건드리는지에 따라 우선순위가 갈리므로, 백로그에 일반 항목으로 남겨두고 다음 마일스톤 스코핑 시 후보로 재검토한다.
- (참고, 백로그에 올리지 않음) 그리기 시작 시 햅틱 피드백 아이디어가 세션 중 논의됐으나 사용자가 지금은 스킵하기로 함 — 다시 생각나면 그때 직접 꺼내기로 함.
- **플랜 체크박스 실시간 갱신을 마일스톤 전체에 일관되게 적용** — draw-gesture에서 부분적으로 개선됐으니 다음 마일스톤부터는 처음부터 습관화한다(MVP15 회고와 동일한 다짐이 반복되지 않도록).
