## 1. 범위 결정

MVP16은 다음 세 재료를 **전부** 담는다 (킥오프 선택지 A):

1. **항목 1** — 탭/시트/페이지 구조 개편 (방향 스펙 전반부)
2. **항목 2** — 코스 탐색 그리기 제스처 개편 (방향 스펙 후반부)
3. **포인트 구간 폴리라인 표시** — MVP15 QA 이월 백로그. 같은 지도-표현 설계 영역이라 별도 브레인스토밍 대신 ui-direction 마일스톤에서 표현 방식을 함께 결정.

**범위 밖 (방향 스펙 그대로):** 러닝 트래킹 계산 로직, 기록 저장 스키마 등 비-표현 영역. 표현/구조만 바꾸는 개편이다.

## 2. 구조 결정 (사전 리뷰 의제 종결)

### 2.1 탭 구성 — 2탭 (코스 / 러닝) 유지

- 기록 탭은 지금 승격하지 않는다. 기록 목록·상세는 러닝 탭 안에서 진입하되 **전체화면 페이지 문법**으로 정리.
- 근거: MVP13 결정("기록 목록 입구는 러닝 탭 버튼, 통계가 커지면 탭 승격 — 되돌리기 쉬운 결정")의 트리거가 아직 안 옴. 커스텀 탭바를 직접 만들면 이후 탭 추가 비용이 작아, 러닝 탭 분리 구현을 참고해 나중에 기록을 분리하는 것도 어렵지 않다.

### 2.2 트래킹 중 탭바 — 숨김 (앱 내 탭 전환 진입점 제거)

- 러닝 시작 → 탭바 사라지고 트래킹 전체화면만 존재. 트래킹 중 조작은 **일시정지/종료/포인트(MVP15)** 셋 — 포인트 버튼·포인트 카드는 새 전체화면 문법에서도 유지된다(비-표현 기능이므로 "표현만 바꾸는 개편" 범위 선언상 제거 대상이 아님, 배치는 ui-direction에서 결정).
- 탭바 복귀 시점: **요약 화면까지 숨김 유지** (결정 2026-07-19, 문서 리뷰 후) — 러닝 시작부터 요약 화면을 닫을 때까지가 하나의 전체화면 플로우. 탭바는 러닝 탭 대기 화면에서만 보인다.
- 기술적 잠금이 아니라 UI 진입점 제거 방식. 백그라운드 전환(잠금·홈)은 기존과 동일하게 동작하고 Live Activity가 잠금화면 확인을 담당.
- 근거: ① 기존 설계 상수 "러닝 중 = 화면 꺼짐/백그라운드"와 일치(나이키런 동일 문법) ② 탭 전환을 허용하면 생기는 설계 비용(세션 소유권 승격, 트래킹 중 코스 탭 그리기 등 엣지 검증)을 통째로 제거.
- **트리거:** 트래킹 중 다른 화면을 봐야 할 실사용 필요가 확인되면(예: 코스 연동 MVP에서 코스 확인) 세션 승격 + 자유 전환으로 확장 검토.

### 2.3 러닝 탭 지도 — 완전 제거

- 나이키식 숫자(시간·거리·페이스·목표 진행률) 중심 전체화면. 실시간 폴리라인·스로틀 갱신 코드가 트래킹 화면에서 빠진다.
- 제거 범위는 **러닝 탭 전체** (결정 2026-07-19, 문서 리뷰 후) — 트래킹 화면뿐 아니라 목표 설정(대기)·GPS 탐색 화면의 지도 배경도 제거한다. "지도는 코스 탭 + 기록 상세 두 곳" 원칙이 예외 없이 성립하고, ui-direction이 러닝 탭을 단일 문법(숫자 중심 전체화면)으로 설계할 수 있다.
- 근거: 뛰는 중엔 화면을 거의 안 보고, 볼 때 확인하는 건 숫자. 지도가 사는 곳은 "코스 탭 + 기록 상세" 두 곳이라는 개편 핵심 원칙.
- **트리거:** 실기기에서 "지도가 없으니 허전하다/불안하다" 감각이 확인되면 복원(미니 지도 등) 검토.

### 2.4 커스텀 크롬 베팅 — 확정

- 시스템 탭바/시트 미사용(방향 스펙의 기각 근거 유지: iOS 26 글래스 강제, floating 시트 구조). 그 대가로 VoiceOver·Dynamic Type·세이프에어리어 대응을 직접 책임진다는 것을 **인지하고 베팅**.
- 비용 평가: 세이프에어리어는 기지 영역(`docs/solutions/ui-bugs/safe-area-top-inset-shrinks-with-sibling-size-feedback-loop.md`에 해결 패턴 보유), VoiceOver는 SwiftUI 기본 컴포넌트 + 라벨로 대부분 해결, Dynamic Type 극단 크기의 탭바 잘림은 현 단계(1인 사용) 수용.
- **트리거:** tab-restructure 구현 중 커스텀 시트/탭바의 제스처 충돌·접근성 대응 비용이 마일스톤 범위를 넘게 커지면 시스템 컴포넌트 복귀를 재평가 — 방향 스펙이 보존해 둔 `presentationBackgroundInteraction(.enabled)` 경로(시스템 시트에서도 뒤 지도 터치 가능)가 복귀로이며, 그 경우에도 비율 기반 detent 로직은 시스템 detent로 이식 가능.

### 2.5 그리기 제스처 (항목 2) — 방향 A 유지

- "한 손가락 = 항상 지도 이동, 그리기 = 롱프레스-드래그" 채택 방향을 새 눈으로 재검토 — MVP15 이후 이 방향과 충돌하는 학습 없음, 그대로 확정.
- 모드 통합 옵션은 방향 스펙대로 "A가 실기기에서 검증된 뒤에만" 이연 유지.
- 플랜 요구(사전 리뷰): 롱프레스 임계값 튜닝 + MVP10 탭 판정(더블탭 줌·탭 보류 확정)과의 충돌 점검을 **구체적 검증 태스크로 명시**할 것.
- 플랜 요구 추가(문서 리뷰 2026-07-19): A안의 진짜 관건은 임계값이 아니라 **제스처 중재 재설계** — 그리기 모드에서 꺼둔 지도 스크롤(`isScrollEnabled = false`)을 다시 켜면서 MKMapView 내장 팬 vs 롱프레스-드래그 캡처의 배타 처리(롱프레스 인식 후 지도 팬 차단, 스트로크 종료 후 복원)와 기존 두손가락 팬 GR·그리기 모드 탭 판별기의 거취를 별도 검증 태스크로 명시한다. 이 배타 처리가 실기기에서 실패하면(꾹 누르는 동안·드래그 중 지도가 같이 움직이면) **A안 자체의 반증 조건**으로 취급.

## 3. 마일스톤 구성 (킥오프 선택지 A — 총 4개)

| # | 슬러그 | 성격 | 내용 |
|---|---|---|---|
| 1 | **ui-direction** | 경량 (문서 전용) | 커스텀 탭바 형태·전환 트랜지션, 러닝 전체화면 레이아웃(숫자 문법 + **포인트 버튼 배치**), 기록 상세 **포인트 구간 폴리라인 표현 방식**(구간별 색칠 vs 단일색+화살표 등)을 디자인 차원에서 확정. 코드 없음 — MVP12 design-direction 방식 |
| 2 | **tab-restructure** | 표준 구현 | 앱 루트 커스텀 탭바(+트래킹 중 숨김) + 코스 탭 커스텀 시트 detent 검증·보완(이미 비율 기반 — 방향 스펙 정정 참고) + 플로팅 버튼 시트 연동(이동·페이드·풀 시트 소멸) |
| 3 | **run-fullscreen** | 표준 구현 | 러닝 탭 나이키식 전체화면(러닝 탭 전체 지도 제거, 포인트 버튼 유지) + 기록 목록/상세 전체화면 페이지 정리 + 포인트 구간 폴리라인 표시 |
| 4 | **draw-gesture** | 표준 구현 | 그리기 제스처 개편(§2.5) + MVP10 이력 충돌 검증 |

- 순서는 1 → 2 → 3 → 4 기본. 2·3·4는 상호 의존이 약하므로 필요 시 순서 조정 가능하나, 전부 ui-direction의 결정을 참조한다.
- 근거: MVP12에서 "디자인 결정 선행 → 구현 사이클들이 결정만 참조" 방식이 잘 작동했고, 이번엔 화면 문법 자체를 바꾸므로 디자인 선행 가치가 더 크다.
