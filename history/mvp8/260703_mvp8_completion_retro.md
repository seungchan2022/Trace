# MVP8 완료 회고 — 코스 편집·가독성 UX 마무리 (구간 패널 + 핀치 줌 + 겹침 오프셋)

작성일: 2026-07-03

## Keep (잘 된 것)

**Subagent-driven-development의 task별 리뷰가 실제 버그 2건을 머지 전에 잡음**
course-ux-polish Task 2에서 `restoreScrollPosition`이 스펙이 이미 "결정 완료"로 못박아둔 "재펼침 시 최신
근처였다면 최신으로" 요구사항을 빠뜨린 걸 리뷰가 잡아냈고, 브랜치 전체 리뷰(opus)는 `scrollPosition`의
`.center`/`.bottom` 앵커 불일치를 별도로 잡아냈다. 둘 다 플랜 코드 자체에 있던 결함이었는데, "플랜이 그렇게
써놨으니 맞겠지"라고 넘기지 않고 리뷰어가 독립적으로 trace-through한 덕에 머지 전에 걸러졌다.

**실기기 전용 버그를 임시 print + 콘솔 로그로 확정**
"겹침 오프셋이 실기기에서 안 되는 것 같다"는 애매한 보고를 여러 턴 대화로 좁히다가, 결국 `MapViewRepresentable`에
임시 `print()`를 넣고 사용자에게 한 번 재현시켜 콘솔 로그를 받는 방식으로 전환했다. 로그 한 번으로
"계산이 항상 0"처럼 보이던 현상이 실은 "겹치는 긴 선 자체가 안 만들어지고 있었다"는 걸 확정할 수 있었고,
이어서 "그리기 모드로 온전히 되짚기"를 시켜 `maxDisplacementMeters≈4.0`(설계값 정확히 일치)을 확인해
알고리즘 자체는 처음부터 정상이었음을 증명했다. 추측 기반 대화보다 로그 한 줄이 훨씬 빨랐다.

**사용자의 회의적인 실기기 테스트가 진짜 원인을 드러냄**
사용자가 "이게 우연히 손이 삐뚤어져서 그런 거 아니야?", "정밀하게 겹치면 왜 다시 안 갈라져?"처럼 스스로
가설을 의심하고 재현 조건을 통제해가며 테스트한 덕분에, 겹침 오프셋 알고리즘 문제가 아니라 **더 이전(MVP5~6)
탭 자동 연결 기능이 짧은 연결 구간을 만들어내는** 전혀 다른 원인이 드러났다. 에이전트 혼자 코드만 읽었다면
찾기 어려웠을 결합 지점이었다.

## Problem (아쉬운 것)

**plan.md 체크박스가 이번에도 실행 중 갱신되지 않음**
MVP7 회고에서 이미 지적했던 문제("plan.md 체크박스가 실행 중 갱신되지 않음")가 MVP8에서 **똑같이 반복**됐다.
Task 1~4/1~5가 전부 커밋까지 끝났는데도 두 플랜 파일의 체크박스는 하나도 안 켜져 있었고, 아카이빙 직전에야
발견해 소급 확인 노트로 처리했다. 두 번 반복된 이상 다음 MVP에서는 "Task 완료 = 커밋 + 체크박스 갱신"을
같은 턴에 묶어서 처리하는 습관을 의식적으로 만들어야 한다.

**XcodeBuildMCP 테스트 툴이 44분+ 무응답 행을 유발**
서브에이전트가 지시받은 raw bash 대신 XcodeBuildMCP의 테스트 툴을 사용해, 병렬 테스트가 이 프로젝트의
알려진 크래시성 플레이키 테스트와 만나 무한 대기에 빠졌다. 사용자가 "너무 오래 걸린다"고 직접 멈추지
않았다면 세션이 통째로 낭비될 뻔했다. `docs/agent-rules/testing.md`에 명문화해 재발을 막았지만, MCP 서버의
"Prefer XcodeBuildMCP tools" 시스템 지시와 컨트롤러의 명시적 bash 지시가 충돌할 때 프롬프트만으로는
100% 방지되지 않는다는 걸 확인했다.

**세션 한도로 서브에이전트가 중간에 끊긴 경우가 여러 번**
Task 2·3(overlap-offset) 구현 도중 세션 한도 초기화, 백그라운드 작업 방치 등으로 서브에이전트가
검증/커밋을 마무리 못하고 중단된 경우가 몇 차례 있었다. 다행히 파일 내용 자체는 브리프와 정확히
일치했어서 컨트롤러가 직접 검증·커밋을 이어받는 것으로 해결됐지만, 매번 "어디까지 됐는지" 재확인하는
비용이 들었다.

## Surprise (예상 밖)

**핵심 기능이 의심받았지만 실제로는 처음부터 정상이었다**
겹침 오프셋(MVP8의 핵심 산출물)이 "실기기에서 안 되는 것 같다"는 보고를 받고 상당한 시간을 들여
조사했는데, 결론은 "알고리즘은 처음부터 맞았고, 문제는 테스트 시나리오를 만드는 앞단(탭 자동 연결)에
있었다"였다. 코드 리뷰(유닛테스트 통과 + 2회의 상세한 opus 리뷰)로는 확신할 수 없었던 걸, 실기기
디버그 로그 한 번으로 명확히 가릴 수 있었다.

**겹침 오프셋을 조사하다가 전혀 다른 버그(핀 라벨링)를 발견**
왕복 구간이 prepend되면 "출발"/"도착" 핀이 `course.coordinates.first/last`(배열 순서) 기준이라
실제 뛴 순서와 반대로 붙는다는 걸, 겹침 오프셋 조사 과정에서 우연히 발견했다. 이건 MVP5~6부터 있던
설계 특성이고 이번 마일스톤과 무관하지만, 왕복 코스가 실제로 늘어날수록 계속 부딪힐 문제라 백로그에
상세히 기록해뒀다.

**탭 모드는 구조적으로 "온전한 왕복"을 만들 수 없다**
탭 한 번 = "가장 가까운 기존 끝점 → 그 탭 지점"으로 즉시 라우팅되는 설계라서, 원래 출발점을 다시
탭하면 "그 지점에서 그 지점까지" = 0m로 귀결된다. 그리기 모드는 문제없이 왕복을 만들 수 있지만
탭 모드는 안 된다는 게, 실기기 QA로 처음 명확히 드러났다.

## 남은 기술부채 (`docs/backlog.md` 연결)

- **탭 모드로는 온전한 길이의 왕복 구간을 만들 수 없음** — 브레인스토밍 필요, "왕복" 전용 액션 추가 여부 결정
- **왕복(prepend) 시 출발/도착 핀이 실제 뛴 순서와 반대로 붙음** — 코스 저장 설계와 함께 다룰 가능성
- **두 손가락 탭 줌아웃 vs 커스텀 2손가락 pan, 빠르게 하면 충돌** — 제스처 인식기 우선순위 조정 필요
- **구간 패널: 출발 지점(prepend) 추가 시 스크롤 위치 밀림** — 원인 미확정, 추가 조사 필요
- **짧은 겹침 구간의 테이퍼 미도달, turnaround 시각적 틈, 프레임 히치 등** — QA 체크리스트에 기록됐으나 실기기 검증 미완료
