핵심 의사결정

1. 듀얼 도구 — "완벽히 같게"가 아니라 "같은 능력"

상황·선택지 단일 플랜은 토큰/세션이 빨리 소진된다. 완벽 동일 vs 룰 공유+어댑터 분리.

결정 진입 파일·커맨드만 심볼릭으로 도구별, 룰·플랜·git은 리포 공유.

왜 룰의 도구 중립 분리가 이미 끝나 추가 비용 ≈ 0. 중요한 90%는 이미 리포에 있다.

→ 도구가 아니라 능력(기획·플랜·TDD·리뷰·iOS 빌드)을 맞추는 것이 목표.

2. 폴더 구조 — 단순 이동이 아니라 재설계

결정 Shared/Routing → Domain/CoursePlanning, Route→Course 전면 rename + Clean Architecture 레이어 재배치.

왜 폴더만 옮기면 네이밍이 또 어긋난다. 이전 설계를 버리고 새 설계로 정합화하는 게 핵심.

→ "진짜 블로커 하나"(pbxproj 그룹 모델)만 먼저 확인 → 이동 trivial.

3. 체크박스·compound 규율화

플랜 task별 실행 + 체크박스 실시간 갱신, 리뷰 직후 ce-compound 교훈 추출을 룰로 의무화. 룰 문장 하나는 약하니 워크플로에 박았다.

4. 플러그인 전략 · 토큰

superpowers·compound·XcodeBuildMCP를 Claude Code에도 설치. 스킬은 한 줄 설명만, MCP는 deferred(이름만) → 설치가 세션 수명을 깎지 않는다.

5. compound 통합 gap

Codex superpowers(openai-curated)는 리뷰 끝에 ce-compound 내장, Claude obra(v6.0.2)엔 없음 → 플러그인이 아니라 리포 룰에 연결을 박았다(업데이트에도 생존).

6. 커밋 오염 복구

git commit이 staged 전체를 담는 탓에 잔여가 엉뚱한 커밋에 섞임 → reset(--hard 아님)으로 베이스까지 풀고 깨끗이 6개 재커밋.

배운 것

🎯 체크박스 실시간 갱신이 핸드오프의 핵심. 세션은 예고 없이 죽고, 다음 도구가 읽을 수 있는 건 git + 결정 문서 + 플랜 체크박스뿐. 도구별 메모리는 상대가 못 본다.

• git commit은 staged 전체를 담는다 — 분할 커밋 땐 staged를 정확히 통제.

• 스킬/MCP 설치는 토큰을 거의 안 먹는다.
