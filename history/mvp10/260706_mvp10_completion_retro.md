작성일: 2026-07-06

## Keep (잘 된 것)

**초기 3개 마일스톤(tap-pending-commit, draw-start-pixel-snap, two-finger-gesture-tuning)이 subagent-driven-development로 깔끔하게 완료**
6개 태스크 전부 태스크 리뷰 통과, 최종 전체 브랜치 리뷰(opus)도 Critical/Important 0건으로 통과했다.
탭 판별기 도입, 화면 24pt 히트 판정, 두손가락 제스처 delegate 조정까지 판단이 필요한 통합 지점도
리뷰어가 코드를 직접 손으로 추적해 검증했다.

**적대적 서브에이전트 리뷰가 실제로 위험한 설계를 두 번 잡아냈다**
근접 끝점 대칭 처리(1차 수정) 때, "끝점에 대해 독립적인 절대 임계값 두 개를 쓰는" 첫 제안이
"거의 닫힌 루프" 코스에서 1m 차이로 결과가 정반대로 뒤집히는 반례를 서브에이전트가 직접 구성해
기각시켰다. 원거리 확장(2차 수정) 때는, TDD로 구현하던 서브에이전트 스스로가 브리프의 "정확한
교체 코드"에 있던 실제 회귀(닫힌 코스가 규칙 1을 어기고 prepend될 수 있는 버그)를 발견하고, 임의로
고치지 않고 정확히 BLOCKED로 보고했다. 두 경우 모두 "설계를 검토용으로 넘기고 결과를 신뢰하기보다
반례를 직접 구성해보라"는 지시가 실제 결함을 찾아냈다.

**advisor가 (백엔드 장애 복구 후) 내가 과신했던 안전성 주장을 정확히 잡아냈다**
원거리 확장을 설계하면서 "근접 처리가 이미 왕복 모호성을 가로채므로 원거리 영역은 안전하다"고
과도하게 일반화했는데, advisor가 "타이트한 왕복만 근접 분기가 막는다 — 평행하게 반대 방향으로 그은
스트로크는 원거리 영역에서도 양끝이 서로 다른 핀을 가리키며 거의 동률일 수 있다"고 정확히 반박하며
구체적인 반례(30m 옵셋 평행 스트로크) 구성을 요구했다. 이 지적 덕분에 다음 리뷰에서 그 반례를 직접
검증하고서야 설계를 확정할 수 있었다.

**사용자가 "이미 알려진 한계"로의 재분류 시도를 정확히 정정했다**
시나리오 19 재실패를 보고받고 내가 "알려진 원거리 제약"으로 재분류하려 하자, 사용자가 "시나리오
19도 실패를 했다는거야. 내가 말한 상황이 이미 실패한 상황 아니야?"라고 명확히 정정했다. 이 정정이
없었다면 실제 버그가 백로그의 "알려진 한계"로 조용히 묻힐 뻔했다.

## Problem (아쉬운 것)

**같은 근본 원인의 버그를 두 번에 걸쳐서야 완전히 고쳤다**
"attach 판정이 스트로크 끝점을 안 본다"는 근본 원인은 한 번(근접 판정, commit c8b6f0f)에 끝내지
못했다. 원거리(Rule 4) 쪽은 그대로 뒀는데, 그게 정확히 시나리오 19가 실기기에서 다시 실패한 지점이
됐다. 근접 수정 시점에 "이 근본 원인이 원거리에도 똑같이 적용되는가"를 먼저 점검했으면 한 번에
끝났을 작업이 두 라운드로 늘어났다.

**advisor 백엔드 장애가 세션 중 반복됐다**
"Response stalled mid-stream" 오류가 advisor 직접 호출과 서브에이전트(model:opus) 디스패치에서
여러 차례 발생했다. `SendMessage`로 트랜스크립트에서 재개하는 방식으로 우회했지만, 그때마다 사용자가
"어드바이저에 문제 있는거 아니야?"를 반복해서 물어야 했다.

**QA 체크리스트 "쉬운 말" 규칙이 스텝 본문에만 반영되고 헤딩/인트로/푸터에는 두 번 누락됐다**
사용자가 "처음 쓰는 유저 기준으로" 요청한 뒤, 스텝 본문은 고쳤지만 섹션 제목·안내문에 여전히 규칙
번호·백로그·마일스톤 이름 같은 용어가 남아있어 사용자가 "이게 템플릿에 반영이 안됐어?"로 재지적해야
했다. 템플릿(`docs/agent-rules/testing.md`) 갱신 시 "헤딩/인트로/푸터도 포함"을 명문화한 뒤에야
안정됐다.

## Surprise (예상 밖)

**원거리 확장의 안전성 증명이 근접 수정보다 오히려 더 튼튼했다**
`anchorIsEnd = endMin < startMin`(스트로크 양끝 각각 "더 가까운 핀까지 거리" 중 작은 쪽 선택)은
4개 조합(양끝×양핀) 중 실제로 가장 짧은 연결을 고르는 것과 수학적으로 동일함이 증명됐다 — 근접
수정의 임시방편적 타이브레이크보다 더 명확한 안전성 근거를 갖게 됐다.

**정확한 스펙대로 구현하라고 지시받은 서브에이전트가 스펙 자체의 결함을 찾아냈다**
"이 정확한 코드를 그대로 적용하라"는 브리프를 받은 구현자가, TDD로 실제로 테스트를 돌려보다가
브리프의 코드 자체가 회귀를 낸다는 걸 발견하고 임의로 고치지 않았다. "정확한 스펙 = 검증 불필요"가
아니라는 걸 구현자-리뷰어 분리 원칙이 실제로 증명한 사례.

**세션 한도로 서브에이전트가 중간에 끊겨도 작업이 안전하게 이어졌다**
구현자 서브에이전트가 수정 코드를 적용하던 도중 세션 한도로 끊겼는데, 파일 상태를 직접 읽어 어디까지
반영됐는지 확인하고 컨트롤러가 그대로 이어받아 완료할 수 있었다 — 중간 상태가 파일에 그대로 남아있어
복구에 손실이 없었다.

## 마일스톤별 핵심 의사결정

| 항목 | 결정 | 이유 |
|---|---|---|
| 탭 즉시 확정 | 0.3초 보류 → 확정/취소(TapClassifier) | 더블탭·원핑거 줌과 즉시 확정이 계속 충돌 |
| 그리기 시작점 근접 판정 | 화면 24pt 히트 + 실거리 20m 폴백 | 줌 레벨과 무관하게 일관된 히트 판정 |
| 두손가락 제스처 경쟁 | 네이티브 `UIPinchGestureRecognizer` 명시적 제외 | 커스텀 2손가락 팬과 충돌해 끊김 발생 |
| 원거리 방향 판정 1차 재설계 | 시작점 단일 최근접 비교(탭 nearestEndpoint와 동일 `<=`) | 왕복 모호성 재발 방지하며 "중간지대" 문제 해결 |
| 근접 끝점 대칭 처리 | 근접(threshold 이내) 판정에 한해 끝점도 대칭, 상대 비교(`<=`) 단일 결정 | 독립 절대 임계값 두 개는 근접-루프 코스에서 1m 차이로 결과 반전 위험 |
| 원거리 끝점 대칭 처리(2차) | 스트로크 양끝 중 더 가까운 쪽(anchor)을 단일 상대 비교(`<`)로 선택, 닫힌 코스는 게이트 밖 | 근접 처리가 이미 왕복 모호성을 가로채 원거리 영역은 구조적으로 안전; 닫힌 코스 게이트 누락은 TDD로 즉시 발견해 수정 |
