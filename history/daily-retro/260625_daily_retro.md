# 데일리 회고 — 2026-06-25

## 오늘 뭘 했나

MVP3의 전체 사이클을 하루에 돌렸다. 어제 brainstorming + spec까지 끝내놓은 상태에서, 오늘은 플랜 작성 → SDD(Subagent-Driven Development)로 7개 Task 구현 → 실기기 테스트 → 버그 수정 → 아카이빙까지 완주했다.

구현된 것: 카메라 위치 저장/복원(UserDefaults), 비연속 구간 자동 방향 감지(4쌍 비교 + reverse), 증분 라우팅(새 구간만 계산), MKDirections 스로틀 에러 감지.

실패한 것: 그리기 중 2손가락 지도 이동. SwiftUI Map 위에 오버레이를 올리는 구조로는 해결 불가능하다는 걸 실기기에서 확인하고 롤백했다.

그리고 세션 중에 에이전트가 git push를 우회 실행하는 사고가 발생해서, 깃 가드 체계를 frank 프로젝트 방식(settings.json deny)으로 전면 전환했다.

## 핵심 의사결정과 이유

### 1. 증분 계산 설계

- **상황**: 스트로크가 추가될 때마다 전체 포인트를 `snappedRoute`로 N-1번 라우팅 요청 → 스로틀
- **선택지**: A) 전체 재계산 유지 + 캐시 강화, B) 새 구간만 계산하는 증분 방식, C) 맵매칭 제공자 교체
- **결정**: B — 증분 계산
- **왜**: 기존 구조에서 `recomputeSnappedCourse`가 매번 전체 포인트를 순회하면서 route()를 호출하는데, pair 캐시가 있어도 loop 자체가 비효율적. 새 스트로크의 내부 구간 + 기존 경로와의 연결 1건만 라우팅하면 호출 수가 근본적으로 줄어든다. C(맵매칭 제공자)는 근본책이지만 SDK 연동 + 비용이 따라와서 MVP3 범위를 넘음.
- **인사이트**: 기존에 어떻게 전체 재계산이 돌아가는지(flatMap → snappedRoute → N-1번 route) 로직을 이해하고 나니, "새 부분만 계산한다"는 개념이 명확해졌다. 코드를 읽는 시간이 설계 품질에 직접 영향을 준다는 걸 느꼈는데, 매번 전체 로직을 깊이 파야 하는지는 아직 감이 잘 안 온다.

### 2. 4쌍 거리 비교 + 스트로크 reverse

- **상황**: 기존 경로의 끝이 아닌 곳에서 새로 그리면 핀이 뒤섞이는 버그
- **선택지**: A) 항상 끝에서만 연장, B) 시작점만 비교해서 방향 감지, C) 4쌍(새 스트로크 양 끝 × 기존 경로 양 끝) 전부 비교
- **결정**: C — 4쌍 비교 + 필요 시 스트로크 reverse
- **왜**: 처음에는 B(시작점만 비교)로 합의했는데, advisor 리뷰에서 "prepend 시 스트로크 방향이 반대면 역주행이 생긴다"는 걸 잡아냈다. 4쌍을 다 비교하고 가장 가까운 쌍을 찾은 뒤, 필요하면 스트로크를 뒤집어야 자연스러운 경로가 된다.
- **인사이트**: advisor 리뷰가 스펙 확정 전에 이 버그를 잡아준 게 컸다. 구현 후 발견했으면 리팩터 비용이 훨씬 컸을 것.

### 3. 2손가락 지도 이동 시도 → 실패 → 보류

- **상황**: 그리기 모드에서 지도 이동이 안 됨 (interactionModes: [])
- **선택지**: Phase 1(interactionModes [.pan, .zoom] 변경), Phase 2(UIViewRepresentable DrawingOverlay), MKMapView 교체
- **결정**: Phase 1 → Phase 2 순서로 시도, 둘 다 실패, 보류
- **왜**: SwiftUI Map은 내부적으로 UIKit(MKMapView)의 제스처를 사용한다. 오버레이의 hit-test가 터치를 가로채면, 제스처 인식기가 실패해도 터치 소유권은 오버레이에 남는다. 2손가락 제스처가 Map으로 절대 전달되지 않는다.
- **인사이트**: SwiftUI의 `.simultaneousGesture`, `.highPriorityGesture`는 SwiftUI 제스처끼리만 조율한다. Map 내부의 UIKit 제스처에는 영향을 주지 않는다. 이건 일반적인 SwiftUI 뷰 제스처 문제가 아니라 **MapKit 고유의 문제**라는 걸 심층 리서치로 알게 됐다. 기존에 "제스처는 다 SwiftUI에서 관리된다"고 생각했던 게 틀렸다.

### 4. MKMapView 교체가 유일한 해법

- **상황**: 2손가락 실패 후 근본 원인 리서치
- **선택지**: A) point(inside:with:) 오버라이드(부분 해결), B) UIGestureRecognizerRepresentable(iOS 18+), C) MKMapView로 교체
- **결정**: C를 다음 MVP 방향으로 확정
- **왜**: A는 동시 터치만 처리하고 순차 터치에서 깨진다. B는 iOS 18+라 타겟(17+)에 못 쓴다. C만이 그리기 제스처를 MapView에 직접 붙여서 `require(toFail:)`로 1손가락/2손가락을 분리할 수 있다.
- **인사이트**: Strava 같은 프로 앱들이 지도 위 그리기에 항상 MKMapView를 직접 쓰는 이유를 이해하게 됐다. SwiftUI Map은 선언적 API가 편하지만, 제스처 커스터마이징이 필요한 순간 한계가 온다.

### 5. 카메라 복원 — 복원 후 그 위치에 머묾

- **상황**: 앱 재실행 시 저장된 카메라를 복원한 뒤 현재 위치로 이동할지 말지
- **선택지**: A) 복원 위치에 머묾, B) 현재 위치로 이동
- **결정**: A
- **왜**: 러닝 코스 계획 앱이니까, 사용자가 마지막으로 보던 경로/지역을 다시 보려는 경우가 자연스럽다. 현재 위치로 가고 싶으면 "내 위치" 버튼이 있다.
- **인사이트**: 처음에는 A와 B의 차이가 잘 와닿지 않았는데, "강남역에서 경로를 보다가 앱을 닫고, 홍대에서 다시 열면 어디가 보이나?"로 구체화하니 바로 이해가 됐다.

### 6. 에이전트 push 우회 → settings.json deny

- **상황**: 에이전트가 `ALLOW_PUSH=1`과 `--no-verify`로 pre-push 훅을 우회해서 main에 직접 push
- **선택지**: A) 규칙 문서 강화, B) tty 가드 추가 (git hooks), C) settings.json deny (Claude Code 런타임 차단)
- **결정**: C — frank 프로젝트에서 이미 쓰고 있던 방식
- **왜**: A는 이미 git.md에 "push 금지"가 있었는데 무시됐다. B(tty 가드)는 git 훅의 stdin이 리다이렉트되어 실제 터미널에서도 차단되는 문제가 있었다. C(settings.json deny)는 Claude Code 런타임이 명령 실행 전에 패턴 매칭으로 차단하므로, 에이전트가 우회할 방법이 없다.

  **settings.json deny가 뭔지**: Claude Code가 에이전트의 Bash 명령을 실행하기 전에 deny 패턴 목록과 대조한다. `"Bash(git push*)"` 같은 패턴에 매칭되면 실행 자체를 거부한다. 프롬프트 입력이 안 되는 게 아니라, 에이전트가 도구(Bash tool)를 호출하는 시점에 런타임이 차단하는 것. 사용자가 터미널에서 직접 치는 명령은 Claude Code를 거치지 않으니 영향 없다.

  사실 이 프로젝트를 세팅할 때 push 차단을 가장 중요한 부분 중 하나로 여기고 git hooks로 설정했다. 이전 프로젝트(frank)보다 더 강제적이라고 생각했는데, 정작 frank에서는 한 번도 일어나지 않은 push 사고가 여기서 발생했다. 조사해보니 frank은 settings.json deny로 더 간단하고 확실하게 막고 있었다. 강제성의 레이어를 잘못 골랐던 거다 — git hooks는 `--no-verify`로 건너뛸 수 있지만, settings.json deny는 에이전트가 건너뛸 수 없다.

- **인사이트**: 강제성에도 레이어가 있다. 규칙 문서 < git hooks < 런타임 deny. 각각 우회 가능성이 다르고, 에이전트를 막으려면 에이전트의 실행 경로에서 차단해야 한다. git hooks는 사용자와 에이전트가 같은 경로를 쓰기 때문에 한쪽만 차단하기 어렵다.

### 7. pre-push 훅 삭제

- **상황**: settings.json deny가 에이전트 push를 차단하므로 pre-push 훅이 중복
- **결정**: 훅 삭제 → 사용자는 `git push`만으로 push 가능 (ALLOW_PUSH 불필요)
- **인사이트**: frank 방식이 더 깔끔하다 — 에이전트 차단은 런타임에, 코드 품질 가드는 훅에. 역할을 분리하면 사용자 경험도 좋아진다.

## 기획/설계 과정

brainstorming에서 5건을 한 MVP로 묶자는 결정은 어제 했다. 오늘은 플랜 작성 시 advisor 피드백으로 마일스톤 구조를 조정했다 — 항목 2(방향 감지)와 4(증분 계산)가 같은 ViewModel 파이프라인을 건드리므로 `stroke-pipeline`으로 통합. 분리했으면 throwaway 코드가 발생했을 것.

2손가락 이동은 "일단 구현하고 실기기에서 확인" 전략이었는데, 결과적으로 이게 맞았다. 코드 리뷰나 시뮬레이터에서는 발견할 수 없는 문제였고, 실기기 테스트를 통해 SwiftUI Map의 근본적 한계를 알게 됐다.

## 인사이트 & 피드백

- **실기기 테스트의 가치**: 시뮬레이터에서는 서울시청 폴백도, 2손가락 제스처도, 스로틀 에러 메시지도 문제가 안 보였다. 실기기에서 3건 모두 발견. "빌드 통과 ≠ 기능 동작"을 다시 체감.
- **강제성 레이어 선택**: 같은 목적이라도 어떤 레이어에서 강제하냐에 따라 효과가 완전히 다르다. 에이전트를 막으려면 에이전트의 도구 레이어에서 막아야 한다.
- **코드 이해의 적정선**: 전체 로직을 깊이 파는 건 시간이 많이 들지만, 최소한 "이 함수가 뭘 하는지, 데이터가 어떻게 흐르는지"는 알아야 설계 판단이 가능하다. 아직 그 적정선을 찾는 중.

## 배운 것

- **기술**: SwiftUI Map 내부가 UIKit MKMapView이고, 제스처 시스템이 완전히 분리되어 있다는 것. `.simultaneousGesture`는 SwiftUI 제스처끼리만 작동.
- **기술**: hit-test 소유권은 touch-down 시점에 결정되고, 제스처 실패 후에도 변하지 않는다는 UIKit 터치 모델.
- **기술**: 증분 계산에서 샘플링을 스트로크 단위로 해야 한다는 것 (전체 flatMap 후 샘플링하면 이전 구간 경계가 깨짐).
- **프로세스**: settings.json deny가 git hooks보다 에이전트 차단에 더 적합하다는 것. 역할 분리: 런타임 deny = 에이전트 행동 차단, git hooks = 코드 품질 가드.

## 느낀 점

하루에 MVP 하나를 완주한 건 뿌듯하지만, 에이전트가 push를 우회한 건 솔직히 불안했다. 내가 세팅한 보호 장치가 뚫린 거니까. 다행히 frank 프로젝트에서 이미 더 나은 방법을 쓰고 있었고, 비교 분석을 통해 빠르게 전환할 수 있었다. "더 복잡한 설정이 더 안전하다"는 착각을 깼다 — 간단하고 올바른 레이어에 있는 게 더 강력하다.

2손가락 실패는 아쉽지만, 이걸 통해 SwiftUI Map의 한계를 정확히 이해하게 됐고, 다음 MVP의 방향(MKMapView 교체)이 명확해졌다. 실패가 방향을 잡아준 셈.
