# 2026-07-08 하루 회고

## 오늘 뭘 했나

MVP11(코스 저장 + 구간 왕복)을 완전히 매듭지은 날. 커밋 23개, +2,166/−576. 하루가 세 덩어리로
나뉘는데, 가운데에 "세션 리셋"이라는 분기점이 있다.

- **오전 — 기능 마무리와 정정 두 건.** 실기기 QA에서 발견된 왕복 버그(역+정 병합으로 거리가
  3배가 되던 것)를 "역방향만 붙여 정확히 2배"로 재설계하고 전체 왕복 버튼도 추가했다. 그리고
  구현까지 다 끝냈던 초안 자동 저장·복원을 실사용 판정으로 통째로 제거했다 — 앱을 완전히 껐다
  켰을 때 옛 코스가 저절로 살아나는 게 오히려 원치 않는 동작이었다.
- **분기점 — 줌아웃 버그 4연패와 세션 리셋.** 저장 알럿에 이름을 입력하는 동안 지도가 줌아웃되는
  버그를 4번 고치려 했는데 전부 증상만 가리는 패치였고, 패치가 새 부작용을 낳았다. 5번째 시도
  대신 시도한 코드를 전부 되돌리고, "확정된 사실 / 막다른 시도 4개 / 금지 사항 / 미해결 질문"을
  인수인계 문서로 남기고 세션을 끝냈다.
- **오후 — 새 세션, 새 모델, 한 번에 해결.** 인수인계 문서에서 출발해 가설(SwiftUI keyboard
  avoidance)을 세우고, 시뮬레이터에서 bounds 로그로 원인을 관측한 뒤, `.ignoresSafeArea(.keyboard)`
  **1줄**을 body 최상위에 넣어 해결했다. 실기기 QA 전 시나리오 통과. QA 콘솔에서 나온 SwiftData
  스레딩 경고도 "다음에 하자"였다가 규모를 보고 당일 수정(lazy 생성)했고, 그 과정에서 전체
  리빌드가 Swift 6 관련 경고 40건을 드러내서 별도 리팩토링 항목으로 백로그에 정리했다.
- **마무리.** MVP11 아카이빙(산출물 6개 → `history/mvp11/`), 병합·푸시는 직접, 끝난 브랜치 삭제.

## 핵심 의사결정과 이유

### 결정 1 — 줌아웃 버그: "리사이즈 이후 수습"을 전부 기각하고 레이아웃 층위에서 차단

- **상황:** 알럿 TextField의 키보드가 뜨면 지도가 줌아웃. 이전 4회 시도(writeback 차단 플래그,
  setRegion 보정, async 지연 등)는 모두 "지도 프레임이 줄어든 **이후**의 값을 어떻게 수습할까"였고
  전부 실패.
- **선택지:** ① 수습 계열을 더 정교하게 ② 프레임이 줄어드는 것 자체를 막기 ③ 알럿을 sheet로
  교체(구조 변경).
- **결정:** ②. 키보드가 뜰 때 SwiftUI 자동 keyboard avoidance가 지도 프레임을 축소시키는 게
  근본 원인임을 로그로 확정하고(576→275 실측), `.ignoresSafeArea(.keyboard)`를 body 수정자 체인
  **가장 바깥**에 적용.
- **왜:** 원인이 "프레임 축소"인데 축소된 값을 수습하는 건 정의상 증상 패치다. 4연패가 그 증거.
- **인사이트:** 같은 수정자도 **적용 위치가 다르면 다른 답**이다. 이 수정자는 이전 세션에서
  이미 시도됐다가 "실패"로 기각된 답이었는데, 안쪽 뷰에 적용하면 바깥 인셋 레이아웃이 이미
  줄어들어 무효가 된다. 정답이 위치 문제로 오답 처리돼 있었다.

### 결정 2 — 왕복 재설계: 역+정 3× 대신 "역방향만 2×"

- **상황:** 실기기 QA에서 구간 왕복 버튼이 거리를 정확히 2배가 아니라 3배로 만드는 버그 발견
  (역방향+정방향을 둘 다 붙이고 있었다).
- **선택지:** ① 기존 의미(역+정)를 유지하며 거리만 보정 ② 의미 자체를 "역방향 하나만"으로 교체.
- **결정:** ②. 자유 끝 구간에서 역방향만 붙여 코스 끝이 그 구간 시작점으로 되돌아오게.
- **왜:** 백로그에 이미 "되짚어 오기"(막다른 길에서 돌아 나와 이어 그리기) 요구가 있었는데,
  ②가 정확히 그 동작이다. 버그 수정과 기능 요구가 하나로 합쳐져서 별도 기능을 만들 필요가
  없어졌다. 실사용 시나리오(방파제, 막다른 골목)에 맞는 의미론이기도 하다.
- **인사이트:** 버그 수정 방향을 정할 때 백로그를 먼저 보면, "고치기"와 "만들기"가 하나로
  합쳐지는 경우가 있다.

### 짧게 — 세션 리셋 (결정 3)

4회 실패 시점에 5번째 시도가 아니라 **기록하고 멈추는 것**을 선택했다. 컨텍스트가 실패한 접근
방향에 고착되면 같은 곳만 계속 파게 되는데, 실패 기록(무엇이 왜 안 됐는가, 뭘 다시 하면 안
되는가)을 자산으로 정리해 넘기면 다음 세션이 그 지점에서 출발한다. 실제로 오후 세션은 문서
덕분에 재조사 없이 가설 수립부터 시작해 한 번에 끝냈다. 인수인계 문서가 작동한 첫 실증 사례.

### 짧게 — 초안 자동 저장 제거 (결정 4)

만들어서 실기기로 써본 뒤에야 "자동으로 되살아나는 동작 자체가 별로"임을 알았고, 고치는 대신
제거하고 스펙을 정정했다("완전 종료 = 빈 상태 시작"). 아깝지만 실사용 판정이 스펙 확정의
근거라는 원칙대로. 구현 전에 실사용 가치를 검증할 방법이 있었다면 더 쌌을 것.

## 기획/설계 과정 — 재분석을 어떻게 좁혔나

새 세션에서 상위 모델로 바꾸며 "문서를 전부 재분석할까?"를 먼저 정해야 했다. 선택지는
① 바로 진행하며 분석 ② 문서 전면 재분석 ③ 로그로 확정된 실증 사실은 그대로 믿고, 문서 스스로
"한 번도 조사 안 됨"이라 표시한 질문("왜 키보드가 지도 bounds를 줄이는가")만 가설을 세워 검증.

③을 골랐다. 실기기 로그로 확정된 사실은 모델이 바뀐다고 달라지지 않는다 — 이전 실패는 사실
파악이 아니라 **전략**(증상에 반응하는 패치)의 실패였으니까, 상위 모델의 가치는 재검토가 아니라
가설 형성에 쓰는 게 맞았다. 결과적으로 몇 분의 가설 정리가 "무작정 찍어보기"를 "가설 확인
한 번"으로 바꿨다.

재현 환경도 설계가 필요했다. 시뮬레이터에서 지도 탭 자동화가 막혀 있어서(기록된 한계) 코스를
만들 수 없었는데, 이 버그에 필요한 건 코스가 아니라 "알럿+키보드"뿐이라는 걸 깨닫고 런치 인자로
알럿만 자동 표시하는 조사용 스캐폴딩을 넣었다. 소프트웨어 키보드는 시뮬레이터 메뉴 토글로 강제.
오히려 지도·코스 변수가 빠져서 인과가 더 선명해졌다.

## 인사이트 & 피드백

- **실패 기록에는 "무엇을"만이 아니라 "어디에/어떻게"까지.** `.ignoresSafeArea(.keyboard)`가
  조건(적용 계층) 없이 "실패"로 기각돼 있어서 정답이 묻힐 뻔했다. 다음부터 가설을 기각할 때는
  시도 조건을 반드시 함께 기록한다. (`docs/solutions/`에 교훈 문서로 박제)
- **막히면 더 파지 말고, 실패를 자산으로 정리해서 리셋.** "3회 실패하면 승격"을 기준선으로.
  리셋은 후퇴가 아니라 실패 데이터를 압축해서 넘기는 전달이다.
- **"고쳐졌다"의 기준은 증상이 아니라 원인의 소멸.** 줌아웃이 "안 보인다"가 아니라 프레임
  리사이즈가 "로그에서 사라졌다"를 통과 기준으로 삼았다. 증상 기준이었다면 수습 계열 패치도
  통과했을 것이다.

## 워크플로우 고민 — 버그에도 전체 플로우를 태워야 하나?

MVP 기능 개발은 superpowers 플로우(브레인스토밍→스펙→플랜→실행→리뷰)를 따르는 게 맞는데,
버그 하나에 이 전체를 태우는 건 비효율이라는 고민이 있었다. 오늘 하루가 답의 실례가 됐다 —
버그는 "코드가 이미 정해진 스펙과 다르게 동작하는 것"이라 열린 질문이 없으므로 전체 플로우가
아니라 규모별 경로를 탄다:

| 규모 | 경로 | 오늘의 실례 |
|---|---|---|
| 소형 | 백로그 기록 → 같은 브랜치에서 바로 분석·수정·테스트·커밋 | SwiftData affinity 경고 |
| 중형 | systematic-debugging만: 가설→계측→최소 수정→원인 소멸 검증 | 줌아웃 (오후) |
| 대형 | 3회+ 실패 또는 구조적 원인 의심 → 인수인계 문서 + 세션 리셋 | 줌아웃 (오전) |

승격 기준 두 가지: ① 제품 결정을 바꾸는가(그러면 스펙/사용자 상의로) ② 같은 버그에 3회
실패했는가(그러면 대형으로). 이 경로를 `workflow.md`에 규칙으로 박제할지 검토 중.

## 배운 것

- **알럿(UIAlertController)의 TextField 키보드에도 배경 SwiftUI 뷰가 반응한다.** 키보드 알림은
  전역이라 포커스가 알럿에 있어도 뒤 페이지가 keyboard avoidance를 수행한다.
- **`ignoresSafeArea`는 수정자 체인에서의 위치가 의미를 바꾼다.** 인셋(`safeAreaInset`)보다
  바깥에 있어야 키보드 리전이 인셋 레이아웃 계산에 들어가기 전에 제거된다.
- **actor 직렬화와 컨텍스트 affinity는 별개 문제.** 저장소가 actor라 동시 접근이 없어도,
  main에서 만든 `ModelContext`를 actor 스레드에서 쓰면 경고가 난다. 컨텍스트는 쓰는 곳에서
  (lazy) 만든다.
- **증분 빌드는 경고를 숨긴다.** 전체 리빌드를 하고서야 `SWIFT_DEFAULT_ACTOR_ISOLATION =
  MainActor`가 만드는 경고 40건이 보였다. 순수 도메인 값 타입까지 암묵 @MainActor가 되어
  있었고, Domain 프로토콜에 붙은 명시적 @MainActor도 이때 재발견했다.
- **부분 수정은 경고를 옮길 뿐일 수 있다.** `nonisolated`를 한 함수에만 붙였더니 경고가 그
  함수 몸통 안으로 이동했다. 타입 단위 일괄 정리가 필요한 문제는 부분 수정으로 건드리면 안 된다.

## 느낀 점

4연패 후의 리셋은 솔직히 자존심 상하는 결정이었을 텐데, 오후에 그 문서에서 출발해 1줄로
끝났을 때의 통쾌함이 그걸 다 상쇄했다. "실패를 잘 기록하는 것"이 실제로 다음 시도의 속도가
된다는 걸 몸으로 확인한 날. 초안 자동 저장을 지울 때는 아까웠지만, 써보기 전엔 몰랐을 판단이라
후회는 없다. MVP11이 기능 → 버그 → 리셋 → 해결 → QA → 아카이빙까지 한 호흡에 닫힌 것도
뿌듯하다.
