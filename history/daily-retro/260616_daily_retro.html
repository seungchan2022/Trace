Trace 세팅과 에이전트 작업 흐름을 정리한 날

오늘은 앱을 구현하기보다, 앱을 안전하게 만들기 위한 바닥을 놓았다. 완벽한 세팅을 기다리기보다 시작 가능한 기준선을 만들고, 실제 작업을 하며 Trace에 맞게 계속 조정하기로 했다.

오늘 뭘 했나

Trace라는 새 iOS 앱을 시작하기 전에 개발 환경과 에이전트 작업 흐름을 정리했다. 이전 Claude Code 프로젝트에서는 멘토에게 받은 하네스 엔지니어링 자료와 세팅을 많이 활용했지만, 이번에는 그 경험을 그대로 복사하지 않고 Trace에 맞게 다시 조립해보려고 했다.

Superpowers, Compound Engineering, Build iOS Apps, XcodeBuildMCP를 설치하고, iOS 17.0+, SwiftUI, MVVM, Clean Architecture 방향의 규칙을 문서화했다. `/trace-init`, `/daily-retro`, compact/history 설정도 넣어서 세션이 끊겨도 다시 이어갈 수 있게 했다.

이후 Xcode 기본 프로젝트를 만들고, 중첩된 구조를 repo 루트 기준으로 정리했다. `Trace.xcodeproj`, `Trace/`, `TraceTests/`, `TraceUITests/`가 루트에 오도록 맞췄고, 배포 타깃도 iOS 17.0으로 낮췄다.

핵심 의사결정

범용 하네스 위에서 시작하기

직접 하네스를 처음부터 만들기보다 Superpowers와 Compound Engineering 같은 범용 도구를 먼저 적용했다. 이전 프로젝트의 설정을 그대로 복사하면 Trace에 맞지 않는 규칙까지 가져올 수 있어서, 이번에는 적용 후 덜어내는 방향을 택했다.

완벽한 세팅보다 시작 가능한 기준선

아직 앱 기능이 없기 때문에 지금 구조와 규칙이 완벽한지 알 수 없다. 그래서 완벽한 하네스를 먼저 만들기보다, 안전한 기본값을 만들고 실제 작업에서 불편한 부분을 계속 조정하기로 했다.

Git 강제성은 push와 main 커밋에 집중

처음에는 commit, push, merge, rebase까지 넓게 막았지만 checkout과 merge 흐름에서 혼란이 생겼다. 커밋은 로컬 기록이라 고칠 수 있지만, push는 원격 상태를 바꾸므로 더 강하게 막아야 한다는 기준이 생겼다.

Xcode 기본 프로젝트로 시작

아직 모듈 경계가 없기 때문에 Tuist나 XcodeGen은 미뤘다. 기본 Xcode 프로젝트로 시작하고, 기능이 커져 구조가 보이면 모듈화를 검토하는 편이 낫다고 판단했다.

인사이트

세팅은 정답을 맞히는 문제가 아니었다. 이전 프로젝트에서 좋았던 하네스도 새 프로젝트에 그대로 들어오면 과할 수 있다. 반대로 처음에는 약해 보이는 규칙도 실제 작업을 하다 보면 충분할 수 있다.

오늘 특히 Git 훅에서 이걸 많이 느꼈다. 안전장치는 문제의 원인을 더 쉽게 찾게 해야지, 모든 문제를 훅 때문인지 의심하게 만들면 안 된다. 앞으로는 정말 되돌리기 어려운 위험만 먼저 기계로 막고, 나머지는 문서와 습관으로 관리하는 쪽이 맞다.

배운 것

  • Codex 커스텀 프롬프트는 스킬이 아니라 호출형 프롬프트다.
  • 하네스 엔지니어링은 복붙보다 프로젝트별 조정이 중요하다.
  • Xcode 기본 프로젝트도 생성 직후 repo 구조에 맞게 확인해야 한다.
  • Git 훅은 많을수록 좋은 게 아니라 위험 경계에 정확히 있어야 한다.
  • 로컬 커밋과 원격 push는 위험도가 다르다.

느낀 점

오늘 세팅이 완벽한지는 아직 모르겠다. 실제 앱을 만들면서 맞지 않는 부분이 보일 것이다. 그래도 완벽하게 준비하려고 계속 멈춰 있는 것보다는, 지금 정도의 안전한 바닥을 만들고 앱 기획으로 넘어가는 편이 낫다.

제일 첫 세팅은 확실히 힘들다. 하지만 오늘은 이전 프로젝트의 방식을 그대로 가져오지 않고, 왜 필요한지 다시 생각해봤다는 점에서 의미가 있었다.
