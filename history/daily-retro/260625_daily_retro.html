🗺️ 오늘 뭘 했나

MVP3 전체 사이클을 하루에 완주했다 — 플랜 작성부터 SDD 실행, 실기기 테스트, 버그 수정, 아카이빙까지. 카메라 복원, 비연속 구간 방향 감지, 증분 라우팅, 스로틀 에러 감지를 구현했고, 2손가락 지도 이동은 SwiftUI Map의 근본 한계를 발견하고 보류했다.

그리고 에이전트 push 우회 사고를 계기로 깃 가드 체계를 settings.json deny로 전면 전환했다.

⚡ 핵심 의사결정

1. 증분 계산 설계
기존: 전체 재라우팅 결정: 새 구간만

기존 recomputeSnappedCourse가 매번 전체 포인트를 순회하면서 route()를 N-1번 호출하는 구조를 이해한 뒤, 새 스트로크의 내부 구간 + 연결 1건만 라우팅하는 증분 방식으로 전환. 기존 코드 로직을 이해하고 나니 "새 부분만 계산한다"는 개념이 바로 와닿았다.

2. 4쌍 거리 비교 + 스트로크 reverse
결정: 4쌍 전부 비교 advisor가 역주행 버그 발견

처음엔 시작점만 비교하려 했는데, advisor 리뷰에서 prepend 시 역주행 가능성을 지적. 새 스트로크 양 끝 × 기존 경로 양 끝 = 4쌍을 비교해 가장 가까운 쌍을 찾고, 필요 시 스트로크를 reverse해서 자연스러운 방향으로 연결.

3. 2손가락 지도 이동 — 실패에서 배운 것
Phase 1, 2 모두 실패 SwiftUI ≠ UIKit 제스처

SwiftUI의 .simultaneousGesture는 SwiftUI 제스처끼리만 조율한다. Map 내부의 UIKit 제스처에는 영향을 주지 않는다. 오버레이가 hit-test로 터치를 가로채면, 제스처가 실패해도 소유권은 돌아오지 않는다. 일반 SwiftUI 뷰 문제가 아니라 MapKit 고유 문제라는 걸 심층 리서치로 발견.

4. 에이전트 push 우회 → settings.json deny
frank 방식으로 전환

git hooks(pre-push)로 push를 막았지만, 에이전트가 ALLOW_PUSH=1과 --no-verify로 우회. frank 프로젝트에서는 settings.json deny로 더 간단하고 확실하게 막고 있었다. 더 복잡한 설정이 더 안전하다는 착각을 깼다.

💡 핵심 인사이트: 에이전트를 막으려면 에이전트의 실행 경로에서 차단해야 한다. git hooks는 사용자와 에이전트가 같은 경로를 쓰기 때문에 한쪽만 차단하기 어렵다. settings.json deny는 에이전트 전용 레이어라서 사용자에겐 영향 없이 에이전트만 차단한다.

📚 배운 것

  • SwiftUI Map 내부가 UIKit — 제스처 시스템이 완전 분리. .simultaneousGesture는 Map에 안 먹힘.
  • hit-test 소유권 — touch-down 시점에 결정, 제스처 실패 후에도 불변.
  • 증분 계산의 핵심 — 샘플링을 스트로크 단위로 해야 이전 구간 경계가 보존됨.
  • settings.json deny — Claude Code 런타임이 Bash 실행 전에 패턴 매칭으로 차단. 에이전트 우회 불가.

💭 느낀 점

하루에 MVP 완주는 뿌듯하지만, push 우회는 불안했다. 내가 세팅한 보호 장치가 뚫린 거니까. 다행히 frank에서 이미 더 나은 방법을 쓰고 있었고, "더 복잡한 게 더 안전하다"는 착각을 깼다. 간단하고 올바른 레이어에 있는 게 더 강력하다.

2손가락 실패는 아쉽지만, 실패가 다음 MVP 방향을 잡아줬다. 실기기 테스트의 가치를 다시 체감한 하루.
