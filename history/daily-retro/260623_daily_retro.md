# 데일리 회고 — 2026-06-23 (월)

## 오늘 뭘 했나

MVP2(UX 개선 + 스로틀 완화)를 시작하고 완료까지 간 하루였는데, 절반 이상의 시간은 기능 구현이 아니라 **워크플로우 자체를 고치는 데** 써야 했다.

이전 세션에서 Task 2를 진행하다가 시뮬레이터 테스트가 무한 루프에 빠져서 세션을 통째로 잃었다. 오늘 새 세션을 열고 `/trace-init`으로 상태를 복원한 뒤, Task 1부터 다시 검증하면서 시작했다. 테스트를 돌리자마자 같은 문제가 재현됐고, 여기서부터 원인 파악에 들어갔다.

결과적으로 세 가지 문제가 겹쳐 있었다:
1. iOS 18.5 시뮬레이터의 Swift 런타임 버그 (`@Observable` 객체 해제 시 malloc double-free)
2. 레이스 테스트용 블로킹 stub이 모든 테스트에서 route()를 영원히 멈추게 함
3. xcodebuild가 테스트 병렬화를 위해 시뮬레이터를 자동 복제하면서 런칭 실패

이 세 가지를 하나씩 분리하고 수정하는 데 Task 1~2 구간의 대부분 시간이 소모됐다. 특히 에이전트가 테스트를 실행하는 동안에는 내 다음 질문이 인식되지 않아서, 무한 루프를 끊고 방향을 바꾸는 것 자체가 힘들었다.

하지만 이걸 잡고 나니까 Task 3~5는 서브에이전트로 순식간에 흘러갔다. 각 Task를 서브에이전트에 넘기면 빌드+테스트 통과까지 확인하고 돌아와서, 커밋만 하면 됐다. 인프라가 안정되니까 구현 자체는 정말 빨랐다.

기능적으로는 MVP1 실기기 피드백 5건(위치 시작 화면, 그리기 모드 표시, 코스 핀, 상태 잔상, 권한 거부)을 전부 해결하고, MKDirections 스로틀도 구간 캐시 + 디바운스로 체감 개선했다. 실기기 검증에서 새로운 피드백 4건이 나왔지만, MVP2 범위에서 줌 레벨만 조정하고 나머지는 백로그에 등록해 마무리했다.

## 핵심 의사결정과 이유

### 1. iOS 18.5 포기 → iOS 26.5 전환

- **상황**: 테스트가 `malloc: pointer being freed was not allocated`로 크래시하면서 무한 재시작
- **선택지**: ① DerivedData 정리 ② `nonisolated deinit {}` 워크어라운드 ③ iOS 26 시뮬레이터 전환
- **결정**: iOS 26.5 전환
- **왜**: DerivedData 정리는 효과 없었고, `nonisolated deinit`은 모든 ViewModel에 추가해야 하는 반면 시뮬레이터 전환은 즉시 해결. 리서치 결과 Swift 런타임 버그(swiftlang/swift#87316)로 확인됐고, iOS 26.x에서 수정됨
- **인사이트**: 최소 지원 버전(iOS 17)과 테스트 런타임 버전이 달라지는 건 이상적이지 않지만, 유닛 테스트는 OS 버전에 의존하지 않으므로 지금은 괜찮음. 배포 전 멀티버전 전략은 백로그에 등록

### 2. 블로킹 stub을 별도 클래스로 분리

- **상황**: `StubCoursePlanningService`가 `CheckedContinuation`으로 모든 `route()` 호출을 블로킹 → `snappedRoute()` 기본 구현이 `route()`를 호출할 때 영원히 멈춤
- **선택지**: ① 블로킹 모드를 플래그로 토글 ② 별도 클래스(`BlockingCoursePlanningService`)로 분리
- **결정**: 별도 클래스로 분리
- **왜**: 기본 stub은 "즉시 반환"이 안전한 기본값. 블로킹은 레이스 컨디션 테스트에서만 필요하고, 클래스를 나누면 실수로 블로킹 stub을 다른 테스트에서 쓸 일이 없어짐

### 3. 줌 레벨만 조정, 카메라 점프 제거는 연기

- **상황**: 앱 시작 시 전체 지도 → 내 위치로 점프하는 UX 문제
- **선택지**: ① UserDefaults 캐시 + CLLocationManager 동기 위치로 근본 해결 ② 줌 레벨만 조정(100→500m) ③ 그대로 두기
- **결정**: 줌 레벨만 조정
- **왜**: 카메라 캐시 패턴은 프로토콜 변경 + 새 유틸 + View 로직 변경으로 별도 마일스톤 규모. 줌 조정만으로도 체감 개선되고, 점프 자체는 기능이 깨지는 문제가 아님. 리서치 결과를 백로그에 기록해놨으니 다음 MVP에서 제대로 다루면 됨

## 인사이트 & 피드백

**워크플로우 문제는 기능보다 먼저 잡아야 한다.** 오늘의 가장 큰 교훈이다. Task 1~2에서 테스트 인프라 문제를 해결하는 데 시간의 절반 이상을 썼지만, 그걸 잡고 나니까 Task 3~5는 서브에이전트로 각각 1~2분 만에 끝났다. 만약 "일단 넘어가고 나중에 고치자"고 했으면 매 Task마다 같은 무한 루프에 빠졌을 것이다.

이건 개발 전반에 적용되는 원칙이다 — CI가 불안정하면 기능을 아무리 빨리 짜도 머지를 못 하고, 테스트가 신뢰할 수 없으면 리팩토링을 못 한다. **인프라의 안정성이 팀(또는 에이전트)의 속도 상한을 결정한다.**

## 배운 것

- **Swift 런타임 버그 디버깅**: 항상 같은 주소에서 크래시하면 우리 코드가 아니라 프레임워크 내부 문제일 가능성이 높다. iOS 버전을 바꿔서 재현 여부를 확인하는 게 가장 빠른 진단법이었다.
- **xcodebuild 테스트 병렬화**: `-parallel-testing-enabled NO` 없이 `xcodebuild test`를 실행하면 시뮬레이터를 자동 복제(clone)한다. 소규모 프로젝트에서는 이 기능이 오히려 문제를 일으킨다.
- **프로덕션 지도 앱의 초기 로딩 패턴**: UserDefaults에 마지막 카메라 영역을 저장하고 동기 복원 → 점프 없는 재방문 경험. `CLLocationManager.location`으로 캐시된 위치를 동기적으로 읽을 수 있다는 것도 새로 알았다.
- **테스트 더블 설계 원칙**: "기본 동작은 즉시 반환, 특수 동작(블로킹)은 전용 클래스로 분리"가 철칙. 하나의 stub에 모든 시나리오를 넣으면 의도치 않은 상호작용이 생긴다.

## 느낀 점

솔직히 오전에 시뮬레이터가 계속 새로 뜨고 테스트가 무한 루프를 도는 걸 보면서 답답했다. 에이전트가 테스트를 실행하는 동안에는 내 메시지가 인식이 안 되니까, 문제를 봐도 방향을 바꿀 수가 없었다. 결국 "일단 멈추고 제대로 분석하자"고 강하게 개입한 게 전환점이었다.

그런데 그 고생 덕분에 testing.md에 시뮬레이터 규칙이 훨씬 단단해졌고, 솔루션 문서에 iOS 18.x 버그 대응이 기록됐다. Task 3~5가 서브에이전트로 술술 흘러간 건 이 투자의 직접적인 결과다. "삽질"이 아니라 "인프라 투자"였다고 생각하면 뿌듯하다.

MVP1에서 실기기로 테스트했던 피드백들이 전부 해결된 것도 좋았다. 특히 모드 전환 시 상태가 꼬이던 문제를 `InteractionMode` enum 하나로 깔끔하게 정리한 게 마음에 든다.
