🔥 오늘 뭘 했나

MVP2를 시작하고 완료까지 간 하루. 하지만 시간의 절반 이상은 기능 구현이 아니라 워크플로우 자체를 고치는 데 써야 했다.

이전 세션에서 Task 2 진행 중 테스트가 무한 루프에 빠져 세션을 통째로 잃었다. 오늘 새 세션을 열고 Task 1부터 다시 검증하면서 시작했는데, 같은 문제가 즉시 재현됐다.

세 가지 문제가 겹쳐 있었다:

런타임 버그 테스트 설계 xcodebuild
  • iOS 18.5의 @Observable malloc double-free (Swift 런타임 버그)
  • 블로킹 stub이 모든 테스트의 route()를 영원히 멈추게 함
  • xcodebuild가 시뮬레이터를 자동 복제하면서 런칭 실패

이걸 잡고 나니까 Task 3~5는 서브에이전트로 각각 1~2분 만에 끝났다. 인프라가 안정되니까 구현 자체는 정말 빨랐다.

🎯 핵심 의사결정

1. iOS 18.5 → iOS 26.5 전환

DerivedData 정리로 안 됐고, nonisolated deinit 워크어라운드는 모든 ViewModel에 추가해야 했다. 시뮬레이터 전환이 즉시 해결. 리서치로 Swift 런타임 버그(swiftlang/swift#87316)임을 확인.

2. 블로킹 stub 별도 클래스 분리

기본 stub은 "즉시 반환"이 안전한 기본값. CheckedContinuation 블로킹은 레이스 테스트 전용 클래스로 분리해서 실수 방지.

3. 줌 레벨만 조정, 카메라 점프는 연기

UserDefaults 캐시 패턴은 프로토콜 변경 + 유틸 + View 로직으로 별도 마일스톤 규모. 줌 100→500m로 체감만 개선하고, 리서치 결과를 백로그에 기록.

📚 배운 것

  • 항상 같은 주소에서 크래시하면 프레임워크 내부 문제 → iOS 버전 바꿔서 재현 여부 확인이 가장 빠른 진단법
  • xcodebuild test는 기본적으로 시뮬레이터를 복제한다 → -parallel-testing-enabled NO 필수
  • 프로덕션 지도 앱은 마지막 카메라 영역을 UserDefaults에 저장하고 동기 복원 → 점프 없는 재방문 경험
  • 테스트 더블 원칙: 기본=즉시 반환, 블로킹=전용 클래스 분리

😌 느낀 점

시뮬레이터가 계속 새로 뜨고 테스트가 무한 루프를 도는 걸 보면서 답답했다. 에이전트가 테스트를 실행하는 동안에는 내 메시지가 인식이 안 되니까 방향을 바꿀 수가 없었다. "일단 멈추고 제대로 분석하자"고 강하게 개입한 게 전환점이었다.

MVP1 실기기 피드백이 전부 해결된 건 뿌듯했다. 특히 InteractionMode enum 하나로 상태 꼬임을 깔끔하게 정리한 게 마음에 든다. Task 3~5가 술술 흘러간 건 "삽질"이 아니라 "인프라 투자"였다는 증거다.
