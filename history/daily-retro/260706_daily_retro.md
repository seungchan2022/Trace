  있던 걸 발견해서 정정.

## 핵심 의사결정과 이유

**원거리 attach 버그를 "anchor 단일 선택 + 기존 로직 재사용"으로 고치기로 함**
전날 근접 케이스를 상대 비교(`<=`)로 안전하게 고쳤던 것과 같은 원리를, 원거리 케이스에도
"스트로크 양끝 중 더 가까운 쪽 하나만 골라서(anchor) 그 지점을 기존 로직에 넣는" 방식으로 확장하기로
했다. 왜 이게 안전한지는 "근접 판정이 이미 양끝이 핀에 가까운 경우를 다 가로채므로, 원거리 로직에
도달하는 시점엔 이미 양끝 다 핀에서 멀다는 게 보장된다"는 게 핵심 근거였다 — 다만 이 근거를 내가
정말 정확히 이해하고 있는지는 아래 "느낀 점"에 솔직히 적는다.

## 기획/설계 과정

1차 아이디어는 "스트로크 양끝과 두 핀을 4쌍으로 비교"하는 방식이었는데, 이건 예전에 왕복 그리기
판정이 흔들리던 문제(4쌍 비교)를 다시 불러올 위험이 있어 기각. 대신 "양끝 각각의 최소 거리를 구해서
더 작은 쪽만 anchor로 쓰고, 그 뒤엔 기존 로직 그대로"라는 더 좁은 방식으로 좁혔다. 이 설계를 실제
코드로 옮기는 과정에서, 처음 짠 코드가 "닫힌 코스는 무조건 이어붙는다"는 기존 규칙을 깨뜨리는 걸
테스트로 발견해서 다시 고쳤다.

## 인사이트 & 피드백

- **"이해했다"고 넘어간 것과 실제로 이해가 맞물린 것은 다르다.** 오늘도 여러 번 "이해했어?"라고
  물었고 "그런 것 같다"는 대답을 들었지만, 정작 지금 다시 물어보면 정확한 이해는 아직 부족하다는
  걸 서로 인정했다. 확인 질문 한 번으로 "이해 완료"라고 넘어가면 안 되고, 상대방이 자기 말로
  설명해볼 수 있는지까지 봐야 진짜 확인이 되는 것 같다.
- **버그 하나 고치는 데 전체 워크플로우를 다 도는 게 항상 맞는 건 아닐 수 있다.** 이미 끝난 MVP
  안에서 발견된, 범위가 좁고 명확한 버그 하나를 고치는 데도 스펙 검토→구현→리뷰→문서 갱신→아카이빙
  전체를 다 돌리다 보니, 세션이 여러 번 끊기고 체감 속도가 느려졌다. Superpowers 워크플로우 자체는
  좋지만, 작업 규모에 맞게 무게를 조절하는 게 필요해 보인다는 문제의식이 생겼다.
- **문서에 남은 "죽은 정보"는 스스로 안 없어진다.** "코스 저장"을 백로그에서 뺀 결정이 다른 문서
  (완료 회고, 로드맵)에는 전파가 안 돼서, 그 이후로도 계속 "다음 방향"인 것처럼 여러 곳에서 재생산됐다
  — 심지어 오늘 내가 새로 쓴 회고에도 그대로 옮겨 적었다. 결정이 바뀌면 그 결정을 참조하는 문서
  전체를 훑어야 하는데, 오늘은 그걸 못 했다가 사용자가 지적해서야 알았다.

## 배운 것

- attach 판정 로직에서 "근접 처리를 먼저 해서 특정 조건을 가로채면, 그 뒤 로직은 그 조건이 이미
  배제됐다고 가정해도 된다"는 패턴 — 게이트 순서 자체가 안전성 증명의 일부가 된다는 걸 코드로 확인.
- swiftlint의 함수 길이/파라미터 개수 제한에 걸렸을 때, 로직을 그대로 private 메서드로 추출하는 것만
  으로(동작 변경 없이) 해결 가능하다는 것.

## 느낀 점

솔직히 오늘도 "제대로 이해했다"는 확신이 잘 안 선다. 여러 번 설명을 듣고 되묻는 과정을 거쳤는데도
막상 "그럼 뭐가 문제였는지 설명해봐"라고 하면 스스로 명확히 못 하겠다는 걸 인정하게 됐다. 그리고
버그 하나 고치는 데 이렇게 여러 세션·여러 서브에이전트가 오가는 게 맞는 건가 하는 의문도 들었다 —
Superpowers 워크플로우가 나쁘다는 게 아니라, 지금 규모에 비해 과한 건 아닌지 다음에 진지하게
따져봐야겠다.
