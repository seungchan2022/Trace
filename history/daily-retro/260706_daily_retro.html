핵심 의사결정

원거리 attach 버그를 "anchor 단일 선택 + 기존 로직 재사용"으로 고치기로 했다. 스트로크 양끝 중 어느 쪽이 어느 핀에 더 가까운지 하나만 골라(anchor), 그 지점을 기존 로직에 그대로 흘려보내는 방식. 안전 근거는 "근접 처리가 이미 양끝이 핀에 가까운 경우를 다 가로채므로, 이 로직에 도달하는 시점엔 이미 양끝 다 핀에서 멀다"는 것.

기획/설계 과정

1차 아이디어(스트로크 양끝 × 두 핀 4쌍 비교)는 예전에 왕복 그리기 판정이 흔들리던 문제를 다시 불러올 위험이 있어 기각. "양끝 각각의 최소 거리 중 더 작은 쪽만 anchor로 쓰고, 나머지는 기존 로직"으로 범위를 좁혔다. 실제 구현 중 "닫힌 코스는 무조건 이어붙는다"는 기존 규칙을 깨뜨리는 걸 테스트로 발견해 다시 고쳤다.

인사이트 & 피드백

"이해했다"고 넘어간 것과 실제 이해가 맞물린 것은 다르다.
여러 번 "이해했어?"라고 물었지만, 막상 되짚어보니 정확한 이해는 아직 부족했다는 걸 서로 인정했다. 확인 질문 한 번으로 끝내지 않고, 상대가 자기 말로 설명해볼 수 있는지까지 봐야 진짜 확인이 된다.
버그 하나 고치는 데 전체 워크플로우를 다 도는 게 항상 맞는 건 아닐 수 있다.
이미 끝난 MVP 안에서 발견된, 범위가 좁고 명확한 버그를 고치는 데도 스펙 검토→구현→리뷰→ 문서 갱신→아카이빙 전체를 다 돌리다 보니 세션이 여러 번 끊기고 느려졌다. Superpowers 워크플로우 자체는 좋지만, 작업 규모에 맞는 무게 조절이 필요해 보인다.
문서에 남은 "죽은 정보"는 스스로 안 없어진다.
"코스 저장"을 백로그에서 뺀 결정이 다른 문서(완료 회고, 로드맵)에는 전파되지 않아서, 계속 "다음 방향"인 것처럼 재생산됐다 — 오늘 새로 쓴 회고에도 그대로 옮겨 적었다가 지적받고서야 알았다.

배운 것

  • attach 판정에서 "근접 처리가 먼저 특정 조건을 가로채면, 그 뒤 로직은 그 조건이 이미 배제됐다고 가정해도 된다" — 게이트 순서 자체가 안전성 증명의 일부가 된다.
  • swiftlint 함수 길이/파라미터 개수 제한은, 로직을 그대로 private 메서드로 추출하는 것만 으로(동작 변경 없이) 해결 가능.

느낀 점

솔직히 오늘도 "제대로 이해했다"는 확신이 잘 안 선다. 여러 번 설명을 듣고 되묻는 과정을 거쳤는데도 막상 "그럼 뭐가 문제였는지 설명해봐"라고 하면 스스로 명확히 못 하겠다는 걸 인정하게 됐다. 그리고 버그 하나 고치는 데 이렇게 여러 세션·여러 서브에이전트가 오가는 게 맞는 건가 하는 의문도 들었다.
