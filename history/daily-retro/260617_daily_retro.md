# 2026-06-17 하루 회고

> 한 줄 요약: "두 개의 AI 도구로 같은 프로젝트를 이어 달리기 위한 레일을 깔고, 그 위에서 첫 MVP의 구조를 새로 설계해 옮겼다."

## 오늘 뭘 했나

오늘은 기능을 막 찍어내기보다 **"앞으로 어떻게 일할지"의 토대**를 만든 날이었다. 흐름은 이랬다.

1. `trace-init`으로 세션을 차갑게 시작하지 않으려다, 곧바로 **구조 불일치**를 발견했다. 문서(`project-decisions`·`architecture`)는 `Domain/CoursePlanning`을 말하는데 실제 코드는 `Shared/Routing`이었다.
2. 그런데 진짜 하고 싶었던 건 따로 있었다 — Codex와 Claude Code를 **세션/토큰에 따라 번갈아 쓰는 듀얼 체제**. 비싼 단일 플랜 대신 두 도구를 오가며 개발하기.
3. 그래서 먼저 **듀얼 도구 셋업**을 깔았다. 진입 파일(`CLAUDE.md`→`AGENTS.md` 심볼릭), 커맨드 서랍(`.claude/commands`→`docs/prompts` 심볼릭), 프롬프트 도구 중립화, `dual-tool.md` 규율.
4. 그 위에서 **폴더 구조를 새로 설계**하고(`Shared/Routing`→`Domain/CoursePlanning`, `Route`→`Course` 전면 rename), 라우트 플래너 **MVP 코드를 그 구조로 옮기며 실제 구현·빌드·테스트**까지 했다.
5. "체크박스를 안 켜서 다음 세션이 장님이 되는" 문제를 막으려 **체크박스·compound 규율을 룰로 의무화**했다.
6. Claude Code에도 **워크플로 플러그인을 동등 설치**(superpowers·compound·XcodeBuildMCP)하고, 그 과정에서 **compound 통합 gap**(Codex의 openai-curated엔 내장, obra 오리지널엔 없음)을 발견해 룰로 보강했다.
7. 마지막으로 **6개 단위로 커밋**했는데, 도중에 staged 누적으로 커밋이 오염돼 `reset`으로 깨끗이 재구성했다.

## 핵심 의사결정과 이유

### 1. 듀얼 도구 — "완벽히 같게"가 아니라 "같은 능력"
- **상황**: 단일 플랜은 토큰/세션이 빨리 소진된다. 두 도구를 오가고 싶다.
- **선택지**: ⓐ 한 도구로 고정 ⓑ 두 도구를 완벽히 동일 세팅 ⓒ 룰·프롬프트는 공유, 도구별 어댑터만 분리
- **결정**: ⓒ. 진입 파일과 커맨드 서랍만 심볼릭으로 도구별로 두고, 룰·플랜·git은 리포에 공유.
- **왜**: 비싼 사전작업(룰의 도구 중립 분리)이 이미 끝나 있어 추가 비용이 거의 없었다. "완벽히 같은 세팅"은 환상이고, 정말 중요한 90%는 이미 리포에 있다.
- **인사이트**: 도구를 맞추는 게 아니라 *능력*(기획·플랜·TDD·리뷰·iOS 빌드)을 맞추는 것이 목표다.

### 2. 폴더 구조 — 단순 이동이 아니라 재설계
- **상황**: 이전에 정한 `Shared/Routing` 구조가 마음에 안 들어, 문서상으로는 이미 새 구조로 바꿔둔 상태였다.
- **결정**: 폴더 이동 + `Route`→`Course` **타입 전면 rename**까지 한 번에. page-first MVVM + Clean Architecture 레이어(`Domain`/`Infrastructure`/`Pages`)로 재배치.
- **왜**: 어중간하게 폴더만 옮기면 네이밍이 문서와 또 어긋난다. 이전 설계를 버리고 새 설계로 정합화하는 게 핵심이었다.
- **인사이트**: Xcode가 `PBXFileSystemSynchronizedRootGroup`이라 파일 이동만으로 자동 반영된 게 큰 도움. "진짜 블로커가 뭔지" 한 가지(pbxproj 그룹 모델)만 먼저 확인한 게 주효했다.

### 3. 체크박스·compound를 룰로 의무화
- **상황**: 구현은 다 됐는데 플랜 체크박스가 전부 꺼져 있어, 다음 세션이 "아무것도 안 한 줄" 알 상황이었다.
- **결정**: 플랜은 task별 실행 + 체크박스 실시간 갱신, 리뷰 해결 직후 `ce-compound`로 교훈 추출 — 둘 다 룰로 박았다.
- **왜**: 룰 문장 하나로는 약하다. 워크플로에 박아야 도구가 바뀌어도 지켜진다.

### 4. 플러그인 전략과 토큰 판단
- **결정**: superpowers·compound·XcodeBuildMCP를 Claude Code에도 설치.
- **왜**: 스킬은 한 줄 설명만 상시 로드되고, MCP는 deferred(이름만)라 **설치가 세션 수명을 의미 있게 깎지 않는다**. 세션을 태우는 건 작업 자체다.

### 5. compound 통합 gap
- **발견**: Codex의 superpowers(openai-curated)는 리뷰 끝에 `ce-compound`를 부르도록 내장됐지만, Claude에 깐 obra 오리지널(v6.0.2)엔 그 연결이 없었다.
- **결정**: 플러그인 SKILL.md를 고치면 업데이트 때 사라지므로, **리포 룰에** 연결을 박았다.

### 6. 커밋 오염 복구
- **상황**: 커밋을 단위로 나누다, `git commit`이 staged 전체를 담는 탓에 이전 시도의 잔여가 엉뚱한 커밋에 섞였다.
- **결정**: `reset`(--hard 아님)으로 베이스까지 풀고 처음부터 깨끗이 6개 재커밋.

## 기획 / 설계 과정

폴더 재설계가 오늘의 가장 큰 설계 작업이었다. "이미 구현된 코드를 어떻게 새 구조로 옮기느냐"인데, 단순 이동이면 쉽지만 네이밍(`Route`→`Course`)과 레이어(`Domain`/`Infrastructure`/`Pages`) 재배치가 얽혀 있었다. 매핑 표를 먼저 만들고, 빌드를 깨뜨리지 않는 순서(새 위치 작성 → 진입점 갱신 → 옛 디렉터리 삭제 → 빌드/테스트)로 진행했다. 동시에 듀얼 도구라는 "메타 작업"과 MVP라는 "실제 기능"이 같은 날 겹쳐, 토대와 산출물을 함께 굴린 하루였다.

## 인사이트 & 피드백

- 같은 문제(룰이 가리키는 것과 실제가 어긋남)가 오늘 하루에만 세 번 나왔다 — 구조 불일치, compound gap, 커밋 오염. 전부 "선언과 실제의 괴리"다. 다음엔 **선언을 바꾸면 실제도 같은 호흡에 맞추는** 습관이 필요하다.
- 큰 작업 전 "진짜 블로커 하나"만 먼저 확인하는 게 빨랐다(pbxproj 그룹 모델).

## 배운 것

- **체크박스 실시간 갱신이 핸드오프의 핵심이다.** 세션은 토큰 소진으로 예고 없이 죽고, 그때 다음 도구가 읽을 수 있는 건 git + 결정 문서 + 플랜 체크박스뿐이다. 도구별 메모리는 상대가 못 본다.
- `git commit`은 staged 전체를 담는다. 분할 커밋 땐 각 커밋 전 staged를 정확히 통제해야 한다.
- 스킬/MCP 설치는 토큰을 거의 안 먹는다(progressive disclosure, deferred tools).

## 느낀 점

기능을 만든 날은 아니었지만, **앞으로 며칠을 빠르게 만들 레일을 깐 날**이라 뿌듯했다. 특히 두 도구를 오가도 작업이 끊기지 않게 하는 구조를 직접 설계한 게 의미 있었다. 한편 커밋 오염을 만났을 때는 살짝 당황했는데, `reset`으로 침착하게 되돌린 경험이 오히려 자신감이 됐다.
