# 2026-07-09 ~ 07-13 회고 — Swift 6 + 디자인 (MVP12)

> 마지막 데일리 회고가 2026-07-08(`260708_daily_retro.md`, MVP11 완결)이었고, 그 이후 며칠은
> 회고를 못 챙겼다. 대신 그 공백 전체가 사실상 MVP12(Swift 6 마이그레이션 + 디자인 입히기)
> 하나의 사이클이었어서, 이번엔 "하루"가 아니라 이 사이클 전체를 범위로 잡아 회고한다.

## 이 기간에 뭘 했나

크게 두 가지: **① Swift 6로 옮기기**, **② 이미 만들어둔 기능들에 디자인 입히기**. 부수적으로
**③ 유튜브 링크를 던지면 자막을 분석해주는 설정**(yt-dlp 기반)도 추가했다 — 이건 MVP12 작업은
아니고, 같은 기간에 곁다리로 만든 도구.

**① Swift 6**: 암묵/명시 isolation을 정리하고 Sendable을 맞추고 GCD 패턴을 현대화해서
`SWIFT_VERSION = 6`으로 전환, 경고 0까지 갔다. 그런데 Task 1~7을 처음 확정한 전략(격리
기본값 = MainActor)대로 다 구현하고 나서, 실기기에서 진짜 크래시가 하나 터졌다 — MapKit
`NSObject` 서브클래스가 MainActor 기본값을 잘못 물려받은 것. 그래서 전략을 반대로
뒤집었다(기본 nonisolated + 명시 `@MainActor`). 계획에 없던 재작업이었지만 구체적인 크래시
증거가 있어서 결정 자체는 흔들리지 않았다.

**② 디자인 입히기**: 클로드 디자인으로 미리 만들어둔 목업("Trace 경로 짜기 v2")에서 토큰·
컴포넌트 스펙을 뽑아내고, 그걸 플래너 화면 전체(탑바·FAB·바텀시트·구간리스트·핀/폴리라인·
저장/목록/왕복/redo)에 적용했다. P1 범위는 예상보다 빨리 끝났는데, 그 다음이 길었다 —
실기기·시뮬레이터로 계속 써보면서 라이트모드 반투명이 안 읽힌다, 드래그로 시트 크기를
조절하고 싶다, 바텀시트가 다이내믹 아일랜드를 덮는다, 로딩 중에 화면이 움찔거린다, 되돌리기/
초기화/현재위치 버튼이 안 보인다 — 이런 걸 하나씩 실사용하면서 계속 찾아냈다. "완료"라고
선언한 커밋(`81e6e0b`) 이후에도 이 발견-수정 사이클이 여러 번 더 돌았다.

## 핵심 의사결정과 이유

### 1) Swift 6 격리 기본값을 실기기 크래시 이후 뒤집음

- **상황**: 확정 스펙대로 `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor`로 Task 1~7을 전부 구현.
- **선택지**: (a) 그대로 유지하고 크래시만 국소 수정 (b) 격리 기본값 전략 자체를 뒤집기.
- **결정**: (b). 프레임워크 연동 타입(MapKit `NSObject` 서브클래스)이 기본 MainActor를 잘못
  물려받는 게 근본 원인이라, 프레임워크 연동이 많은 이 코드베이스에는 기본 무격리가 더 안전한
  실패 모드라고 판단.
- **왜**: 실기기 크래시라는 확실한 증거가 있었기 때문에, 이미 다 구현한 걸 뒤집는 비용을
  감수할 가치가 있었다.

### 2) 디자인 스펙보다 실사용 판단을 우선한 결정들

- 라이트 모드 바텀시트를 스펙(Glassmorphism, 반투명)에서 이탈해 완전 불투명으로 변경 —
  실기기에서 보니 지도 텍스트가 겹쳐 보여 거의 안 읽혔다.
- "출발 지정됨" 상태 칩 제거 — 로직을 보니 사실상 항상 뜨는 의미 없는 상태였다.
- **왜**: 목업은 정적 이미지라 놓치는 실사용 문제가 있고, 그건 실제로 기기에서 써봐야만
  드러난다. 스펙과 다르게 가더라도 실사용 판단이 이겼다.

### 3) 바텀시트 안정화가 "완료" 이후에도 계속 이어진 사이클

드래그-리사이즈 3단계 확장 → 안전영역 피드백 루프(시트가 커질수록 안전영역 측정값 자체가
줄어들어 시트가 상태바를 덮는 문제) → 헤더 정렬 버그(로딩 전환마다 화면이 움찔거리는 원인이
실은 `HStack`의 베이스라인 정렬 방식이었음) → FAB 스택이 collapsed 상태에서도 이미 시트에
가려져 있던 버그. 넷 다 "완료"라고 문서화된 뒤에 실사용하다가 발견됐다.
- **왜 반복됐나**: 매번 코드를 고치고 나서 시뮬레이터로 다시 보지 않고 "이러면 될 것"이라고
  결론 낸 게 원인. 실제로 화면을 보고 나서야 진짜 원인이 드러났다(아래 인사이트 참고).

### 4) 이전 회고 문서를 의심하고 재검증

"자기측정 프레임 순환" 버그 문서가 "움찔거림의 원인을 찾았다"고 결론 내렸었는데, 실제로는
증상 이름이 비슷한 **다른** 버그(콘텐츠 늘어날 때 시트가 커지는 것)만 고친 거였다. 사용자가
"이전 분석이 잘못됐을 수도 있다"고 직접 의심하지 않았다면 이 정정은 없었을 것.

## 기획/설계 과정

목업(클로드 디자인) → 토큰/컴포넌트 스펙 추출 → P1(전면 적용)/P2(드래그 리사이즈, halo, km
마커 등 후순위) 스코프 분리 → P1 적용 → 실사용하며 P2 중 트리거된 것들(드래그 리사이즈, halo)
당겨오기 → 그 과정에서 나온 버그들을 하나씩 실측·수정 → 완료 재확인 → 아카이빙. 처음부터
정적 스펙을 100% 따르기보다, "일단 기능 위에 입혀보고 실사용하면서 계속 다듬는" 접근이었다.

## 인사이트 & 피드백

**코드를 읽고 "이러면 될 것 같다"고 결론 내리는 것과, 실제 화면(스크린샷이든 정밀 측정이든)을
보고 확인하는 것은 다르다 — 후자만 신뢰해야 한다.** 이번 기간 내내 반복된 패턴은: 코드 추론만
믿고 "고쳤다"고 말한 뒤 실사용에서 다시 걸리는 것이었다. 반대로 실제로 막힌 지점을 뚫은 건
전부 "일단 보기"였다 — 사용자가 보내준 스크린샷, 접근성 트리 덤프, 개별 하위 뷰 높이를 하나씩
찍어본 것. 이건 꼭 픽셀 단위로 정밀하게 재는 것만을 뜻하지 않는다 — 스크린샷 한 장으로 "이거
봐, 여기 겹쳐있잖아"라고 보여주는 것도 똑같이 유효한 확인이다. 핵심은 "확인 없이 결론 내지
않기"다.

## 배운 것

- **기술**: SwiftUI `HStack(alignment: .firstTextBaseline)`은 자식의 "첫 서브뷰 타입"(Text vs
  ProgressView 같은 비-텍스트)에 따라 암묵적 베이스라인 기준점이 달라져, 개별 뷰 크기가 전혀
  안 바뀌어도 부모 컨테이너 높이가 흔들릴 수 있다. `GeometryProxy.safeAreaInsets.top`도
  "고정값"이 아니라 형제 뷰가 화면을 거의 다 채우면 동적으로 줄어들 수 있어, 그 값을 크기
  계산에 되먹이면 피드백 루프가 생긴다.
- **프로세스**: XCUITest의 `accessibilityIdentifier`가 특정 SwiftUI 구조에서 자식→부모로
  뭉개지는 현상이 있어, 초반 진단 시도들이 엉뚱한 이유로 계속 실패했다 — 다음엔 이런 걸 더
  일찍 의심하고 좌표 기반으로 우회할 것.

## 느낀 점

디자인 입히는 작업이 생각보다 힘들었다. 클로드 디자인으로 초안은 받았지만, 실제로 구현된
기능에 자연스럽게 맞추려면 여러 번 수정이 필요했고, 특히 UI 디테일은 로직 버그처럼 에러
메시지로 명확히 짚어주지 않아서 정확히 뭐가 문제인지 표현하기가 어려웠다 — "바텀시트가
어디까지 올라갈지", "움찔거리는 느낌" 같은 게 그 예다. 전담 디자이너가 있어서 "이 색, 이
크기로 해달라"고 지정해주면 편했겠지만 지금은 그런 상황이 아니라 스스로 판단해야 하는 부담이
있었다. 그래도 기능이 어느 정도 갖춰진 지금 시점에 디자인 시스템을 입혀두면, 앞으로 새 기능을
만들 때도 일관된 스타일로 자연스럽게 확장할 수 있을 거라 보고 이 타이밍에 진행했다.
