## Keep (잘 된 것)

**3사이클 21태스크 전부 subagent-driven-development로 implementer→reviewer 루프를 돌렸고, 사이클마다 최소 1건 이상 실제 버그를 리뷰·QA 단계에서 잡았다**
사이클 1은 최종 브랜치 리뷰(opus)에서 요약 화면 평균 페이스가 활동 시간이 아닌 세션 duration을 쓰던 버그를 발견해 즉시 수정했고, 사이클 2는 실기기 QA 중 사용자가 "평균 페이스와 킬로미터별 페이스 표 시간 기준이 다르다"를 직접 발견해 `RunSplitCalculator.splits()`에 활동시간 기준 통일을 적용했다. 두 버그 모두 유닛 테스트로는 안 잡히는, 여러 계산 경로가 서로 다른 시간 기준을 쓰던 종류의 버그였다 — 코드 리뷰만으로는 100% 안 잡힌다는 걸 다시 확인했다.

**사이클 3(run-goal)에서 실기기 QA 방식 자체를 진화시켰다**
1·2사이클은 시나리오마다 실제로 10~15분씩 걷거나 뛰어야 했다. 3사이클에서는 Xcode Debug → Simulate Location + 커스텀 GPX로 전환해, 실제로 움직이지 않고도 정확히 원하는 지점(예: "정확히 1.5km 지점")에서 이벤트가 발생하는지 확인할 수 있었다. 화면 잠금 중에도 GPX가 계속 흘러가서 시나리오 6(주머니+음악) 같은 백그라운드 시나리오도 이 방식으로 검증됐다. 방법은 `docs/solutions/workflow-issues/gpx-simulated-location-real-device-qa.md`로 남겨, 다음 QA 사이클부터 재사용 가능하다.

**경량 사이클 판정(run-goal)이 실제로 시간을 아꼈다**
`docs/agent-rules/workflow.md`의 "열린 결정 없음·범위 좁음·새 레이어 없음" 기준에 따라 플랜 서브에이전트 리뷰와 최종 브랜치 리뷰를 생략하고 태스크별 리뷰만 수행했다. 6개 기능 태스크 전부 sonnet 리뷰 Approved, Critical/Important 0건으로 마무리돼 판정이 정확했음이 결과로 증명됐다.

## Problem (아쉬운 것)

**run-goal 체크리스트를 실제 UI 제약과 안 맞게 먼저 만들었다**
사용자가 "시간 목표는 3분 기준으로"라고 요청했을 때, 실제 구현된 휠 피커의 최소값이 5분(`stride(from: 5, through: 180, by: 5)`)이라는 걸 확인하지 않고 GPX 파일·체크리스트를 전부 3분 기준으로 만들었다. 사용자가 실기기에서 피커를 돌려보다가 직접 발견해서야 5분으로 재작업했다 — 플랜의 UI 스펙(휠 피커 범위)을 QA 자료 작성 전에 먼저 대조했어야 했다.

**GPX 파일 하나를 정확히 목표 지점에서 끝내서 확인·대응 여유가 없는 설계를 처음에 만들었다**
거리 목표(3km) GPX를 딱 4km(목표+1km)에서 끝나도록 만들었더니, "4km 지점의 평소 안내를 듣고 반응할 시간"이 없이 파일이 곧바로 되감기는 문제를 사용자가 지적했다. 목표 이후 확인 지점 다음에 최소 1분의 안전 여유를 두는 설계로 다시 고쳐야 했다 — 처음부터 "마지막 확인 지점 + 여유 구간"을 계산에 넣었어야 했다.

**GPX 되감기 문제로 앱의 실제 결함(GPS 거리 이상치 방어 부재)이 우연히 드러났다**
Xcode의 GPX 시뮬레이션이 파일 끝에서 처음 웨이포인트로 자동으로 되감기는데, `RunSession`/`RunTrack`이 연속 샘플 간 거리·속도 이상치를 전혀 검사하지 않아(정확도만 검사) 그 순간이동 점프가 그대로 누적 거리에 합산됐다. 이번 사이클 범위는 아니었지만, 발견된 김에 트리거 조건과 함께 `docs/backlog.md`에 기록해뒀다.

## Surprise (예상 밖)

**같은 종류의 "시간 기준 불일치" 버그가 두 사이클 연속으로 나왔다**
사이클 1의 요약 평균 페이스 버그와 사이클 2의 스플릿 표 평균 페이스 버그는 서로 다른 코드 경로였지만 본질은 같았다 — "활동 시간(일시정지 제외)"이라는 새 개념이 도입된 뒤, 그 개념을 일관되게 반영 안 한 계산식이 남아있던 것. 일시정지 기능(사이클 1)이 만든 개념 변경의 여파가 다음 사이클(스플릿 계산)까지 이어진 셈 — 기반 개념이 바뀌면 그걸 소비하는 모든 계산 경로를 한 번에 감사해야 한다는 교훈.

**목표 달성 후에도 오디오가 "완전히 자유 러닝처럼" 동작하는 게 설계상 자연스럽게 나왔다**
`goalAchieved` 플래그가 한 번 true가 되면 끝까지 유지되고 `RunAudioCoach`가 그 전이만 감지해서 딱 1번만 발화하도록 설계했더니, 별도 분기 없이도 "달성 후에는 목표 관련 언급 없이 평소 km 안내만 계속 나온다"는 동작이 자연스럽게 성립했다. 사용자가 "3km 달성하고 4km 되면 그냥 자유 러닝처럼 동작하는 거지?"라고 물었을 때 코드를 다시 볼 필요 없이 바로 "그렇다"고 답할 수 있었던 건 이 설계가 정확했다는 신호였다.

## 남은 기술부채 (`docs/backlog.md` 연결)

- **GPS 거리 이상치(비정상 점프) 방어 로직 없음** — `RunSession.ingest`/`RunTrack.append`가 정확도만 검사하고 속도·점프 거리는 무검사. 실사용에서 거리 이상 사례가 실제로 확인되면 착수(트리거 조건부).
- **연속 발화(일시정지·재개·km 경계가 짧은 간격으로 겹침) — 부분 확인만 됨** — 실사용 중 소리 겹침이 체감되면 재조사.
- **일시정지 구간 지도 선 표시 — 다르게 표시할지 검토 이연** — 실사용 불편이 나오면 점선/반투명 등 검토.
- **건물 밀집지 GPS 정확도·20분+ 배터리 체감 — 계속 미확인** — 세 사이클 QA 모두 기회가 없어 계속 이월 중. 다음 실사용 기회에 재확인.
