# MVP13 — run-tracking: 자유 러닝 트래킹 + 기록 설계

2026-07-13. Trace의 두 번째 기둥 "러닝(기록)"을 세운다. 코스 탐색과 독립적으로,
일반 러닝앱처럼 그냥 뛰어도 시작→GPS 트래킹→종료 요약→기록 저장이 되는 기능.

- 단위: MVP13 = 마일스톤 3개(`run-tracking`, `run-live-activity`, `run-record-save`)
  = **사이클 2개**
  - 사이클 1 (이 스펙): ① `run-tracking` + ② `run-live-activity` — 브랜치 `feature/run-tracking`, 무게 **표준**
  - 사이클 2: ③ `run-record-save` — ①·②의 실기기 러닝 QA 결과를 저장 스키마에 반영하기
    위해 분리. 별도 경량 스펙으로 진행(이 문서 §8이 출발점)
- 본 러닝 QA(실제로 뛰면서 검증)는 ② 완료 후 통합 실행 — ① 완료 시점엔 가벼운 확인만

## 0. 배경 — 두 기둥 비전 (2026-07-13 킥오프에서 정정·확정)

기존 "러닝 전→중→후" 로드맵(2026-07-07)을 킥오프에서 정정했다: 러닝은 코스의 부속
단계가 아니라 **독립 기둥**이다.

- **기둥 1 — 경로 탐색(계획)**: 지금까지 만든 것. 코스를 그리고 저장한다.
- **기둥 2 — 러닝(기록)**: 코스 없이 그냥 뛰어도 되는 일반 러닝앱 경험. 이번 MVP.
- **연동(제3의 경험)**: 탐색한 코스를 골라 뛰고, 러닝 후 계획 경로 vs 실제 경로를 겹쳐
  비교. **다음 MVP(코스 연동)** — 비교 기능은 연동 시나리오에만 존재한다.

경로 탐색이 필요 없는 사용자는 러닝 탭만 써도 된다. 시장 표준 구조(Strava·NRC —
"그냥 뛰기"가 기본, 코스/루트는 선택 기능)와 같다. MVP12에서 디자인 시스템을 먼저
분리해둔 것도 두 기둥이 독립적으로 서기 위한 기반이었다.

## 1. 범위

### 포함

- **① run-tracking**: 탭 구조(코스/러닝) 도입 + 러닝 탭 — 시작 → 백그라운드 포함 GPS
  트래킹(실시간 경로·거리·시간·페이스) → 길게 눌러 종료 → 요약. 저장 없음(닫으면 소멸).
  단, **DEBUG 빌드 한정 원시 샘플 덤프**는 포함: 요약 화면에서 세션의 샘플 스트림
  (타임스탬프·좌표·고도·속도·정확도·필터 판정)을 JSON/GPX로 공유 시트에 내보내는 개발용
  버튼. "저장 없음"의 유일한 예외이며, 본 러닝 QA에서 회수한 덤프가 사이클 2 저장 스키마
  결정의 근거 데이터가 된다(문서 리뷰 지적: 이 장치가 없으면 사이클 분리의 근거인
  "QA 실데이터 반영"이 기억에 의존하게 됨).
- **② run-live-activity**: 잠금화면 Live Activity + Dynamic Island — 트래킹 중
  거리·시간·페이스 실시간 표시. Widget Extension 타깃 신설.
- **③ run-record-save** (사이클 2): 기록 저장(SwiftData) + 기록 목록/상세.

### 제외 (이번 MVP에서 하지 않음)

- **코스 연동** — 코스 골라 뛰기, 러닝 중 계획·실제 동시 표시, 러닝 후 비교. 다음 MVP.
- **심박·케이던스·칼로리·HealthKit** — 심박·칼로리는 Apple Watch 없이는 사실상 불가.
  워치 연동 MVP에서 심층 조사와 함께. 일반 HealthKit은 entitlement + 사용자 동의만으로
  쓸 수 있음을 확인했고(임상 기록·ResearchKit과 달리 기관 협력 불요), 기능 없이
  entitlement만 미리 넣는 것은 YAGNI + 심사 리스크라 하지 않는다.
- **러닝 오디오 안내**(1km마다 거리·페이스 음성 안내) — **확정 예정 기능**(트리거 대기
  아님, 사용자 확정 2026-07-13). 세션 데이터의 소비자라 이번 구조 위에 그대로 붙는다.
  `docs/backlog.md`에 planned로 기록.
- **일시정지(수동·자동)** — 러닝앱 표준 기능이지만 첫 슬라이스는 시작/종료만.
  본 러닝 QA에서 필요성 확인 후 결정.
- **진행 중 세션 복구**(앱 강제 종료 후) — ③저장 이후 별도 결정. 이번엔 소실 수용.
- 잠금화면 외 위젯, 오디오 ducking, 코스 썸네일 등.

## 2. 측정 데이터 — GPS 열만, 샘플 스트림 원칙

### 이번 MVP에서 측정 (전부 CLLocation에서 나옴)

| 원시 | 파생 (저장하지 않고 계산) |
|---|---|
| 좌표열 + 타임스탬프 | 경로(폴리라인), 총 거리, 경과 시간 |
| 속도(speed) | 현재·평균 페이스 |
| 고도(altitude) | 고도 상승/하강 |
| 정확도(horizontalAccuracy) | 저장 안 함 — 필터 판정 전용 |

- **현재 페이스는 `speed` 순간값을 그대로 쓰지 않는다** — 유효하지 않은 값(음수)은
  무시하고, 최근 구간(예: 30초) 이동 평균으로 스무딩한다. 순간값 직결은 화면·잠금화면
  숫자가 널뛰는 원인(스펙 리뷰 지적, 2026-07-13).
- **필터 공백 구간의 거리 규칙**: 터널 등에서 샘플이 연속으로 버려진 뒤 신호가 회복되면
  마지막 유효 샘플과 직선 연결로 거리를 가산한다(과소 계상 수용 — 러닝앱 관례).
  저장 시 공백을 표시할지는 사이클 2에서 결정.
- **고도 상승도 원시값 합산을 쓰지 않는다** — GPS 고도는 ±5~15m 노이즈가 상시 존재해
  양의 변화량을 그대로 누적하면 평지 러닝에서도 수백 m의 유령 상승이 표시된다(문서 리뷰
  지적 — 페이스 스무딩과 같은 구조의 문제). `RunSample`에 verticalAccuracy를 실어 필터
  판정에 쓰고, 고도 상승은 누적 임계값 방식(연속 상승이 3~5m를 넘을 때만 가산 — 러닝앱
  관례, 수치는 구현 중 튜닝)으로 계산한다. 본 러닝 QA에 "평지 코스에서 고도 상승이 상식
  범위인지" 관찰 항목을 포함한다.

### 저장 설계 원칙 — "기록 = 타임스탬프 샘플 스트림들의 묶음"

- 원시 트랙(타임스탬프 붙은 좌표열)을 저장하고 파생값은 계산한다. 파생값만 저장하면
  나중에 재계산이 불가능하고, 원시 트랙은 코스 비교(다음 MVP)의 원료다.
- 미래 심박·케이던스는 **스트림 하나(시각+값)를 옆에 추가**하는 것으로 끝난다 —
  기존 스키마를 뜯지 않는 additive 확장. 이것이 "워치 대비를 지금 해두는" 실체다.

## 3. 화면 흐름 — 탭 구조 + 러닝 탭 3상태

앱 루트를 `TabView`(코스/러닝)로 감싼다. 두 기둥의 독립성이 구조에 반영된 것
(대안 비교: 플래너에서 풀스크린 진입은 "그냥 뛰는" 사용자도 플래너를 거쳐야 해 기각,
허브 화면 분기는 화면 2개 앱에 과해 기각). "이 코스로 뛰기" 진입로는 코스 연동 MVP에서
코스 탭 쪽에 추가로 붙인다.

러닝 탭(`RunPage`)의 상태 (대기 → 신호 확보 → 트래킹 → 요약):

- **대기(idle)**: 현재 위치 중심 지도 + 큰 "시작" 버튼. 위치 권한 거부 시 기존 플래너
  패턴(설정 이동 알럿) 재사용.
- **신호 확보(acquiring)**: 시작 탭 직후 첫 유효 샘플을 받을 때까지의 짧은 대기 상태 —
  "GPS 신호 찾는 중" 표시 + 취소 가능. `RunSession`이 시작 직후 캐시된 옛 샘플을 버리는
  구조라 첫 유효 위치까지 수 초의 공백이 있는데, 이 상태 없이 바로 트래킹 화면을 그리면
  거리 0·좌표 미확정 화면이 노출된다(문서 리뷰 지적). 첫 유효 샘플 수신 시 tracking으로
  자동 전이.
- **트래킹(tracking)**: 실시간 폴리라인(accent 민트) + 사용자 위치 카메라 추적.
  하단 패널(기존 바텀시트 스타일)에 거리·경과 시간·현재 페이스. 종료는 **길게 누르기**
  (오탭 방지, 러닝앱 표준). 백그라운드·탭 전환에도 세션 유지(§4 세션 소유권).
  지도를 팬하면 카메라 추적이 수동으로 풀리고 자동 복귀하지 않으므로(iOS 동작),
  **"내 위치로 복귀" 버튼**을 둔다(플래너의 recenter 버튼 스타일 재사용).
- **요약(summary)**: 경로 전체 카메라 핏 + 총 거리·시간·평균 페이스·고도 상승.
  ①에서는 "닫기"만(대기로 복귀, 데이터 소멸) + DEBUG 한정 샘플 덤프 내보내기 버튼(§1),
  ③에서 "저장" 추가.

트래킹 중 코스 탭으로 이동해도 세션은 유지되는데, 포그라운드에서는 잠금화면 Live
Activity가 보이지 않아 진행 중임을 알 단서가 없다(문서 리뷰 지적) — **러닝 탭 아이콘에
진행 중 배지**(점 또는 SF Symbol 변형)를 표시해 코스 탭에서도 세션이 돌고 있음을 알린다.

디자인은 별도 목업 사이클 없이 기존 디자인 시스템(`Trace/DesignSystem/` 토큰·컴포넌트)
으로 조립한다 — 플래너와 같은 순서(기능 먼저, 다듬기는 실사용 후). UI 세부 배치는
진행 중 확정(기존 합의: UI 세부는 확인 없이 진행).

## 4. 아키텍처 — 기존 레이어 패턴 확장

- **Domain — `Trace/Domain/RunTracking/`**
  - `RunSample`: 타임스탬프+좌표+고도+속도+**수평·수직 정확도**. 순수 값 타입,
    `Sendable`/`Codable`. 샘플 스트림 원칙(§2)의 단위. 정확도 두 개는 **전송용**(수평은
    좌표 필터, 수직은 고도 필터 판정 — §2) — `RunSession`이 필터를 하려면 샘플에 실려
    와야 한다(스펙 리뷰 지적: 층 배치 모순 해소). ③ 저장 시에는 정확도를 제외한다(§2).
  - `RunTrack`: 샘플 배열 + 파생값 계산(거리·시간·페이스·고도 상승). 거리 계산은
    `CourseCoordinate+Geo` 재사용.
  - `Protocol/RunLocationStreamProtocol`: 연속 위치 스트림 포트, `AsyncStream` 기반.
    Domain은 CoreLocation을 모른다(기존 원칙).
- **Infrastructure — `Trace/Infrastructure/Location/`**
  - `RunLocationTracker`(신규, 단발 조회용 `CoreLocationService`와 별도):
    **`@MainActor` + `CLLocationManager` delegate 방식** — 기존 `CoreLocationService`
    선례와 동일(§ "Infrastructure는 기본 nonisolated"의 예외로, `CLLocationManager`는
    런루프 있는 스레드 생성이 필요해 main 격리가 안전한 실패 모드). delegate 콜백을
    `AsyncStream<RunSample>`으로 브리지해 Domain 포트를 구현한다.
    대안이었던 iOS 17 `CLLocationUpdate.liveUpdates`는 기각 — `LiveConfiguration(.fitness)`로
    activityType 상당 구성은 가능하나, `allowsBackgroundLocationUpdates`가 적용되지 않아
    백그라운드 유지에 `CLBackgroundActivitySession` 별도 수명 관리가 필요하고,
    `desiredAccuracy`·`distanceFilter`·자동 일시정지 수준의 세밀 제어가 없으며, 기존
    `CoreLocationService` delegate 선례와의 일관성 때문(문서 리뷰에서 근거 정정: 원래
    "activityType을 쓸 수 없다"고 적었던 것은 부정확).
  - 설정: `activityType = .fitness`, 최고 정확도, `allowsBackgroundLocationUpdates = true`,
    자동 일시정지 끔. 프로젝트에 **Background Modes(location) capability 추가**.
    권한은 기존과 동일한 "앱 사용 중"으로 충분(백그라운드 모드가 있으면 Always 불필요 —
    러닝앱 표준. 백그라운드 위치 사용 중 상단 인디케이터가 뜨는 것은 정상 동작).
  - **위치 사용 설명 문구 갱신**: 현재 `NSLocationWhenInUseUsageDescription`은 코스
    플래너 전용 문구라 백그라운드 연속 트래킹을 커버하지 않는다 — 두 용도를 모두 설명
    하는 문구로 교체(심사 시 background location 모드의 정당화 근거).
- **Application — `Trace/Application/RunTracking/`**
  - `RunSession`(오케스트레이터, `@MainActor` `@Observable`): 시작/종료, 스트림 구독,
    필터링(정확도 나쁜 샘플·시작 직후 캐시된 옛 샘플 버림), 샘플 누적, 파생값 실시간
    갱신. **`DependencyContainer`가 소유** — 화면이 아니라 컨테이너에 살아서 탭 전환·
    뷰 소멸에도 세션 유지. 화면·잠금화면·(미래)오디오 안내·저장은 전부 이 세션의
    **소비자**다.
- **Pages — `Trace/Pages/RunPage/`**
  - `RunPage` + `RunPageViewModel`(`@MainActor` `@Observable`): 3상태 표현.
  - 지도는 **SwiftUI `Map`** — 표시 전용(폴리라인+사용자 위치)이라 시작점으로 충분.
    플래너가 MKMapView로 간 이유(그리기 중 제스처 소유권)가 러닝 화면에는 없다.
    플래너의 `MapViewRepresentable`은 재사용하지 않는다(플래너 전용 로직 덩어리).
    단, SwiftUI Map은 콘텐츠 갱신 시 오버레이를 통째로 재구성하므로 **자라는 폴리라인을
    매 위치 업데이트(~1Hz)마다 다시 그리면 장거리(수천 샘플)에서 프레임 드랍 위험**이
    있다(스펙 리뷰 지적) — **지도 폴리라인 갱신은 스로틀**(예: 3초 또는 20m 단위, 수치는
    구현 중 튜닝)하고, 본 러닝 QA에서 프레임 문제가 확인되면 러닝 지도도 MKMapView
    표시 전용 래퍼로 내리는 것을 폴백으로 한다.
- **격리**: Swift 6 클래식 모델 유지 — UI/상태 타입(`RunSession`·ViewModel)에만 명시
  `@MainActor`, 값 타입·Infrastructure는 기본 nonisolated (`project-decisions.md`).

## 5. Live Activity (마일스톤 ②)

- **Widget Extension 타깃**(예: `TraceWidgets`) 신설 — Live Activity UI는 확장 타깃에 산다.
  App Groups는 불필요(ContentState가 ActivityKit API로 전달됨).
- **메인 앱 타깃에 `NSSupportsLiveActivities = YES` 필수** — 없으면 `Activity.request`가
  실패한다. 이 프로젝트는 Info.plist 자동 생성이라 `INFOPLIST_KEY_NSSupportsLiveActivities`
  빌드 세팅으로 넣는다(스펙 리뷰 지적).
- `RunActivityAttributes`: 고정값(시작 시각) + `ContentState`(거리·페이스).
- **갱신 주체는 앱 타깃의 `RunActivityController`(@MainActor)** — `RunSession`의 상태
  변화를 `withObservationTracking` 루프(또는 세션이 노출하는 이벤트 스트림)로 구독해
  Activity 생성·갱신·제거를 전담한다. `RunSession`은 ActivityKit을 모른다 — `@Observable`
  관찰은 SwiftUI 뷰 갱신에 묶여 있어 잠금화면·(미래)오디오 같은 비뷰 소비자에게는 이런
  명시적 구독 메커니즘이 필요하고(문서 리뷰 지적), 이 컨트롤러가 미래 오디오 안내
  소비자의 선례가 된다.
- **경과 시간은 매초 갱신하지 않는다** — `Text(timerInterval:)`로 시스템이 스스로 초 단위
  갱신(잠금화면 타이머 표준 기법). 앱은 위치 업데이트가 올 때만 거리·페이스를 갱신
  (스로틀 포함) → ActivityKit 갱신 예산 문제 없음.
- 수명: `RunSession` 시작 시 Activity 생성 → 갱신 → 종료 시 **즉시 제거**
  (`dismissalPolicy: .immediate` — 요약은 앱 화면이 담당하므로 잠금화면 잔류 불필요).
  시스템의 활성 8시간 상한은 러닝 용도에선 사실상 무해하나 인지하고 둔다.
- UI: 잠금화면 카드 + Dynamic Island(축소: 거리 / 확장: 거리·시간·페이스).
- iOS 17+ 전제라 버전 제약 없음.

## 6. 에러 처리

- **위치 권한 거부**: 기존 플래너 패턴(설정 이동 알럿) 재사용.
- **"정확한 위치" 끔**(reduced accuracy, ~1–2km 오차): 권한은 있어도 러닝 트래킹이
  사실상 불가능한 상태 — 시작 시 `accuracyAuthorization`이 `.reducedAccuracy`면 먼저
  `requestTemporaryFullAccuracyAuthorization(withPurposeKey:)`로 **인앱 임시 정밀 권한**을
  요청한다(세션 한정이라 러닝 1회 단위와 정확히 맞음 — 문서 리뷰에서 설정 이동보다 마찰
  낮은 공식 경로로 교체). `INFOPLIST_KEY`… 빌드 세팅으로
  `NSLocationTemporaryUsageDescriptionDictionary` 키 추가 필요. 사용자가 거부한 경우에만
  "정확한 위치" 켜기 안내 알럿(설정 이동)으로 폴백.
- **GPS 신호 불량**(터널·실내): 정확도 나쁜 샘플은 필터가 버림 → 경로가 튀지 않음.
  화면에 "GPS 신호 약함" 표시, 회복 시 자동 재개(공백 거리 규칙은 §2).
- **러닝 도중 권한 회수**(설정에서 끔·위치 서비스 오프): 세션을 자동 종료하고 그때까지
  수집분으로 요약 화면 표시(데이터를 버리지 않는다).
- **앱 강제 종료**: 진행 중 세션 소실 — 이번 MVP에서 수용(§1 제외 참고).
- **Live Activity 생성 실패**(사용자가 설정에서 끔): 앱 내 트래킹은 정상, 조용히 무시.

    카드 형식 체크리스트(`docs/qa/`, `testing.md` 템플릿). **러닝 후 샘플 덤프 회수**(§1)
