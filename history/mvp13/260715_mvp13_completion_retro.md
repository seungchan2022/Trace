# MVP13 완료 회고 — 자유 러닝 트래킹 + 기록 (2026-07-13 ~ 07-15)

## 요약

마일스톤 3개: **run-tracking**(탭 구조 + 러닝 탭 대기→트래킹→종료 요약, 백그라운드 GPS 스트림,
필터링·파생값 실시간 계산 — 저장 없음), **run-live-activity**(잠금화면 Live Activity + Dynamic
Island, Widget Extension 타깃 신설), **run-record-save**(기록 자동 저장 SwiftData + 목록/상세
화면). Trace의 두 번째 독립 기둥("러닝")을 코스 탐색과 분리해 세웠다. 사이클 1(run-tracking +
run-live-activity)은 Task 10개 + 최종 브랜치 전체 리뷰(opus)로, 사이클 2(run-record-save)는
경량 사이클(Task별 리뷰만, 전체 리뷰 생략)로 진행 — 사이클 1의 실기기 QA 결과(원시 GPS 재계산
대조까지 마침)를 저장 스키마 설계에 반영하기 위해 의도적으로 분리했다. 전체 테스트 228개 그린,
두 사이클 모두 실기기 QA 통과.

## Keep

- **실기기 QA 결과를 기다렸다가 저장 스키마를 설계한 순서가 맞았다.** 사이클 1에서 GPS
  샘플·정확도 필터의 실전 동작을 원시 데이터 재계산으로 검증한 뒤에야, 사이클 2에서 "정확도
  2필드는 저장 제외, 원시 5필드는 다운샘플링 없이 전부 저장"을 확정할 수 있었다. 순서를
  바꿔 먼저 스키마부터 정했다면 근거 없는 추측성 결정이 됐을 것.
- **러닝 기록을 코스와 별도 스토어 파일로 분리한 결정.** 기존 코스 스토어에 스키마 변경을
  전혀 가하지 않아 마이그레이션 리스크가 0이었고, 실제로 어댑터 구현·리뷰 전 과정에서 코스
  데이터 쪽 회귀를 전혀 걱정하지 않아도 됐다.
- **Task 3 구현자가 브리프의 Mock 코드가 실제 툴체인에서 컴파일 안 되는 것을 스스로 발견하고,
  advisor 상의 후 최소 범위로 수정한 것.** `@MainActor final class` → `actor` 전환은 브리프에
  없던 편차였지만, 리뷰어가 "이게 정말 필요한 편차인가, 실제로 뭘 증명하는가"를 독립적으로
  검증했고 오히려 프로덕션(actor 기반 어댑터)과 더 일치한다는 걸 확인했다 — 편차를 성급히
  받아들이지 않고 검증한 리뷰 루프가 제대로 작동한 사례.
- **경량 사이클 결정(문서 서브에이전트 리뷰·최종 브랜치 리뷰 생략)이 실제로 시간을 아꼈다.**
  Task별 리뷰만으로도 0 Critical/Important를 유지했고, 사이클 2 규모(신규 기능 얹기, 아키텍처
  재설계 없음)에는 전체 리뷰가 과했을 것.
- **시뮬레이터 자동화 도구의 한계를 판단 근거로 정확히 남긴 것.** 스와이프 삭제 제스처가
  `swipe`/`drag`/`gesture` 세 가지 도구 모두에서 실패했을 때, "왜 실패했는지"(HID 계층이
  touch-move 이벤트 자체를 지원 안 함)를 각각 확인하고 나서야 실기기 QA로 이연했다 —
  한 번 시도해보고 포기한 게 아니라 근본 원인까지 확인한 뒤의 정당한 이연이었다.

## Problem

- **사이클 2 리뷰 서브에이전트가 세 번 연속 인프라 오류(connection closed / stalled)로
  실패했다.** 근본 원인은 알 수 없었고(모델/인프라 쪽 일시적 문제로 추정), 네 번째 시도에서야
  성공했다. 재시도 외에 할 수 있는 게 없었지만, 실패 패턴이 반복될 때 언제 접근을 바꿔야
  하는지(예: 프롬프트 축소, 다른 모델) 판단 기준이 미리 없었다.
- **플랜의 체크박스가 실제 구현 완료와 계속 어긋났다.** 사이클 1·2 플랜 모두 Task 진행 중
  체크박스를 갱신하지 않은 채 커밋했고, 아카이빙 시점에 소급 확인 노트를 추가해야 했다 —
  실행 중에 체크박스를 실시간으로 갱신하는 습관이 아직 없다.

## Surprise

- Swift 6.3 툴체인에서 `Sendable`을 상속하는 프로토콜은 `@MainActor` 클래스로 컨포먼스를
  구성할 수 없다는 제약(`SendableMetatype`-inheriting protocol)이 있다는 걸 몰랐다 — 브리프
  작성 시점(플랜 자체가 완결된 코드를 제공하는 방식이라 평소엔 이런 게 안 남았다)에는 드러나지
  않았던 툴체인 특성.
- 시뮬레이터 스크린샷 캡처 도구가 시트 표시 상태에 따라 이미지를 회전/기울여서 반환하는
  현상 — 실제 앱 레이아웃 버그가 아니라 캡처 도구 자체의 아티팩트로 추정되지만, 재현 조건은
  특정하지 못했다. `snapshot_ui`의 엘리먼트 트리는 항상 정상 portrait 좌표를 반환했으므로
  기능 확인에는 영향 없었다.

## 남은 기술 부채

- 없음 — 이번 사이클에서 발견된 이슈는 전부 리뷰 루프 내에서 해소됨(Task별 리뷰 4건 전부
  0 Critical/Important).
- 실기기 QA 체크리스트의 "의도 일치"·"엣지/견고성 — 빈 목록" 항목은 사용자가 명시적으로
  확인하지 않은 채 통과 처리됨(핵심 시나리오 1~5로 충분하다고 판단) — 실사용 중 위화감이
  나오면 그때 재확인.

## 느낀 점

- 사이클 2를 진행하면서 스스로 되짚어보니, 최근 결정들이 대부분 "무엇을 만들지·어떻게 나눌지"
  같은 제품/의사결정 위주였고, 그 밑단의 기술이 실제로 어떻게 동작하는지는 잘 모르는 채로
  진행하고 있다는 걸 깨달았다. 개발자 취업준비를 목표로 하는 입장에서 이 정도 위임 구조가
  맞는 방향인지 확신이 서지 않는다 — 정확한 코드 레벨까지 알아야 할지, 원리·방법 레벨(왜 이
  방식을 썼는지 설명할 수 있는 정도)이면 될지, 아니면 제품을 보는 눈만 있으면 될지 아직
  결론을 못 냈다.
- 이 고민 자체는 이미 예상했었는지, `docs/prompts/trace-study.md`(`/trace-study`)를
  만들어뒀다는 걸 다시 확인했다 — 목적이 정확히 "코드 암기가 아니라 왜 이 기술을 골랐고 어떻게
  동작하는지를 설명할 수 있는 수준"이라, 방향은 이미 원리·방법 레벨로 잡아뒀던 셈인데 아직 한
  번도 실행해보지 않았다.
- 이 주제는 지금 결론 내지 않고, 별도로 더 의논하기로 함 (관련 배경은 메모리에 기록).
