# MVP 학습 정리

> Codex에서는 `$trace-study MVP1` 또는 `$trace-study`, Claude Code에서는 `/trace-study MVP1` 또는 `/trace-study`로 호출한다. 인자가 없으면 `docs/roadmap.md`의 최근 완료 MVP를 쓴다.
> 목적: 완료된 MVP를 **"흐름 + 개념"**으로 정리해, 만든 방식을 자기 언어로 설명할 수 있게 한다.

## 목적

코드 암기가 아니라 **"왜 이 기술을 골랐고, 어떻게 동작하는가"**를 면접/실무에서 설명할 수준으로 체화.
("만들기만 하고 이해 없이 넘어가서 나중에 설명·정리가 힘들었다"는 과거 교훈을 막기 위함.)

## 절대 원칙

- **추측 금지**: 기획 문서(spec) + plan + 실제 코드(`ios/`)를 읽은 뒤에만 설명 시작.
- **코드 스니펫 나열 금지**: 코드는 "흐름이 실제로 그런지" 확인용으로만 읽는다.
- **한 번에 한 구간 / 한 문제**: 사용자 확인을 받고 다음으로.

## 1. 대상 + 자료 수집

- `docs/roadmap.md`에서 대상 MVP 확인 → `history/<slug>/`의 spec/plan/회고 + 관련 코드 읽기.
- **사용자 행동 기준**으로 단계 분할 (기술 레이어 기준 아님).
- 각 단계에서 처음 등장하는 핵심 개념만 추출.

## 2. 단계별 사이클 (단계마다 반복)

- **(a) 흐름 한눈에**: 그 단계의 동작 순서를 간단히 보여준다 (필요하면 mermaid 텍스트).
- **(b) 구간별 개념 설명**: 흐름상 자연스러운 구간(3~5스텝)으로 묶어 **산문**으로 설명.
  "이게 뭐야 / 왜 이렇게 / 이전 흐름과 차이" 같은 예상 질문을 선제적으로 답한다.
  (불릿 나열·표로 개념 쪼개기 금지 — 흐름이 끊긴다.)
- **(c) advisor 팩트체크** (있으면): 기술적 오류·코드 불일치·누락 개념을 점검하고 반영.
- **(d) 셀프 퀴즈**: 한 문제씩 A/B/C/D. 개념 수에 비례 (≤5개→3문제, 6~10개→5문제 등).
- **(e) 저장**: `history/<slug>/concepts.md`에 해당 단계 내용을 append.

## 3. 마무리 (선택)

- `concepts.md` 상단에 복습용 핵심 요약 표 (질문 ↔ 핵심 답) 삽입.
- 원하면 HTML(daily-retro의 Claude Sunset 테마)로 렌더해 브라우저로 연다.
  **기본은 markdown만** — HTML/플래시카드는 사용자가 원할 때만 tier-up.
